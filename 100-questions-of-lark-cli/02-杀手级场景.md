# 第二章 · 杀手级场景

## 17. 飞书 CLI 最值得先试的 5 个杀手级场景

装好飞书 CLI 之后，最怕的就是不知道拿它干什么。这里给 5 个验证过最能打、上手最快的场景，按推荐顺序排。

第一个，日报周报自动化。让 Agent 从日历拉本周会议、从任务清单拉进度、汇总成文档发到群里。以前 90 分钟的活，现在 5 分钟。这是第 2 层效率白领的刚需，也是最容易让人第一次"哇"的场景。

第二个，会议纪要自动整理。开完会，Agent 读妙记的逐字稿，提取待办事项、标记责任人、生成纪要文档并发给参会人。待办还能自动同步到飞书任务清单。

第三个，每天早上自动推送日程和待办。配一个定时任务，Agent 每天早上 8 点拉你的日历和未完成任务，整理成简报发到私聊或群。

第四个，CI/CD 结果通知。代码合并或部署跑完，自动把结果推到研发群，成功走一个群，失败走另一个群。

第五个，多维表格批量操作。把 CSV 灌进多维表格、用自然语言查表格数据、监控表格变动触发通知。

这 5 个场景的共同点：都是重复性高、规则清晰、但每次手动做都费时间的事。它们是你从 CLI 获得第一个效率爆发的起点。先跑通一个，再扩展到第二个、第三个。

## 18. 怎么用一句话让 AI 替你把周报发到群里

核心思路是把周报拆成三步：拉数据、生成内容、发消息。每一步 CLI 都有现成命令。

拉数据：Agent 调 `lark-cli calendar +agenda` 拿这周的会议，调 `lark-cli task +get-my-tasks` 拉任务进度，如果数据在多维表格里就用 `lark-cli base +records-list` 读出来。

生成内容：Agent 拿到原始数据后，按你的周报模板生成 markdown 正文，然后调 `lark-cli docs +create` 创建飞书文档。

发消息：最后调 `lark-cli im +messages-send --chat-id "oc_xxx" --msg-type interactive` 把文档链接发到目标群。

你对 Agent 说的话就是："帮我整理这周的会议和任务，写成周报，发到研发群。"它会自己完成这三步。

关键点在身份选择。拉日历和任务要用 user 身份，因为这涉及你个人的数据；发消息可以用 user 也可以用 bot，看你希望群里看到是谁发的。第一次跑建议加 `--dry-run`，先预览不发，确认内容对了再真发。

## 19. 周报自动发群怎么避免发错人或发错群

自动发群最大的恐惧就是发错。要防住这件事，靠三层兜底。

第一层，chat-id 固化。不要让 Agent 每次自己找群，它会找错。提前把目标群的 chat-id 存成变量或写死在脚本里，chat-id 是 `oc_` 开头的字符串，从飞书群链接里能拿到。

第二层，dry-run 预览。第一次跑、换了数据源、改了模板，都先加 `--dry-run` 跑一遍，Agent 会把要发的内容打印出来不真发。你确认无误再去掉这个参数重跑。

第三层，小范围测试。正式发大群之前，先发到一个只有你自己的测试群，跑通整条链路再换到真群。

还有个常见坑：机器人不在群里导致发消息失败报错。发之前确认目标群里有你的 bot，不然消息发不出去，Agent 会卡在那一步反复重试。

## 20. 怎么让 AI 自动整理会议纪要并 @ 相关人

飞书的会议纪要链路比较完整，CLI 覆盖了关键节点。

开完会，先拿到妙记的 minute_token。然后 Agent 调 `lark-cli vc +notes --minute-tokens <token>` 拉逐字稿、总结、待办、章节这四类结构化数据。拿到待办列表后，Agent 识别每条待办的责任人，用 `lark-cli task +create --name "xxx" --assignee "ou_xxx"` 创建任务并指派。

纪要文档可以调 `lark-cli docs +create` 生成，内容包含会议基本信息、关键讨论点、待办清单。最后用 `lark-cli im +messages-send` 把文档链接发到参会人群，正文里用 `<at user_id="ou_xxx">` 标签 @ 相关人。

一个细节：待办的识别质量取决于逐字稿的清晰度。多人会议、口音重、讨论发散的场景，Agent 提取待办的准确率会下降。重要会议建议人工过一遍 Agent 生成的待办再发。

## 21. 会议纪要里的待办怎么自动同步到任务清单

纪要生成完，待办要能落到飞书任务里才算闭环。否则纪要文档躺在那没人看，待办就石沉大海了。

做法是 Agent 从纪要数据里提取待办项，逐条调 `lark-cli task +create` 创建任务。每个任务带标题、指派人、截止日期、关联的纪要文档链接。如果用多维表格管任务而不是飞书任务，就改调 `lark-cli base +record-create` 写到表格里。

一个真实的坑：多维表格的 select 字段，Agent 写入时如果选项名不匹配现有选项，API 会默默新建一个选项而不是报错。结果就是表里出现一堆重复选项。要防住这个，要么提前在 prompt 里明确告诉 Agent"只能选已有选项不可新建"，要么用 record-upsert 之前先 field-get 确认选项列表。

## 22. 怎么让 AI 自动查日程找空闲再建会议

约会议最烦的是找大家都空的时间段。飞书 CLI 有专门的命令查空闲。

Agent 先调 `lark-cli calendar +freebusy --user-ids "ou_a,ou_b,ou_c" --start-time xxx --end-time xxx` 查几个参会人某段时间的忙闲状态，拿到哪些时段所有人都空。然后调 `lark-cli calendar +time-suggestions` 让飞书推荐合适的时段。

确定时间后，用 `lark-cli calendar +event-create` 建日程，带上参会人、会议标题、会议室。会议室用 `lark-cli calendar +room-find` 找可用的。

需要注意，查忙闲要 user 身份，bot 身份查不了个人日历。如果你的 Agent 是多人共用的，每个用户的忙闲要用自己的 token 查。

## 23. 怎么用飞书 CLI 批量管理飞书日历日程

批量场景常见的有三个：批量创建、批量更新、批量删除。

批量创建：把一批日程数据整理成 JSON 或 CSV，写个循环调 `lark-cli calendar +event-create`。适合季度排期、项目里程碑这类一次建一批的场景。

批量更新：先 `lark-cli calendar events list` 拉出某段时间的所有日程，筛出要改的，逐个调 `lark-cli calendar events patch` 更新。适合会议时间整体顺延、会议室批量更换。

批量删除：先 list 再逐个 delete。删除是不可逆的，务必先 dry-run 预览要删的列表，确认了再真删。

批量操作最大的风险是并发。不要一次性并发太多请求，飞书 API 有 QPS 限制。稳妥的做法是串行循环，每条之间加个小延迟。

## 24. 怎么让 AI 每天早上自动推送当天日程和待办

这是个典型的定时自动化场景，需要把定时器和 CLI 配合起来。

定时器选 cron 或 launchd 都行。macOS 上用 launchd 更原生，Linux 服务器用 cron 更通用。定时任务调一个脚本，脚本里让 Agent 执行：拉今天的日程、拉未完成任务、汇总成简报、发到指定聊天。

简报格式建议固定，比如："今天你有 3 个会议，2 个待办过期，1 个截止今天。"让 Agent 严格按这个格式输出，别每次变花样。

推送目标看你的习惯。推到私聊最私密，推到工作群能让团队同步。第一次配置建议推到只有你自己的测试群，跑通几天再换真群。

注意 token 过期问题。如果定时任务跑了一段时间突然报授权失败，大概率是 user token 过期了，需要重新 `lark-cli auth login`。

## 25. 怎么把 CI/CD 跑完的结果自动推到飞书群

研发团队的高频需求。CI/CD 跑完，自动把结果推到飞书群，成功和失败走不同通知策略。

做法是在 CI/CD 的 pipeline 配置里加一个 webhook 或 script 步骤，调用 `lark-cli im +messages-send --chat-id "oc_xxx" --msg-type interactive --card '{...}'`。card 结构里带构建状态、提交信息、变更文件、日志链接。

成功和失败分群的做法：pipeline 里判断构建状态，成功走一个 chat-id，失败走另一个。也可以同一个群发不同颜色的卡片，绿色成功红色失败，视觉上一眼区分。

bot 身份发消息比较合适，因为 CI/CD 是系统行为，不需要绑定某个具体用户。但 bot 要在目标群里，不然发不出去。

有些团队的做法更狠：失败的消息 @ 负责人，用 `<at user_id="ou_xxx">` 标签，逼着负责人第一时间响应。

## 26. 怎么让 AI 读完一封邮件自动填进多维表格

邮件处理是白领的高频痛点。每天收一堆邮件，挑出重要的信息填到表格里，纯手工操作。

用 CLI 的链路：`lark-cli mail +list` 拉收件箱，筛选出要处理的邮件（按发件人、主题关键词、未读状态），然后 `lark-cli mail +get --message-id xxx` 读邮件正文。Agent 拿到正文后，按你定义的字段结构提取信息（比如客户名、订单号、需求描述），最后调 `lark-cli base +record-create` 写到多维表格对应字段。

提取质量取决于邮件格式的稳定性。格式固定的业务邮件（系统通知、表单提交、订单确认），Agent 提取准确率很高。格式随意的个人邮件，准确率会下降，建议人工复核。

一个实用技巧：在多维表格里加一个"原始邮件链接"字段，Agent 填数据时把邮件的链接也写进去。后续核对时一键跳回原文。

## 27. 怎么把 CSV 数据批量灌进飞书多维表格

两种做法。

少量数据（几十到几百行）：写个循环，逐行调 `lark-cli base +record-create`。简单直接，但慢，几百行可能要跑几分钟。

大量数据（上千行）：用飞书多维表格的批量创建接口。调 `lark-cli base records batch_create --params '{"table_id":"xxx"}' --data '{"records":[...]}'`，一次提交一批，速度快很多。

无论哪种方式，先确认表结构。字段类型要对得上，select 字段的选项要提前建好，日期字段的格式要统一。最稳的做法是先 `lark-cli base +fields-list` 拉一遍字段定义，按这个定义来准备数据。

跨表关联是另一个坑。如果导入的数据要关联另一张表的记录，先查被关联表的记录 ID，写入时把关联字段填成 record_id 而不是显示名称。

## 28. 怎么用自然语言查询多维表格里的数据

你不需要记住多维表格的字段名和筛选语法，直接对 Agent 说人话就行。

"帮我查一下这个月哪些客户还没回款。"Agent 会先 `lark-cli base +fields-list` 看表结构，理解"客户""回款""这个月"对应哪些字段，然后构造筛选条件调 `lark-cli base +records-search` 或 `+records-list` 拉数据，最后格式化返回给你。

这套交互的前提是 Agent 能看到表结构。第一次查某张表时，让它先拉 fields 和前几条记录做 schema 理解，之后查询就快了。

查询结果的呈现要设计。让 Agent 返回表格而不是长文本，或者直接生成一个新的飞书文档把结果列进去，方便分享和存档。

## 29. 群聊里的有价值信息怎么自动归档到知识库

群聊是最容易产生有价值信息也最容易丢失的地方。一条重要结论混在几百条闲聊里，三天后就找不到了。

归档的做法：用事件订阅监听群消息，每收到一条消息让 Agent 判断"这条值不值得归档"。判断标准由你定：包含某个关键词、@ 了某个标签、被多人回复、包含链接或文件。值得归档的，Agent 提取关键信息调 `lark-cli wiki +node-create` 或 `lark-cli docs +create` 写进知识库。

判断环节是难点。规则太严会漏，太松会把闲聊也归档进来。建议先用宽松规则跑一段时间，看看归档的内容质量，再逐步收紧。

事件订阅需要 bot 在群里，而且要在开放平台配置 `im.message.receive_v1` 事件订阅。配置完，每条群消息都会推给你的 Agent 处理。

## 30. 怎么让 AI 监控某个文档变更并自动通知

监控文档变更的需求很常见。比如需求文档改了要通知开发，排期表变了要通知全员。

做法是定时拉文档内容做 diff。Agent 定时调 `lark-cli docs +fetch --doc-id xxx` 读文档，和上次读到的内容对比，有变化就 `lark-cli im +messages-send` 推通知。

更精细的做法：只通知变化的部分，而不是"文档变了"。让 Agent 做 diff 后提取新增或修改的段落，通知正文里直接带变化内容，省得人还要点进去看。

监控频率别太高。每分钟拉一次对 API 是浪费，每 15 分钟或每小时一次足够大多数场景。文档变更通常不需要秒级响应。

如果是多维表格的变动监控，用多维表格自带的自动化功能比 CLI 监控更高效，因为它有原生的变更事件触发。

## 31. 怎么让飞书和外部系统双向同步数据

双向同步是最复杂的场景之一，出错代价高，建议谨慎。

单向同步相对简单：外部系统到飞书，或飞书到外部系统，都是定时拉一边推另一边。双向同步的难点在于冲突处理，同一条数据两边都改了，以谁为准。

如果必须做双向，建议给飞书作为主数据源。外部系统读飞书的数据做展示或计算，写操作统一回飞书。这样只有一个写入方向，避免冲突。

技术实现上，飞书侧用 CLI 读写，外部系统侧用各自的 API。同步逻辑用一个中间脚本协调，定时拉两边对比差异推更新。

监控必不可少。同步跑着跑着数据对不上的情况很常见，要么是字段映射错了，要么是某一边的数据格式变了。同步脚本要有日志，每次跑完记录同步了多少条、失败了多少条，方便排查。

## 32. 怎么用飞书 CLI 搭一个跨应用的消息桥接

场景：你希望在飞书里收到企业微信、钉钉或 Slack 的消息，或者反过来。

桥接的逻辑：一边监听 A 应用的消息事件，转发给 B 应用发送。飞书侧，用事件订阅监听 IM 消息，收到后调另一平台的 API 发出去。反方向同理。

飞书 CLI 在这里的角色是飞书侧的收发层。监听用 `lark-cli event consume im.message.receive_v1`，发送用 `lark-cli im +messages-send`。其他平台用它们自己的 SDK。

实际操作中，多数团队不会自己搭桥接，而是用现成的工具，比如有些 Agent 框架自带多平台消息路由。CLI 适合定制化程度高的场景，比如你想在转发时做内容过滤、翻译、格式转换，用 Agent 配合 CLI 比较灵活。

## 33. 怎么用 Python 加飞书 CLI 搭一个全自动日报机器人

日报机器人是 CLI 加脚本的经典组合，适合有一定技术基础的人。

整体架构：一个 Python 脚本，负责拉数据、调 Agent 生成内容、调 CLI 发消息。配合一个定时任务，每天固定时间触发脚本。

拉数据环节，脚本用 subprocess 调 `lark-cli calendar +agenda` 和 `lark-cli task +get-my-tasks`，解析 JSON 输出。生成内容环节，脚本把数据喂给一个大模型 API（比如 OpenAI、Anthropic），让它按日报模板生成 markdown 正文。发消息环节，脚本调 `lark-cli docs +create` 建文档，再调 `lark-cli im +messages-send` 发群。

这个架构的好处是每一步都可替换。数据源可以加邮件、加多维表格；生成模型可以换；发送目标可以换群。坏处是要维护的东西多，脚本、token、定时任务任何一个环节出问题日报就断。

建议加一个简单的健康检查：脚本跑完往一个监控群发一条"日报已发送"，没收到就说明哪里挂了。

## 34. 怎么搭建一个飞书智能问答客服让群消息自动回

群里有人提问，AI 自动回答，这是很多团队想要的场景。

核心组件：一个飞书自建应用（bot）负责在群里收发消息，一个桥接层把飞书消息转发给本地 AI Agent 处理，Agent 调 CLI 或知识库生成回复再发回群里。

配置步骤：先在开放平台创建应用，开通 `im:message` 权限和 `im.message.receive_v1` 事件订阅。然后搭桥接层，社区里有人用 lark-channel-bridge，你也可以自己写。桥接层收到群消息后转发给你的 Agent（Claude Code、Codex 或任何能调 CLI 的 Agent），Agent 处理完把回复发回群。

知识库是回答质量的关键。Agent 的知识来源分几层：系统提示词定行为规则，Skills 提供飞书 API 操作能力，长期记忆存历史交互，外部知识库存业务文档。没有知识库的客服只能回答通用问题，接了知识库才能回答具体业务。

安全上，建议 bot 只响应 @ 消息，不要响应群里所有消息，避免噪声和权限滥用。

## 35. 妙记逐字稿导出到一半就卡住怎么办

这是个真实高频的坑。用 CLI 导出妙记逐字稿，跑到某个时间点就停了，后面的内容拿不到。

排查方向有三个。

第一，分页问题。妙记逐字稿接口默认只返回第一页，长会议（比如 30 分钟以上）的内容可能跨多页。要加 `--page-all` 参数让它自动翻页拉完所有内容。

第二，接口选错。`lark-cli minutes minutes get` 只返回标题、时长这些元信息，不含逐字稿。逐字稿要走 `lark-cli vc +notes --minute-tokens xxx`，这是 skill 层封装好的正确路由。

第三，点数用完。妙记的转写服务有点数限制，用完了后续内容就拿不到。这种情况 CLI 报错不一定明显，要去看妙记后台确认点数余额。

确认是哪个原因后对症解决。分页加参数，接口选错换命令，点数用完充值。

## 36. 本地视频怎么用飞书 CLI 上传转成飞书妙记

本地视频转妙记是两步操作，不能一步到位。

第一步，上传到云盘。调 `lark-cli drive +upload --file ./your-video.mp4`，拿到返回的 file_token。

第二步，用 file_token 生成妙记。调 `lark-cli minutes +upload --file-token xxx`，飞书会启动转写流程，转写完后返回 minute_token。

拿到 minute_token 后，就能用 `lark-cli vc +notes --minute-tokens xxx` 拉逐字稿、总结、待办。

限制要先确认。妙记转写支持时长 6 小时以内、文件 6GB 以内的视频，格式覆盖 mp4、mp3、wav、mov 等常见类型。超长的视频要先切分。

转写不是实时的，视频越长等待时间越久。建议脚本里做个轮询，每隔一段时间查一次转写状态，完成了再拉数据。

## 37. 怎么让 AI 自动读取飞书会议纪要并提取待办

开完会，纪要生成完，待办要能自动落到任务系统里才算闭环。

Agent 做法是：调 `lark-cli vc +notes --minute-tokens xxx` 拿纪要的结构化数据，里面包含总结、待办列表、章节、逐字稿。待办列表通常是结构化的，每条带责任人、内容、有时带截止时间。

Agent 拿到待办列表后，逐条调 `lark-cli task +create --name "xxx" --assignee "ou_xxx" --due "2026-07-15"` 创建任务。如果责任人识别不准（逐字稿里只有名字没有 user_id），需要先调 `lark-cli contact +search-user` 把名字解析成 open_id。

一个实用做法：所有从纪要生成的任务，描述里自动带上纪要文档链接和会议名称。这样后续做任务时一键跳回上下文。

## 38. 用飞书 CLI 写的飞书文档导出 PDF 会多出 AI 生成水印吗

这个问题在群里被问过多次。答案是：取决于飞书平台的水印策略，CLI 本身控制不了。

飞书对 AI 生成或 AI 编辑过的文档，可能在导出 PDF 时标注"内容由 AI 生成"或类似水印。这是平台层面的合规要求，CLI 调的是飞书的 API，API 怎么标记就怎么标记，没有开关让 CLI 绕过。

如果你确实需要无水印的 PDF，有几个思路：文档生成后人工编辑一下（哪怕只是改一个字），让平台判定为"人工修改过"；或者用 CLI 导出 markdown 再用其他工具转 PDF，绕开飞书的导出链路。

但本质上，AI 生成的内容标注来源是行业趋势，对抗它不是长久之计。接受这个标记，把精力放在让 AI 生成的内容质量足够高，让人看到标记也不觉得水。

## 39. 怎么让 Claude Code 直接操作你的飞书

这是第 4 层 AI Agent 开发者最关心的问题。Claude Code 原生支持调终端命令，所以接入飞书 CLI 很顺。

步骤：先在机器上装好飞书 CLI，`npx @larksuite/cli@latest install`。然后配置应用凭证，`lark-cli config init`。再登录授权，`lark-cli auth login --recommend`。最后装 Skills 包，`npx skills add larksuite/cli -y -g`，让 Claude Code 能理解飞书 CLI 的命令结构。

装完之后，你在 Claude Code 里直接说"帮我查今天的日程"，它会自己调 `lark-cli calendar +agenda --as user` 执行，拿结果给你。

Claude Code 调 CLI 的优势在于它是多轮对话，能根据返回结果动态决定下一步。第一调失败了它会看报错自己修正参数重试，这种自适应能力是固定脚本做不到的。

安全上，建议第一次跑任何有副作用的操作都加 `--dry-run`。Claude Code 很强，但它也会犯错，预览再确认是稳妥的做法。

## 40. 飞书 CLI 自动化到底能省多少时间

拿周报这个场景做量化对比。

手动流程：打开飞书，切到日历翻这周每天的会议，切到任务清单看每个人的进度，切到多维表格拉项目数据，新建文档把数据粘进去排版，切到群聊发链接。全流程 60 到 90 分钟。

CLI 自动化：对 Agent 说一句话，它自己拉日历、任务、多维表格数据，生成文档，发群。你的参与只在最后确认 dry-run 预览那一下。全流程 5 到 10 分钟，其中你实际花的时间 2 到 3 分钟。

效率提升不是简单的"90 分钟变 5 分钟"。更关键的是，这 90 分钟里大部分是低价值的重复操作（切窗口、复制粘贴、排版），自动化后你的时间花在确认内容质量上，这是高价值的判断。省下来的时间可以用来做真正需要人思考的事。

不同场景的省时幅度不同。批量操作、定时任务、数据汇总类场景，省时幅度最大，通常能到 10 倍以上。需要判断和创意的场景，省时幅度小一些，但也能省一半以上。

## 41. 多维表格数据变动怎么让 AI 实时感知到

两种方式。

第一种，多维表格自带的自动化。多维表格有原生的"自动化"功能，可以配置"当记录被修改时触发"，触发动作可以是发消息、调 webhook。这是最轻量的方式，不用写代码。

第二种，定时轮询。Agent 每隔一段时间调 `lark-cli base +records-list --filter "LastModifiedTime > xxx"` 拉最近变动的记录，做 diff 后处理。适合自动化功能覆盖不到的复杂触发逻辑。

实时性要求高的场景，用第一种，原生自动化几乎是实时的。实时性要求不高的（比如每小时同步一次），用第二种更灵活，因为你可以在轮询逻辑里加任意判断。

注意一个限制：多维表格的仪表盘目前 CLI 不支持读取。如果你想做"仪表盘数据变了通知你"，目前做不到，只能监控底层数据表。

## 42. 哪些办公场景最适合先用飞书 CLI 自动化

选场景的判断标准有三条：重复性高、规则清晰、每次手动做费时间。三条都满足的就是好场景。

最适合先做的几类：报表类（日报、周报、月报，数据固定格式固定）、通知类（CI/CD 结果、文档变更、任务到期）、汇总类（多源数据拉到一起生成文档或表格）、批量类（批量创建、批量更新、批量导出）。

不建议先碰的几类：需要创意输出的（写文案、做设计）、需要人际沟通判断的（回复客户投诉、协调冲突）、数据源不稳定或格式随意的（处理自由格式邮件、整理群聊讨论）。这些场景 Agent 能帮一部分忙，但自动化程度低，投入产出比不高。

新手建议从"每天都要做的重复事"入手。你每天花 20 分钟以上做的同一件事，大概率值得自动化。先跑通一个，建立信心和方法论，再往复杂场景扩展。
