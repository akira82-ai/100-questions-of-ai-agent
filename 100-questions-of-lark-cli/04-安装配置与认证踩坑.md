# 第四章 · 安装配置与认证踩坑

## 58. 让 AI 操作你的飞书到底安不安全

这是每个第一次接触飞书 CLI 的人都会问的问题，也是阻碍很多人迈出第一步的心理门槛。

先说结论：安全性是可以保障的，但前提是你理解了机制并按规矩操作。

飞书 CLI 的安全模型有三层防线。第一层是 OAuth 授权，CLI 只能做你明确授权过的事，没授权的业务域它碰都碰不了。第二层是 dry-run 预览，有副作用的操作可以先看后发，发现不对就不执行。第三层是飞书侧的权限体系，文档、群聊、日历都有独立的权限控制，CLI 的操作受这些权限约束。

真正需要警惕的是你授了什么权限，CLI 本身没问题。`lark-cli auth login --recommend` 会一次性开一大堆读写权限，方便是方便，但如果你只打算读文档不打算写，开一堆写权限就是给自己挖坑。社区里有人提过，`--recommend` 默认开通几乎所有功能的读写权限，越权风险不小，建议至少提供一个只读的全局推荐配置。

务实的安全策略分三步。起步阶段只用 user 身份操作自己的数据，bot 身份等做多人共享时再开。权限按需授予，能只读就别开读写，能用单业务域权限就别开全量。生产环境用最小权限原则，定期跑 `lark-cli auth status` 审一遍授权范围，不需要的 scope 关掉。

AI 会不会乱发消息、乱删文档？只要你在 prompt 里告诉 Agent"有副作用的操作先 dry-run 确认"，养成这个习惯，绝大多数事故都能避免。CLI 的设计哲学就是人类负责装好和授权，执行环节的可控性比你想的要强。

## 59. 为什么 403 是飞书 CLI 用户最常见的报错

用过飞书 CLI 的人对 `Permission denied [230027]` 这个报错都不会陌生。它不是 CLI 的 bug，是飞书开放平台的权限拦截。

触发 403 的原因无非三类：应用没开对应权限、权限开了但没重新授权刷新、用了错误身份操作。

第一类最常见。你在开发者后台建了应用，默认只开了基础权限，想发消息发现报 403，想读文档发现报 403。解决方法直奔飞书开放平台，进应用 → 权限管理，把需要的 scope 打开，然后重新跑 `lark-cli auth login` 让授权刷新。

第二类容易被忽略。权限在后台改了，但 CLI 本地缓存的 token 还携带旧权限，必须重新登录才会生效。有人反复 `lark-cli auth login` 无数遍还是不行，就是因为没在后台先开权限，只在 CLI 侧反复登录，方向搞反了。

第三类是身份用错了。用 bot 身份查个人日历，用 user 身份做 CI/CD 通知，都会出权限问题。遇到 403 第一步先确认你用的是 `--as user` 还是 `--as bot`，再看报错信息里的 identity 字段。

排查 403 的标准流程：先跑 `lark-cli doctor` 看网络和凭证状态，再跑 `lark-cli auth status` 看授权范围，对照你执行的命令需要什么 scope，缺什么开什么，改完权限重新登录。

## 60. 机器人装好了发消息不回复怎么排查

这是新手最崩溃的时刻之一：CLI 装好了，应用创建了，机器人配了，结果群里 @ 它毫无反应。

排查路径从近到远。

第一步看机器人在不在群里。打开群设置 → 群机器人，确认你的 bot 在成员列表里。bot 不在群里，发消息收不到事件，自然不回复。

第二步看事件订阅开了没。进入开放平台 → 应用 → 事件订阅，确认 `im.message.receive_v1` 这个事件已开通。没开这个事件，bot 根本不知道有人 @ 它。

第三步看 mention 策略。有些框架（比如 OpenClaw）的频道配置里有 mention 策略选项，要确认是不是配置成了"只响应 @ 消息"。如果配成不响应任何消息，bot 就彻底沉默。

第四步看网关在不在运行。如果你用的是 OpenClaw 或 Hermes 这类运行时框架，确认 Gateway 进程还活着，监听端口正常。Gateway 挂了，事件推过来也没人处理。

第五步看日志。跑 `lark-cli doctor` 检查飞书关键域名连通性，看 Gateway 日志有没有收到事件、有没有处理报错。日志是最诚实的，它能告诉你问题卡在哪一层。

大多数"机器人不回复"的问题，走完这五步基本能定位。

## 61. 跳过 config init 这一步后面为什么会全崩

`lark-cli config init` 是所有后续命令的地基，跳过它等于没打地基就盖楼。

config init 做的事情：在飞书开放平台创建一个应用（或绑定已有应用），生成 app_id 和 app_secret，把这些凭证存到本地配置文件。后面所有命令的鉴权都依赖这套凭证。

如果跳过这一步直接跑别的命令，你会遇到各种莫名其妙的错误。"鉴权失败""找不到应用""无法连接飞书服务"这些报错的根因都是 config init 没做。

有一种常见翻车场景：你在 A 电脑上 config init 过，把配置文件拷到 B 电脑，但 B 电脑的 CLI 版本不同或配置路径不一样，导致 CLI 读不到正确配置。解法是在 B 电脑上重新 config init。

另一个坑是 config init 用的账号和后续 auth login 用的账号不一致。init 时用账号 A 创建了应用，login 时用账号 B 扫码授权，会报"当前账号无飞书 CLI 使用权限"。原因是你登录的账号对这个应用没有管理权限。解法是确保 init 和 login 用同一个飞书账号。

## 62. 为什么你的 chat-id 总是填不对

chat-id 是飞书群聊的唯一标识，形如 `oc_7f363f14d647ead384b55749ea581fcf`。填错了，消息发到别的地方或直接报错。

最容易犯的错是从群聊名称猜 chat-id，或者从某个截图里抄一段像 ID 的字符串。这些都是靠不住的，chat-id 只能从可靠来源获取。

可靠来源有三个。第一，`lark-cli im +chat-list` 命令列出你加入的所有群和它们的 chat-id，这是最直接的方式。第二，从飞书群链接里提取。群链接里包含的 token 就是 chat-id，格式是 `oc_` 开头的一长串。第三，从消息事件里拿。如果你配了事件订阅，每条消息事件都携带所在群的 chat-id。

填错 chat-id 的后果可大可小。轻则报错发不出去，重则消息发到了另一个名字相似的群。曾经有人想给研发群发周报，chat-id 填错一个字符，周报发到了客户群。dry-run 能在这种场景救你一命，发消息前先预览目标群和内容。

## 63. chat-id 怎么从飞书 URL 里正确拿到

从飞书群链接里提取 chat-id 有讲究，因为链接里混杂了其他参数。

飞书群链接的格式通常是 `https://feishu.cn/chat/thread_id=oc_xxx` 或者通过群设置里"复制群链接"得到的一段 URL。真正的 chat-id 是 `oc_` 开头的那一串，长度固定。

常见的错误提取方式：把整个 URL 当 chat-id 传进去，把 URL 里的 thread_id 参数名也一起复制了，或者复制时多带了空格和换行符。这些都会导致 CLI 报 invalid param。

正确的提取方式：从 URL 里定位 `oc_` 开头的字符串，到下一个 `&` 或 URL 结尾截止，中间不要有空格。拿不准就用 `lark-cli im +chat-list` 查一遍，对比一下。

有些群的 URL 里 chat-id 出现在 path 部分，有些出现在 query 参数里，格式不完全统一。最保险的做法是 CLI 命令查群列表，拿到准确的 chat-id 再用。

## 64. config init 被系统拒绝怎么排查

有人在 Windows 上跑 `lark-cli config init`，报错 `dial tcp ... actively refused it`，但 curl 访问同一个域名又能通。

根因是代理配置的差异。Windows 上 curl 默认走系统代理设置，但飞书 CLI 用 Go 语言开发，Go 只读 `HTTP_PROXY` 和 `HTTPS_PROXY` 这两个环境变量。系统代理开了但环境变量没设，CLI 就直连飞书域名，结果被网络环境拒绝。

排查方法：在 CMD 里跑 `netsh winhttp show proxy` 看系统代理状态。如果配了代理，设上环境变量：

```cmd
set HTTPS_PROXY=http://your-proxy-host:port
lark-cli config init
```

Linux 和 macOS 同理，检查 `echo $HTTPS_PROXY` 有没有值。

另一个常见原因是公司防火墙拦截。有些企业的安全策略会拦 `accounts.feishu.cn` 和 `open.feishu.cn` 这两个关键域名。解法是让 IT 把这两个域名加白名单。跑 `lark-cli doctor` 能快速判断关键域名通不通。

还有一种被拦的情况是安全软件阻止。Windows 上 Cylance 这类端点安全软件可能阻止 PowerShell 跑 lark-cli，需要 IT 把 lark-cli 加入白名单。

## 65. --recommend 推荐的权限不够用怎么办

`lark-cli auth login --recommend` 的设计是让你一次开够常用权限，免得后面频繁补授权。但有两种情况会觉得它不够用。

第一种是你需要某个特定业务域的权限，但 `--recommend` 没包含。比如你想操作 Wiki 知识库，默认只开了只读权限，增删改都不行。这时候要手动去开放平台把 `wiki:wiki` 这类 scope 开起来，然后重新 auth login 刷新授权。

第二种是你只想开一部分权限，`--recommend` 开太多。有人反馈它默认开通几乎所有功能的读写权限，越权风险大，企业管理员甚至会禁止某些高危权限（比如删除云文档）。这种情况用 `--scope` 参数精确指定你要的权限：

```bash
lark-cli auth login --scope "docs:document.content:read docx:document calendar:calendar:read"
```

社区里有人提过一个建议：官方应该提供一个 `--recommend-readonly` 选项，默认只开只读权限，写权限按需再开。这个建议被标记为高优先级需求，但截至当前版本还没有落地。

权限管理的最佳实践是按业务域精确授权。只做日历自动化就只开日历权限，只做文档读写就只开文档权限。别图省事开全量，出事了缩小攻击面都来不及。

## 66. 公司不是 feishu.cn 域名怎么初始化

飞书国内版和国际版 Lark 是两套独立的系统，域名和 API 端点完全不同。国内版用 `feishu.cn` 和 `open.feishu.cn`，国际版用 `larksuite.com` 和 `open.larksuite.com`。

如果你的公司用的是国际版 Lark，config init 时需要告诉 CLI 用国际版域名。具体做法是初始化时指定品牌为 lark 而不是 feishu，CLI 会自动切换到国际版端点。

有人用 Claude Code 接飞书 CLI，授权了但读不了多维表格，排查发现数据在 Lark 环境（larkoffice.com）而 CLI 默认连 Feishu API（open.feishu.cn），两个域名不通所以读不到。解法是确认 CLI 配置的品牌和你的飞书版本一致。

如果你的公司用了私有化部署的飞书，域名是企业自己的，config init 时需要指定自建域名。具体参数看 CLI 的帮助文档 `lark-cli config init --help`，一般是通过自定义 endpoint 参数配置。

混合场景也常见：公司主体在国内版飞书，但某些跨国协作的团队用国际版 Lark。两边的 token、应用、数据完全不通用。跨版本操作需要分别初始化两套 CLI 配置。

## 67. Node.js 版本不对会引发哪些奇怪报错

飞书 CLI 对 Node.js 版本有要求，版本不对会出各种诡异的错误，而且报错信息往往不直接指向版本问题。

低于要求的版本会出现：安装时报语法错误，运行时报 `SyntaxError: Unexpected token`，某些命令静默退出没有输出，性能异常缓慢。这些都是新版 JavaScript 语法（比如可选链 `?.`、空值合并 `??`）在旧版 Node.js 上不支持导致的。

推荐用 nvm 管理 Node.js 版本，切换方便。装 CLI 前先确认版本：

```bash
node --version
```

如果版本太低，用 nvm 升级：

```bash
nvm install 22
nvm use 22
```

装完 nvm 但运行 `nvm` 提示 command not found 是另一个高频坑。原因是 nvm 没被加载到当前 shell。解法：

```bash
export NVM_DIR="$HOME/.nvm"
[ -s "$NVM_DIR/nvm.sh" ] && \. "$NVM_DIR/nvm.sh"
```

把这两行写进 `~/.zshrc`（macOS）或 `~/.bashrc`（Linux），让它永久生效。

内网环境装 CLI 报错通常是 npm registry 问题。换成国内镜像源：

```bash
npm config set registry https://registry.npmmirror.com
```

## 68. Windows PowerShell 传 JSON 参数为什么报错

这是 Windows 用户碰到的最普遍的平台兼容性问题。

在 PowerShell 里给 `--params` 或 `--data` 传 JSON 字符串，CLI 报 `invalid JSON format`，但同样的命令在 macOS 或 Linux 上完全正常。

根因在 PowerShell 的引号处理机制。PowerShell 对单引号和双引号的解释和 Bash 不同，JSON 里的双引号会被 PowerShell 各种转义和重组，传到 CLI 时已经面目全非。

这个问题的波及面很广。不只 `--params`，包括 `--data`、`--markdown`、带复杂 JSON body 的 `reply_elements` 都会中招。有人测试过，多行 `--markdown` 参数在 Windows 上会只写入第一行。

几种绕过方式。第一种，把 JSON 存到文件，用 `--params @file.json` 的方式传。第二种，在 PowerShell 里用变量赋值再传，`$json = '{"key":"value"}'` 然后 `--params $json`，但这个方式有人反馈仍然失败。第三种，不在 PowerShell 里跑，换用 CMD 或 WSL。第四种，用 Node.js 写个脚本，通过子进程的方式调用 CLI，绕过 shell 的引号处理。

这个问题的本质是 Windows shell 生态和 Unix 风格 CLI 工具之间的兼容裂缝，不只飞书 CLI 碰到，所有在 Windows 上跑的命令行工具或多或少都有类似问题。

## 69. 授权链接点开显示已失效或被拒绝怎么办

授权流程走到一半链接失效，通常有几种原因。

最常见的是时间窗口过期。OAuth 授权链接有有效期，超过一定时间（通常几分钟到几十分钟）就失效。你在终端跑 `lark-cli auth login`，CLI 打印一个授权 URL，你复制到浏览器打开，中间喝了杯咖啡回来点授权，链接已经过期了。解法是重新跑 auth login 拿新链接。

第二种是 app secret 被修改过。你在开放平台重置了 app secret，之前签发的授权请求全部失效，包括正在进行的授权流程。报错 `device authorization failed: The client secret is invalid`。解法是重新 auth login，用新的 secret 走一遍。

第三种是账号没权限。打开授权页面提示"当前账号无飞书 CLI 使用权限"，原因是你的飞书账号对这个应用没有管理权限。应用是别人建的，你不是管理员也不是协作者。解法是让应用创建者把你加为协作者，或者你自己 config init 新建一个应用。

第四种是授权页面能打开但点"授权"报错。可能是开发者后台的应用版本没有发布，或者权限配置不完整。进开放平台确认应用状态是"已发布"，权限配置无误。

## 70. 机器人不在群里发消息报错怎么解决

用 bot 身份往群里发消息，报 `Permission denied` 或 `Bot not in chat`，根因就是 bot 不在目标群里。

飞书的安全策略是 bot 只能往自己加入的群发消息。你拿 chat-id 往一个没加 bot 的群发消息，直接被拦截。

解法是把 bot 拉进群。打开群设置 → 群机器人 → 添加机器人，选你的应用机器人加进去。加完再发消息就能通。

如果加了 bot 还是报错，排查几个点。第一，确认 chat-id 对应的就是你刚加 bot 的那个群，别搞混了。第二，确认应用有 `im:message:send_as_bot` 权限，没这个权限 bot 发不了消息。第三，如果是企业群或有特殊安全策略的群，可能需要群管理员审批才能加 bot。

bot 加群这个动作本身也可以用 CLI 完成，`lark-cli im +chat-list` 相关命令能管理群成员。但首次操作建议手动加，确认流程跑通后再考虑自动化。

## 71. 换账号登录时残留缓存怎么清干净

换飞书账号登录是最容易踩缓存坑的场景。你以为换了账号，CLI 实际上还在用旧账号的凭证。

残留缓存的来源有几个。`~/.lark-cli/` 目录下的配置文件存了应用凭证和 token，keychain（macOS）或凭据管理器（Windows）里存了加密凭证，环境变量里可能设了旧的 token。

彻底清理的步骤：

```bash
# 清配置目录
rm -rf ~/.lark-cli/

# macOS 清 keychain（在"钥匙串访问"app 里删 lark-cli 相关条目）
# 或用命令行
security delete-generic-password -s "lark-cli" 2>/dev/null

# 清环境变量（如果之前通过环境变量设过凭证，全部清掉）
unset LARK_CLI_NO_PROXY
unset HTTPS_PROXY
unset HTTP_PROXY

# 重新初始化
lark-cli config init
lark-cli auth login
```

如果你用了多 profile（多个工作空间），每个 profile 有独立的配置目录，清的时候确认清的是对的那份。

多人共用一台机器更容易出残留问题。A 用完没清干净，B 接手后用的还是 A 的凭证，操作出来的文档和消息归属全是 A。生产环境建议每人独立账户，或者统一用 bot 身份加文件存储凭证，不依赖个人 keychain。

## 72. 飞书 CLI 创建的文档自己还要申请权限怎么办

用 CLI 创建了一份文档，自己点开发现要"申请编辑权限"，这种荒谬感是 bot 身份的典型副作用。

原因在第三章讲过：bot 身份创建的文档，owner 是机器人应用，不是你个人。应用是应用，你是你，对飞书权限系统来说是两个独立的主体。

三种解法。

最直接的解法是改用 user 身份创建文档。命令里加 `--as user`，创建出来的文档 owner 直接是你个人，不存在权限申请问题。适合个人使用的场景。

第二种是创建后主动授权。bot 创建完文档，立刻调权限接口给自己加权限。CLI 有相关命令可以管理文档权限成员。适合 bot 必须创建但你需要管理权的场景。

第三种是升级到最新版 CLI。新版本在检测到同时有 user 和 bot 身份时，会自动处理权限授予，bot 创建文档后自动给当前 user 加权限。如果你还卡在这个问题，先升级试试。

多人协作场景更复杂。bot 创建的文档默认只有 bot 有权限，其他人想编辑都要单独授权。解法是把文档放到共享文件夹或知识库节点下，继承该位置的权限策略。或者创建后批量调权限接口给团队成员加权限。

权限这件事，核心原则是先想清楚"谁需要什么权限"，再决定用 bot 创建后授权，还是用 user 创建后共享。

## 73. Base 附件下载报 403 是哪里出了问题

多维表格的附件字段存了文件，你想用 CLI 批量下载，结果报 HTTP 403。

多维表格开了高级权限（advanced permissions）是触发这个问题的关键条件。高级权限下，即使你有表格的编辑权限，附件字段的文件下载可能仍然被拦。

附件下载的路径有多条，权限要求各不相同。`lark-cli base +record-download-attachment` 是专门为多维表格附件设计的命令，理论上最对路。但在高级权限模式下，它可能仍然 403，因为底层 API 需要一个 `extra` 参数来声明 bitable 权限上下文，而 CLI 在某些版本里没正确传这个参数。

有人试过 `lark-cli drive +download` 和 `lark-cli docs +media-download`，这两条路径对同一个 file_token 返回 400 而不是 403，因为它们调的底层端点根本不支持多维表格附件所需的 `extra` 参数。

当前的务实战法分几步。第一步，确认你的 scope 包含 `base:record:read` 和 `drive:file:download`，缺了先补。第二步，升级到最新版 CLI，官方在持续修这类问题，新版会打印 log_id 方便定位。第三步，如果 CLI 还是搞不定，绕过它直接调飞书原生 OpenAPI 的多维表格附件下载端点，手动带上 `extra` 参数。第四步，报 issue，附上 log_id 让官方排查。

## 74. 多维表格 select 字段被 AI 擅自新建选项怎么办

让 AI 通过 CLI 往多维表格写数据，select 类型字段（单选/多选）被 AI 自作主张新建了一堆选项，原本的选项体系被污染。

这是 AI 理解灵活性带来的副作用。飞书多维表格的 select 字段在写入时，如果传了一个不存在的选项值，API 默认会新建这个选项。AI 看到字段是 select，又不确定有哪些选项，干脆随便写一个，结果选项越来越多。

根治方法是在 prompt 里明确约束 AI。告诉它 select 字段只能用现有选项，不确定有哪些选项时先查（`lark-cli base +fields-list` 能列字段定义和可选值），不要擅自新建。

工程层面的兜底是写数据前做校验。你的脚本先查字段定义，拿到合法选项列表，写入时过滤掉非法值。CLI 的 dry-run 能帮你看请求参数，发现选项名不对先拦住。

select 字段的选项本质是一种受控词汇表，不应该被写入方随意扩展。如果你搭的是多人协作的多维表格，建议在表结构设计阶段就定好选项体系，写入侧做好校验，别依赖 AI 的"智能"去猜。

## 75. Wiki 知识库现在有哪些功能缺口要绕开

Wiki 是飞书 CLI 覆盖最不完整的业务域之一，有些能力缺失需要提前知道，免得方案设计到一半才发现走不通。

早期版本 CLI 对 Wiki 只开了只读权限，增删改都做不了。手动去开发者平台把 `wiki:wiki` 开起来后，基础增删改才能用。这是权限层面的坑，不是功能缺失。

功能层面的缺口包括：创建新节点的能力曾经缺失，后来补上了；删除节点在某些版本仍不稳定；直接上传文件到 Wiki 不支持，上传只能到 Drive，然后手动移到知识库；Wiki 和 Drive 的自动集成弱，上传文件后不会自动归类到 Wiki 目录；批量操作能力不足，批量移动和创建节点的命令覆盖不全。

还有一个隐性缺口：Wiki 节点的元数据不完整。`lark-cli wiki +node-list` 返回的节点数据里没有 `created_time` 和 `modified_time` 字段，你想按时间排序或筛选最近修改的文档，做不到。

Wiki 的原生 `sub-page-list` 块也有读写不一致的问题。`docs +fetch` 能读出来，但 `docs +update` 写入时会被当成不支持标签直接丢弃。Wiki 知识库的 hub 页如果想用 CLI 维护，写完发现 hub 页变空了。

绕开这些缺口的方法是分而治之。能用 CLI 的部分用 CLI，缺失的部分用原生 OpenAPI 补，实在不行的部分接受手动操作。别指望 CLI 一次性把 Wiki 全覆盖了，它还在演进中。

## 76. 创建的文档移不进知识库是什么原因

用 CLI 创建了一份文档，想把它移到知识库的某个目录下，发现移不进去或者报错。

最常见的原因是身份不匹配。文档是 user 身份创建的，但移到知识库需要应用对知识库有写权限。或者文档是 bot 创建的，bot 对知识库节点没有管理权限。

第二种原因是知识库的权限策略。知识库可以配置"谁能往里添加内容"，如果策略限制只有管理员能添加，普通用户或 bot 都移不进去。解法是让知识库管理员调整策略，或者用管理员身份操作。

第三种原因是文档归属。文档在"我的空间"里时移动相对自由，但文档已经在某个共享文件夹下时，移动会受源文件夹权限策略约束。

实际操作中，用 CLI 移文档到知识库的命令是 `lark-cli wiki +node-create`，传入 obj_token（文档 token）和 obj_type（文档类型），指定 parent_node_token 挂到哪个目录下。移不进去时，先确认这几个参数对不对，再确认操作身份对知识库有没有 write 权限。

绕开方法：先创建一个知识库节点（空文档），再用 docs +update 往里写内容。这样从一开始文档就在知识库里，避免了后续移动的权限问题。

## 77. md 导入飞书文档出现一堆空行怎么解决

用 CLI 把 Markdown 写入飞书文档，打开一看段落之间全是多余的空行，原本紧凑的排版变得稀稀拉拉。

根因在 Markdown 到飞书文档 block 结构的转换逻辑。飞书文档的底层是 block 数组，每个段落、标题、列表项都是一个独立 block。Markdown 里的空行在转换时可能被当成 block 分隔符，导致额外插入空 block。

几种缓解方式。第一，写入前清理 Markdown 源文件，去掉多余的连续空行，只保留必要的单行间隔。第二，用 `docs +update` 的增量模式，逐 block 写入而不是全量覆盖，减少结构重组带来的副作用。第三，写入后用 `docs +fetch` 拉回来检查实际 block 结构，发现多余的空 block 用 update 命令删除。

更深层的问题是 Markdown 和飞书文档的结构不对等。Markdown 是线性文本，飞书文档是嵌套 block，两者之间的转换天然有损。列表嵌套引用块、引用块嵌套列表、多层嵌套列表这些复杂结构，转换时尤其容易出错。

实际操作建议：简单文档用 Markdown 写入问题不大，复杂结构的文档（深度嵌套列表、混合引用）最好直接用 block API 操作，不要走 Markdown 这层转换。

## 78. 飞书 CLI 写出来的文档有序列表序号变成纯文本

用 Markdown 写了一份文档，有序列表格式标准（`1. 2. 3.`），用 CLI 写入飞书后打开一看，序号变成了纯文本"1."，不是飞书原生的有序列表 block。

这也是 Markdown 转换层的损耗。飞书的有序列表是专门的 block 类型，有自动编号、缩进控制等原生能力。Markdown 的有序列表语法在转换时，如果解析器没正确识别，会被当成普通段落文本，前面的"1."就成了内容的一部分。

触发这个问题的常见写法：列表项之间有空行、列表嵌套了其他 block、Markdown 用了非标准缩进。这些都可能让解析器判断失误。

规避方式：写 Markdown 时列表项之间不留空行，保持紧凑；嵌套列表用标准的 4 空格或 2 空格缩进，不要混用；有序列表统一用 `1.`（飞书会自动算序号），不要写 `1. 2. 3.` 然后指望它保持。

写入后检查：`docs +fetch` 拉回文档的 block 结构，看列表 block 的 type 是不是 `ordered_list` 或类似类型。如果变成了 `text` 类型，说明转换出了问题，手动调整源 Markdown 或者用 block API 直接写列表 block。

## 79. callout 高亮块颜色和 mermaid 图为什么读不出来

Markdown 里有 callout 高亮块和 mermaid 流程图，用 CLI 写入飞书后这些特殊语法全部丢失或变成乱码。

原因很直接：飞书文档的 block 体系和 Markdown 扩展语法不完全对应。callout 是某些 Markdown 方言（如 Obsidian）的扩展，mermaid 是代码块的特殊语言标记，飞书文档的原生 block 类型里没有直接对应的实现。

callout 在飞书里有类似的高亮块 block，但 CLI 的 Markdown 解析器可能不认这个语法，写入时直接丢弃。mermaid 图更彻底，飞书文档没有原生 mermaid 渲染能力，写入的代码块只会显示成纯文本代码。

解法分两种。第一种，对这些特殊内容不要用 Markdown 写入，改用 block API 直接操作。飞书的高亮块（callout block）有专门的 block type，用 API 创建能正确渲染。mermaid 图如果想展示，只能截图后作为图片 block 插入，或者引导用户在飞书里手动用画板画。第二种，接受这些语法的丢失，在 Markdown 源文件里把 callout 改成普通引用块 `>`，把 mermaid 图换成文字描述或截图链接。

这反映了一个普遍规律：Markdown 和飞书文档不是一一对应的关系，Markdown 的某些扩展语法在飞书体系里没有落点。用 CLI 做文档自动化时，先搞清楚目标文档的结构，能用原生 block 体现的就用 block API，只有纯文本内容才适合走 Markdown 这层。

## 80. 用了一个月飞书 CLI 最后悔没早知道的几件事

第一条，user 和 bot 身份的坑要第一天就搞清楚。这个坑踩了的人不在少数，创建的文档自己没权限、发的消息归属混乱、token 互相踢，根因都是身份没分清。第一天花十分钟理解这两种身份的区别，能省后面几天的排错时间。

第二条，dry-run 是肌肉记忆不是可选操作。任何有副作用的命令，第一次跑先 dry-run。换了参数、换了目标、换了数据源，再 dry-run 一次。这不是保守，是专业。发错群、写错表、删错文档这些事故，dry-run 能拦掉绝大多数。

第三条，权限按需给，别图省事开全量。`--recommend` 方便但风险高，企业管理员可能直接禁某些高危权限。按业务域精确授权，定期审一遍 scope，关掉不需要的。

第四条，Windows 用户准备好绕路。PowerShell 的 JSON 参数问题是系统级的，不是 CLI 的 bug。要么习惯用文件传参，要么用 WSL，别和 shell 较劲。

第五条，keychain 别当成唯一存储。macOS 升级、换账户、重装系统都可能让 keychain 出问题。重要环境用文件存储或多备份一份凭证，别等到 keychain 挂了才发现没有 plan B。

第六条，Wiki 和文档格式转换的坑提前知道。Wiki 覆盖不完整，Markdown 转飞书 block 有损耗，复杂结构别走 Markdown。把这些限制写在方案设计阶段，免得开发到一半发现走不通。

第七条，CLI 版本勤升级。飞书 CLI 迭代很快，7 月前 9 天发了 6 个版本。很多你踩的坑可能新版已经修了，升级是最便宜的排错手段。跑 `lark-cli update` 比跑 `lark-cli doctor` 还省事。

第八条，社区和 issue 是宝藏。你踩的坑大概率别人踩过，GitHub issue 里搜一搜往往能找到答案或 workaround。找不到就提 issue，官方响应速度还行。

用 CLI 这件事，前期有学习曲线，踩一两周坑之后会进入顺畅期。关键是别在踩坑阶段就放弃，把高频坑趟一遍，后面就是效率红利。
