# 第三章 · 三层架构与安全机制

## 43. user 和 bot 身份混用为什么是飞书 CLI 最大的坑

飞书 CLI 有两种执行身份，user 和 bot。user 是通过 OAuth 登录的某个具体的人，bot 是飞书应用本身。两种身份的权限范围、能访问的数据、操作归属完全不同。

最大的坑在于：用错身份会得到完全不符合预期的结果，而且报错往往不直观。

用 bot 身份创建文档，文档 owner 是机器人不是你，你想编辑还得自己申请权限。用 bot 身份查日历，只能查应用自己创建的日程，查不到你个人的会议。用 user 身份在群里发消息，消息显示的是你的名字，但你离开公司后这条消息的归属就乱了。

判断用哪个身份的原则：操作涉及个人数据（日历、任务、邮件、私人文档）用 user；操作是系统行为（CI/CD 通知、批量创建、服务账号）用 bot。命令里用 `--as user` 或 `--as bot` 切换。

第一次配 CLI 时建议先用 user 身份跑通，因为大部分高频场景（查日程、读文档、发消息）user 身份都能覆盖。bot 身份更适合做后台服务和多人共享场景。

## 44. 为什么说 dry-run 是 AI 操作飞书的后悔药

AI Agent 调 CLI 执行有副作用的操作时，最大风险是它理解错了你的意思就真发了。比如你说"把周报发到研发群"，它理解成发到产品群，消息发出去了收不回来。

dry-run 是防这层风险的机制。在发消息、创建文档、修改记录这类有副作用的命令上加 `--dry-run`，CLI 会把要执行的请求完整打印出来，包括目标 chat-id、消息内容、API 参数，但不真正发送。你确认无误后去掉这个参数重跑，才真正执行。

推荐的使用节奏：第一次跑任何有副作用的命令都先 dry-run。换了数据源、改了模板、换了目标群，再 dry-run 一次。养成这个习惯，能避免绝大多数"发错群""发错人""数据写错地方"的事故。

Agent 自身也能调 dry-run。你在 prompt 里告诉 Agent"发消息前先 dry-run 让我确认"，它会自动在第一次执行时带上这个参数。

## 45. 让 AI 发消息前先给你看预览是怎么做到的

预览机制底层就是 dry-run，但应用层有几层封装。

最基础的一层，命令行加 `--dry-run` 参数，输出 JSON 格式的请求预览，包含 endpoint、params、body。这层适合技术用户直接看。

第二层，Agent 把 dry-run 的 JSON 结果解析成人话。"我要往 oc_xxx 群发一条文本消息，内容是'本周周报已完成'，确认执行吗？"这层适合非技术用户。

第三层，交互卡片预览。对于发卡片消息这种复杂内容，Agent 可以先生成卡片 JSON，用 dry-run 预览结构，确认后真发。卡片的视觉预览飞书客户端不支持命令行侧预览，要看效果只能真发到测试群。

最高级别的预览是"发到测试群确认再转发"。把目标群换成一个只有你自己的测试群，真发一次看效果，满意了再改 chat-id 发到正式群。这是最稳但也是最慢的方式，适合重要消息。

## 46. lark-cli doctor 能帮你查出哪些配置问题

`lark-cli doctor` 是 CLI 内置的一键诊断工具，跑一遍能扫出大多数常见配置问题。

它检查的核心项：网络连通性，看你的机器能不能访问飞书的关键域名（accounts.feishu.cn、open.feishu.cn 这几个）。这些域名被防火墙拦了或被代理劫持，CLI 会各种莫名报错，doctor 能第一时间定位。

凭证状态，看你的 user token 和 bot token 有没有过期，keychain 里凭证存得对不对。macOS 的沙箱机制偶尔会让 keychain 出问题，doctor 能识别这类异常。

权限覆盖，看你当前授权的 scope 够不够支撑你想跑的命令。比如你想读文档评论但没开 `docs:document.comment` 权限，doctor 会提示缺哪个 scope。

版本状态，看你装的 CLI 版本是不是最新的，有没有重要安全补丁没打。

遇到任何 CLI 报错，第一步跑 doctor。很多看似复杂的问题，doctor 扫一眼就定位了。

## 47. user 身份能干什么 bot 身份干不了的事

user 身份的核心优势是能访问个人维度的数据。

查个人日历，只有 user 身份能看你自己的会议和日程。bot 身份查日历只能查 bot 自己创建的日程，对你的会议一无所知。

读个人邮件，邮件是强个人属性的数据，bot 身份完全访问不了。`lark-cli mail +list` 必须用 user 身份。

查个人任务清单，`lark-cli task +get-my-tasks` 返回的是当前登录用户的任务，bot 身份跑这条命令没意义。

访问私人文档，别人单独分享给你的文档、你个人的草稿，只有你的 user 身份能读。bot 要读这些文档得单独被授权。

发消息显示你的名字，用 user 身份发的消息，群里看到的是你的头像和名字。这对某些场景很重要，比如你以个人身份回复客户、做工作汇报。

简单原则：凡是"我的"开头的操作（我的日历、我的邮件、我的任务、我的文档），都要用 user 身份。

## 48. bot 身份能干什么 user 身份干不了的事

bot 身份的优势在系统级操作和多人共享场景。

7×24 小时不间断运行。user token 有过期问题，长时间不刷新会失效，半夜跑的定时任务容易因为 token 过期挂掉。bot 用的是 tenant_access_token，刷新机制更稳定，适合做常驻服务。

多人共享同一身份。一个团队共用一个 bot，所有人通过这个 bot 操作飞书，权限管理统一。如果用 user 身份，每个人的操作都要各自授权，管理成本高。

CI/CD 和自动化管道。CI/CD 跑完推飞书通知，用 bot 身份最合理，因为这是系统行为不是某个人的行为。用 user 身份推通知，消息显示成某个具体的人，语义上不对。

避免个人数据绑死。bot 创建的文档、发的消息，归属是应用不是个人。员工离职换人，bot 创建的资源不需要迁移。user 身份创建的资源，员工离职后权限和归属会变成麻烦。

跨用户场景。如果你搭一个客服 bot 服务多个用户，bot 身份是统一入口，每个用户的请求都走同一个 bot，后端逻辑简单。

## 49. 飞书 CLI 的三层命令体系怎么选才不会错

三层命令从粗到细：shortcut、API command、Raw API。选层的判断标准是"你要做的事 CLI 封装到什么程度"。

shortcut 层，命令前面带加号，比如 `lark-cli im +messages-send`。这层做了最多抽象：智能默认值、表格输出、dry-run 预览、常见参数组合都内置好了。日常 90% 的操作这层够用。判断标志是"我要做的事属于高频操作"。

API command 层，按业务域和资源组织，比如 `lark-cli calendar events list`。这层和飞书开放平台 API 一一对应，参数完整暴露，没有额外封装。适合 shortcut 覆盖不到但仍是标准 API 的操作。判断标志是"shortcut 没有这条命令但 API 文档里有"。

Raw API 层，用 `lark-cli api GET/POST` 直接调用任意飞书 API。这层没有任何封装，就是帮你处理鉴权和错误格式。适合 CLI 完全没覆盖的边缘 API，或者你需要极度定制参数的场景。判断标志是"这层是唯一选择"。

降级顺序：先查 shortcut 有没有，没有再查 API command，还不行用 Raw API。不要一上来就用最底层，那样等于自己重新造 CLI 已经造好的轮子。

## 50. token 过期了怎么处理才不用每次手动登录

user 身份的 token 过期是最高频的运维痛点。user_access_token 有效期大概 2 小时，refresh_token 有效期大概 30 天但默认安全策略下可能只有 7 天。

要让它持久化，关键是登录时开 `offline_access` scope。开了之后，refresh_token 在有效期内只要被使用过一次，会自动续期，理论上一年内都不用重新扫码登录。`lark-cli auth login --recommend` 会自动带上这个 scope。

如果已经登录但 token 总过期，排查方向有两个。第一，看是不是多环境多 profile 导致 token 没持久化。你在本地登录了，但脚本跑在容器里用了另一套配置，两边各刷各的 token 互相覆盖。第二，macOS 的 keychain 偶尔出问题导致凭证存不住，跑 `lark-cli config keychain-downgrade` 切到文件存储能绕过。

bot 身份的 token 没这个问题。tenant_access_token 是用 app_id 和 app_secret 现签现用的，CLI 内部自动刷新，你感知不到。

## 51. Skills 全局注入会吃掉多少上下文额度

飞书 CLI 的 Skills 包有 26 个，每个 skill 是一份 markdown 文档，告诉 Agent 这个业务域有哪些命令、参数怎么填、什么场景用什么命令。

全局注入意味着所有 skill 文档都塞进 Agent 的系统提示词或上下文里。好处是 Agent 对飞书的全貌都了解，任何业务域的命令都能调。坏处是这些文档加起来体量不小，会占用模型上下文窗口。

实际占用看 skill 包版本和 Agent 的上下文窗口大小。早期版本的 skill 文档写得很详细，全部注入可能占几万 token。新版已经做了精简，但仍然是一个不可忽视的上下文负担。

优化策略有几种。第一，只装你常用的几个 skill。飞书支持选择性安装，比如你只做日历和文档自动化，就只装 lark-calendar 和 lark-doc，不装 lark-base 和 lark-sheets。第二，用 Agent 框架的 skill 加载机制，让 Agent 按需读取 skill 而不是全量注入。第三，文档别写太长，skill 文档越精简上下文占用越少。

社区里有人提过这个痛点。如果你的 Agent 频繁超 context 限制，先看看是不是 skill 全量注入的问题。

## 52. keychain 里的 master key 丢了怎么救

macOS 上 CLI 把凭证存在系统 keychain 里，keychain 由一个 master key 加密。master key 丢了，所有存在 keychain 里的凭证都解不开，CLI 会持续报鉴权失败。

最容易触发这个问题的场景：macOS 升级、重装系统、换了登录账户、keychain 数据库损坏。任何一个发生，master key 可能就找不回来。

救援步骤：先试 `lark-cli config keychain-downgrade`，这个命令会切到文件存储模式，绕过 keychain 直接把凭证存在本地配置文件里。前提是你还能跑 CLI，哪怕是用 sudo 或新账户跑。

如果 keychain-downgrade 也跑不了，只能重置：删掉 `~/.lark-cli/` 配置目录，重新 `lark-cli config init` 和 `lark-cli auth login`，从零开始配。代价是历史凭证全没，但 CLI 的凭证本来就该能随时重建。

预防方法：重要环境别只依赖 keychain。生产服务器用 Linux，凭证存文件；macOS 开发机定期确认 keychain 能正常读；多人共享的环境统一用 bot 身份加文件存储，不依赖某个人的 keychain。

## 53. 飞书 CLI 的 Skills 和 MCP 工具到底什么区别

两者不在一个层面，一个是具体工具，一个是标准协议。

飞书 CLI 的 Skills 是一组 markdown 文档，告诉 Agent 飞书 CLI 有哪些命令、怎么调、什么场景用什么。它的载体是 CLI 本身，Agent 通过执行终端命令来用。Skills 解决的是"Agent 怎么理解 CLI 的命令结构"这个问题。

MCP（Model Context Protocol）是 Anthropic 提出的标准协议，定义了 Agent 调用外部工具的统一接口规范。任何工具只要实现 MCP 接口，任何支持 MCP 的 Agent 都能调用。MCP 解决的是"Agent 和工具之间用什么协议通信"这个问题。

两者关系：CLI 可以被包装成 MCP server，让支持 MCP 的 Agent 通过标准协议调用。事实上社区已经有人在做这件事。

选哪个取决于你的 Agent。Claude Code、Codex 这类能直接调终端的 Agent，用 CLI 最直接。如果你的 Agent 框架只支持 MCP，那把 CLI 包成 MCP server 是必经一步。

生产环境建议优先用 CLI 本身，因为它是飞书官方维护的，更新和覆盖有保证。MCP 包装是社区驱动的，覆盖面和及时性取决于贡献者。

## 54. 怎么查看当前飞书 CLI 到底授了哪些权限

权限透明是安全的基础。飞书 CLI 提供了几种方式查授权状态。

`lark-cli auth status` 显示当前登录状态和已授权的 scope 列表。跑一遍能看到你是以什么身份登录的、token 有没有过期、授权了哪些权限域。

`lark-cli auth scopes` 列出当前应用支持的所有 scope，包括已授权和未授权的。对比 status 的输出，能看出哪些 scope 你能申请但还没申请。

`lark-cli auth check --scope "calendar:calendar:read"` 检查某个具体 scope 有没有授权。返回 exit code 0 是有，1 是没有。适合在脚本里做前置检查，命令跑之前先确认权限够不够。

定期审授权范围是好习惯。你可能在早期为了跑通某个场景临时开了高危权限，事后忘了关。每隔一两个月跑一次 auth status，把不需要的 scope 关掉，缩小攻击面。最小权限原则在 CLI 场景同样适用。

## 55. 多个 Agent 共享同一个 bot 身份为什么会互相踢下线

这是个真实踩过的坑。两个 Agent 共用同一个飞书应用的 bot 身份，跑着跑着一个突然 401 报错挂了。

根因在 tenant_access_token 的管理机制。飞书的 tenant_access_token 有有效期，多个独立 client 各自维护 token 缓存、各自刷新时，可能出现一个刷新使另一个缓存失效的情况，表现就是先登录的被后登录的"踢下线"。

两个 Agent 各自维护自己的 token 缓存，就会出现互相踢。Agent A 拿了 token T1 缓存起来，Agent B 触发刷新拿了 token T2，T1 可能很快失效，Agent A 下次请求就 401。

解决方案有几种。第一，两个 Agent 共享同一个 token 缓存，比如都从一个文件或 Redis 里读 token，谁刷新了都更新到共享存储。第二，错开刷新时机，让刷新不会同时发生。第三，干脆给两个 Agent 各建一个飞书应用，各自独立的 bot 身份，从根本上避免冲突。

如果你搭的是生产级多 Agent 系统，建议每个 Agent 一个应用。共用的省事是表面的，踩 token 互踢的坑不划算。

## 56. OpenAPI 到飞书 CLI 到 Skills 三层各解决了什么

飞书的能力从底层到上层分三层，每层解决不同的问题。

最底层是 OpenAPI，飞书开放平台的 2500 多个 API。这是所有能力的源头，覆盖消息、文档、日历、任务、审批等全部业务域。它解决的问题是最基础的"飞书的能力是什么"，但直接用要自己处理鉴权、错误、分页、参数拼装，开发成本高。

中间层是 CLI，把 OpenAPI 封装成 200 多条命令行工具。它解决的是"怎么让调用更省事"。OAuth 鉴权内置、token 自动刷新、错误格式统一、分页自动处理、dry-run 兜底安全、结构化输出方便 Agent 解析。开发者不用再重复写样板代码。

最上层是 Skills，一组告诉 Agent 怎么用 CLI 的 markdown 文档。它解决的是"怎么让 Agent 知道有什么命令、什么场景调哪条"。每个 skill 对应一个业务域，告诉 Agent 这个域有哪些 shortcut、参数怎么填、常见错误怎么避。

三层叠加的效果是：Agent 从"面对 2500 个 API 不知所措"，变成"读懂 skill 文档就能直接调命令"。每往上一层，使用门槛降一个数量级。

## 57. 一个 Skill 内部是怎么和飞书 API 交互的

Skill 本身只是一份文档，它不直接执行任何代码。它的工作是告诉 Agent"遇到这类需求应该调哪条 CLI 命令"。

完整链路是这样的。Agent 收到你的自然语言指令，比如"帮我查今天的日程"。Agent 读取 lark-calendar skill 文档，理解到查日程要用 `lark-cli calendar +agenda` 这条命令。Agent 在终端执行这条命令。

CLI 接收到命令后，内部走的是：先查鉴权状态，确认当前身份有日历读取权限；构造 OpenAPI 请求，带上 OAuth token 和必要参数；发请求到飞书服务器；拿到 JSON 响应后按 CLI 的输出契约格式化，成功结果走 stdout、错误走 stderr、exit code 区分成败。

Agent 拿到 CLI 的 stdout 输出，解析 JSON，理解今天的会议列表，用自然语言返回给你。

整条链路：你的话 → Agent 理解 → skill 文档指路 → CLI 执行 → OpenAPI 调用 → 飞书服务器返回 → CLI 格式化 → Agent 解析 → 自然语言回你。Skill 是导航，CLI 是执行体，OpenAPI 是底层通道。
