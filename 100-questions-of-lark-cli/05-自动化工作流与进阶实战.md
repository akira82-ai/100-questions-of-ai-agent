# 第五章 · 自动化工作流与进阶实战

## 81. 怎么让飞书 CLI 的自动化任务定时跑起来

把 CLI 接到定时调度上，是从"偶尔用一下"走向"持续自动跑"的分水岭。

最轻量的方案是用系统自带的定时工具。Linux 和 macOS 用 cron，Windows 用任务计划程序。写一个 shell 脚本封装你要跑的 CLI 命令，然后让 cron 按时触发。每天早上八点推当天日程，每周五下午发周报，这种固定频率的任务用 cron 完全够用。

cron 条件的写法举例：

```bash
# 每天早上 8:00 推送当天日程
0 8 * * * /usr/local/bin/lark-cli calendar +agenda --as user >> /var/log/lark-daily.log 2>&1

# 每周五 17:00 发周报
0 17 * * 5 /home/user/scripts/weekly-report.sh
```

进阶方案是用专门的任务调度器。如果你已经在用 n8n、Airflow、或者 GitHub Actions，把 CLI 命令封装成一个 step 接进去，能拿到更完整的重试、监控、告警能力。

容器化方案适合需要稳定常驻的场景。把 CLI 装进 Docker 镜像，用容器编排工具（Kubernetes CronJob、Docker Compose）管理调度，适合企业级自动化。

几个工程细节要注意。第一，CLI 命令里不要写死个人路径或凭证，用环境变量传入。第二，每个任务配独立日志，出问题时能快速定位是哪一步挂了。第三，对有副作用的命令（发消息、写文档）做幂等处理，避免定时器重复触发时发两遍。第四，token 过期会让任务静默失败，调度脚本里加上 `lark-cli auth status` 的预检，token 不行就告警而不是硬跑。

## 82. 怎么把飞书 CLI 接进 CI/CD 流水线

把 CLI 放进 CI/CD 流水线，让构建结果、部署状态、测试报告自动推到飞书群，是研发团队最高频的自动化场景之一。

接入方式分两层。第一层是 Runner 层，CI/CD 的执行机器上要装好 CLI 并完成 auth login。GitHub Actions、GitLab CI、Jenkins 这些主流工具都支持自定义 Runner，在 Runner 初始化脚本里加一行 `npx @larksuite/cli@latest install` 就能装好。第二层是 Pipeline 配置层，在流水线的某个 stage 调用 CLI 命令。

GitHub Actions 的接入示例：

```yaml
- name: Notify Feishu on deploy success
  run: |
    lark-cli im +messages-send \
      --as bot \
      --chat-id ${{ secrets.FEISHU_DEPLOY_CHAT_ID }} \
      --text "✅ ${{ github.repository }} 部署成功，commit: ${{ github.sha }}"
```

凭证管理用 CI/CD 平台的 secrets 机制，不要把 app_secret 或 token 写进代码仓库。GitHub Actions 用 secrets，GitLab CI 用 variables，Jenkins 用 credentials。

CI/CD 场景下身份选择有讲究。推消息用 bot 身份更稳定，token 是 tenant_access_token，不需要人工重新授权。如果要看个人相关的数据（比如某人的 PR 列表），用 user 身份但要接受定期重新登录的代价，CI 环境下 relogin 比较麻烦，最好固定用 bot。

通知内容设计上，别把整个构建日志一股脑塞进消息。构建结果摘要加一个详情链接（指向 CI 平台的 build 页面），让人想看细节时点链接，飞书群的消息体量保持精简。

失败告警要分层。构建失败推一个红色消息到研发群，部署失败推一个加急消息到负责人私聊，测试覆盖率下降推一个 warning 到质量群。不同严重度走不同渠道，避免告警疲劳。

## 83. 事件订阅怎么让 AI 实时响应飞书消息

定时任务是 CLI 主动去查，事件订阅是飞书主动推给你。想让 AI 在有人 @ 机器人时秒回，有人人加群时自动欢迎，文档被修改时触发通知，全靠事件订阅。

飞书 CLI 提供了专门的 `lark-event` skill 处理事件订阅。它通过 WebSocket 长连接接收飞书推送的实时事件，支持正则路由和 agent 友好的输出格式。

基本用法：

```bash
# 监听 IM 消息事件
lark-cli event consume im.message.receive_v1

# 监听多个事件并用正则过滤
lark-cli event consume --filter 'im.message.receive_v1' --regex '@bot\s+help'
```

事件订阅的工作机制：你在飞书开放平台订阅某些事件（比如 `im.message.receive_v1`），飞书在事件发生时通过 webhook 或 WebSocket 推送给你。CLI 的 event consume 命令维护 WebSocket 连接，收到事件后按你的过滤规则路由到不同的处理逻辑。

和 webhook 方式相比，WebSocket 的优势是不需要公网 IP 和域名。webhook 要飞书能访问你的服务器，对内网部署不友好；WebSocket 是你的客户端主动连飞书，防火墙友好。这也是 CLI 默认走 WebSocket 的原因。

要让 AI 真正"响应"事件，需要把事件 consume 和 AI Agent 串起来。一种模式是事件 consume 把事件写到队列，另一个 Agent 进程从队列读并处理。另一种模式是 consume 收到事件后直接调 Agent 的接口（比如 Claude Code 的 prompt 接口）。

工程上的注意点：事件消费要保证幂等。飞书在不确定你是否收到时会重推，同一个事件可能被消费多次，处理逻辑要能识别重复。处理耗时操作别阻塞 consume 主循环，用异步队列承接。WebSocket 断连要有重连机制，CLI 内置了自动重连，但你自己的消费逻辑挂了不会自动恢复。

## 84. 怎么写一个自定义 Skill 封装重复操作

用了一段时间 CLI 后你会发现某些操作反复出现：每次发周报都是同样的命令组合，每次同步任务都是同样的流程。这时候就该写自定义 Skill 把它们封装起来。

飞书 CLI 提供了 `lark-skill-maker` 这个 skill 专门帮你创建自定义 skill。它本质上是一个框架，让你按规范写一份 SKILL.md，定义这个 skill 的触发条件、参数、执行逻辑，CLI 会把它当成内置 skill 一样加载。

写一个自定义 skill 的基本步骤。

第一步，明确这个 skill 要做什么。比如"每周五下午自动从多维表格拉本周完成的事项，整理成周报 markdown，发送到指定群"。把它拆成几个原子动作：查多维表格、整理数据、生成 markdown、发消息。

第二步，写 SKILL.md。文件结构包括 skill 名字、描述、触发词、参数定义、执行步骤。执行步骤里写清楚每个原子动作调哪个 CLI 命令、传什么参数、怎么处理返回。

第三步，测试。用 dry-run 跑一遍执行步骤，确认每一步的输入输出符合预期。

第四步，部署。把 SKILL.md 放到 skills 目录下，CLI 会自动加载。

实际写 skill 时几个建议。参数设计要为 Agent 友好，用清晰的名字和描述，让 Agent 能从用户自然语言里正确提取参数。错误处理要稳健，某一步失败时给出明确的下一步建议（重试、降级、人工介入）。复杂逻辑拆成多个小 skill 组合，别把一个 skill 写成几百行的大怪物。

社区里有人提过希望有"选择性安装"skill 的能力，只装自己需要的几个域，避免 26 个 skill 全注入占用上下文。这个需求官方还在评估，目前你可以手动管理本地安装的 skill 列表，删掉用不到的。

## 85. 多维表格跨表关联用飞书 CLI 怎么处理

多维表格的强大在于跨表关联：订单表关联客户表，任务表关联成员表。但这种结构用 CLI 操作时会遇到一些特殊问题。

跨表关联的字段类型叫 lookup 字段，它的值不是直接数据，是指向另一张表某条记录的引用。CLI 读这类字段时拿到的是 record_id，不是关联表里的实际内容。你想看到客户名而不是一串 recXXXXXX，需要再调一次 API 查关联表的对应记录。

写自动化脚本时这是高频操作。务实做法是写一个辅助函数：输入 record_id 和字段定义，自动解析 lookup 字段并返回关联表的实际值。这个函数复用率极高，建议放在你的工具库里。

反向操作（写入关联字段）也要注意。写入 lookup 字段时传的值必须是关联表里已存在的 record_id，传错了要么报错要么建一个空关联。写入前先查关联表确认 record_id 存在，是个稳妥的习惯。

跨表查询的性能问题。如果你要批量处理订单表的 500 条记录，每条都关联了客户表，朴素做法是 500 次查询客户表。优化方案是先批量查出本批次涉及的所有客户 record_id，一次性查回客户表的数据，在内存里做 join。这样从 500 次请求降到 2 次。

多维表格还支持 formula 字段做跨表计算，rollup 字段做聚合。这类字段是只读的，CLI 能读不能写，写入操作时要避开它们。

复杂业务场景的实操建议：把多维表格当成结构化数据存储 + 简单关联，复杂的业务逻辑放到外部代码里处理。多维表格的关联能力适合轻量场景，重度关联还是上正经的数据库更可靠。

## 86. 怎么用飞书 CLI 批量管理飞书任务清单

飞书的任务系统比想象的能扛事，配合 CLI 可以做相当深度的批量管理。

基础操作链路。`lark-cli task +get-my-tasks` 列出任务，`lark-cli task +create` 创建任务，`lark-cli task +complete` 完成任务，`lark-cli task +update` 更新任务详情。这些原子命令组合起来能覆盖绝大多数批量场景。

典型场景一：批量创建。项目启动时要给五个成员各建三个任务，一共十五条。用脚本读一个 CSV 或 JSON 配置文件，循环调 `task +create`。每条任务的标题、截止时间、负责人、所属清单都从配置里读。

典型场景二：批量完成。某个 milestone 结束，把这个 milestone 下的所有未完成任务一次性标记完成。先 `task +get-my-tasks --list-id xxx --status incomplete` 拉未完成列表，再循环 `task +complete`。

典型场景三：批量重新分配。某员工离职，把他名下的所有任务转交给接手人。先按负责人筛选拉列表，再循环 update 改 assignee。

典型场景四：逾期任务提醒。每天扫一遍所有未完成任务，发现逾期的推消息给负责人。配合 cron 定时跑，就是自动化的逾期催办系统。

工程上的几个要点。批量操作用 `--page-all` 自动翻页，别手工翻。任务 ID 字段叫 task_guid 或 task_id，不同命令的参数名略有差异，写脚本前先 `--help` 确认。子任务的处理和主任务类似，但参数里要带 parent_id。

任务和飞书其他业务域的联动。任务可以从会议纪要里自动提取（用 minutes 的待办字段），可以从多维表格的某一行同步过来，可以挂到日历上提醒。这些联动需要你自己写胶水代码，CLI 提供原子能力，组合方式看你业务。

## 87. 自定义 Skill 怎么打包分发给团队用

写好一个自定义 skill 后，如果团队其他人也要用，怎么分发是个实际问题。

最简单的方式是 Git 仓库。把 skill 文件提交到一个仓库，团队成员 clone 下来复制到自己的 skills 目录。优点是版本可控，更新走 git pull。缺点是手动操作多，skill 更新了成员不一定知道。

进阶方式是做成 npm 包。skill 文件本质上是 markdown 加可能的脚本，打包成 npm 包后，团队成员用 `npm install` 装，配合 `npx skills add` 注册到 CLI。这种方式适合跨团队、跨公司分发，也方便做版本管理和依赖声明。

企业内部可以用内部包管理。把 skill 包推到企业私有 npm registry，或者打包成 tar 文件放内网共享盘。团队成员从内部源安装，不依赖公网。

分发时要处理的几个细节。第一，skill 里的硬编码路径和凭证要参数化，不同人的环境路径不同，硬编码会让 skill 跑不起来。第二，skill 的依赖要明确声明（依赖哪个 CLI 版本、依赖哪些 scope），让安装者知道前置条件。第三，写一份简单的 README 解释这个 skill 干什么、怎么装、怎么用。

版本管理上，给 skill 打 semantic version。破坏性变更升大版本，新增功能升中版本，bug 修复升小版本。团队成员能从版本号判断升级会不会出问题。

skill 分发的理想形态是有一个官方 marketplace，像 VS Code 扩展商店那样一键安装。目前飞书 CLI 还没到这一步，但社区的 issue 里已经有人讨论这个方向。现阶段靠 Git 和 npm 包够用，只是流程没那么自动化。

## 88. 一个飞书 CLI 授权能不能给团队多人共用

这是团队推广时绕不开的问题。简单回答：技术上可以，实践上不推荐。

技术上的可行性。一个飞书应用有一套 app_id 和 app_secret，多人用同一套凭证 auth login，每个人拿到的是自己的 user token（用户身份）或共享的 tenant token（bot 身份）。bot 身份天然是共享的，user 身份每人独立。

不推荐共用的原因。

第一是权限边界模糊。共用应用意味着共用权限范围，A 需要文档写权限，B 不需要，但 B 拿到的 token 也带文档写权限。最小权限原则被破坏。

第二是审计困难。所有操作挂在同一个应用名下，出了问题查不到是哪个人执行的。飞书后台的审计日志只能看到应用维度，看不到用户维度。

第三是稳定性风险。共用应用意味着一个人重置 app_secret，所有人的凭证全部失效。一个人改了权限配置，影响所有人。

第四是 token 互踢问题。多个 Agent 用同一个 bot 身份时，后登录的会踢掉先登录的，详见第三章第 55 题。

推荐的多用户方案。每个用户自己 config init 创建独立应用，用自己的凭证 auth login。应用之间完全隔离，权限、审计、稳定性互不影响。如果企业管理严格，由 IT 部门统一创建应用并分配给员工，每人拿到的还是独立凭证。

bot 身份的共享要看场景。CI/CD 通知这类无个人归属的场景，用共享 bot 没问题。涉及个人数据操作的场景，bot 身份不如 user 身份合规。

企业管理员层面，可以通过飞书后台的应用管理控制谁能创建应用、谁能授权 CLI，从源头管控共用风险。

## 89. 企业管理员怎么管控谁能用飞书 CLI 创建应用

企业环境推广 CLI，管理员最担心的是失控。员工随便建应用随便授权，权限满天飞，审计困难。好在飞书开放平台提供了几个管控抓手。

第一道抓手是应用创建审批。飞书企业版可以在后台设置"创建应用需要管理员审批"。员工想 config init 新建应用时，必须经过管理员批准才能创建。这是从源头控制应用数量的关键。

第二道抓手是应用权限审批。每个应用要开通哪些 scope，管理员可以设置高危权限需要审批。员工把 CLI 的应用开了文档删除权限，管理员收到审批请求，觉得不合适可以拒绝。

第三道抓手是应用可见性控制。管理员可以限制应用只在企业内部可见，禁止发布到应用市场。CLI 这类内部工具应用，默认应该设为内部可见。

第四道抓手是审计日志。飞书后台提供完整的操作日志，哪个应用调了哪个接口、什么时候调的、返回什么结果，都能查。出了问题能追溯到具体应用，再结合应用的使用者名单定位到人。

第五道抓手是应用停用机制。员工离职或转岗时，管理员停用其创建的 CLI 应用，立即切断访问权限。这比等 token 自然过期要快。

实施建议。先在小范围试点，让 IT 或效率团队先用起来，跑通一套管理流程再开放给全员。配套文档要跟上，写清楚"员工怎么申请 CLI 应用""管理员怎么审批""出问题找谁"。

对企业来说，CLI 的管控重点在于让员工用得可控，别一刀切禁了。禁了反而会让员工绕路用更不可控的工具。开正门堵偏门，是治理的正确姿态。

## 90. 事件订阅和轮询怎么选才不浪费配额

获取飞书数据变化有两种方式：事件订阅（飞书推给你）和轮询（你主动去查）。两种方式各有成本，选错了要么错过实时性，要么浪费 API 配额。

事件订阅适合的场景：消息到达、群成员变动、审批状态变化、文档评论新增。这些事件发生时机不可预测，用轮询要高频查才能及时响应，配额消耗大。事件订阅由飞书主动推，实时性最好，你只需要在事件到达时处理，平时零消耗。

轮询适合的场景：定期统计（每天算一次数据汇总）、批量同步（每小时拉一次增量数据）、状态确认（等某个异步任务完成）。这些场景对实时性要求不高，定时查一次就够，用事件订阅反而过度设计。

混合模式是务实选择。核心业务用事件订阅保证实时，非核心数据用低频轮询（比如每 15 分钟一次）兼顾成本。比如客服消息走事件订阅秒级响应，数据报表每 15 分钟轮询更新一次。

配额消耗的考量。飞书开放平台对每个应用有 API 调用频率限制，轮询频率太高会触发限流。事件订阅不消耗 API 配额（事件推送是飞书侧主动行为），但 WebSocket 连接数有上限。大批量轮询前先算清楚：你要查几个对象、每个对象多大、多久查一次，把总请求量估算出来对比配额。

减少轮询消耗的技巧。用增量查询而不是全量查询，每次只拉上次同步后的变化数据。用 webhook 替代轮询，如果飞书支持对应事件的话。用缓存减少重复查询，查过的数据短时间内不查第二次。

选型判断标准：实时性要求高 + 事件不可预测 = 事件订阅；实时性要求低 + 数据稳定 = 轮询；混合需求 = 核心事件订阅 + 边缘轮询。

## 91. 自动化脚本上线后怎么监控它有没有正常跑

脚本开发完只是开始，上线后的稳定运行才是真正的挑战。没人监控的自动化脚本，挂了你都不知道，等用户反馈"咦怎么没收到通知"才发现已经停了一周。

监控的第一层是进程存活。定时任务（cron、Kubernetes CronJob）执行完后记录执行状态，连续几次没执行说明调度出了问题。常驻服务（WebSocket 消费、轮询服务）用进程监控工具（systemd、supervisor、PM2）守护，挂了自动重启并告警。

监控的第二层是执行结果。每次脚本执行完记录关键指标：执行时长、成功/失败、处理了多少条数据、有没有异常。这些指标推到监控系统（Prometheus、CloudWatch）或者简单地写到日志文件，定期看一眼。

监控的第三层是业务效果。脚本"跑成功"不代表"业务正常"。比如你的任务是每天推日程，脚本返回 success 但实际没发出去（token 过期、网络抖动），业务上是失败的。这类要监控终端效果：消息有没有真发出去（看飞书返回的 message_id）、文档有没有真更新（fetch 确认）。

告警机制要分级。脚本失败一次可能是偶发，连续失败三次才告警。重要业务（部署通知、客户消息）失败立刻告警，次要业务（周报、统计）容忍延迟。告警渠道别只用飞书消息，万一飞书本身就是故障源，告警也发不出来，要有短信或邮件兜底。

日志管理。每个脚本配独立日志文件，按日期切割保留 30 天。日志里记录关键步骤的输入输出，出问题时能回溯。别只记"执行成功"，要记"执行了什么命令、传了什么参数、返回了什么"。

健康检查端点。如果是常驻服务，暴露一个 HTTP 健康检查接口，让外部监控（uptime robot、Prometheus blackbox）定期探活。服务正常返回 200，异常返回非 200。

CLI 自带的 `lark-cli doctor` 可以集成到监控脚本里，定期检查凭证状态、网络连通性，发现问题提前预警，别等到任务真的失败。

## 92. 多个飞书应用之间怎么做身份隔离

一个人同时管多个飞书应用是常见场景：公司主体应用、个人测试应用、客户委托管理应用。这些应用的身份必须严格隔离，混了就出大事。

飞书 CLI 的 profile 机制就是为多应用隔离设计的。每个 profile 对应一个独立的配置目录，存自己的 app_id、app_secret、token、scope。切换 profile 等于切换应用身份，互不干扰。

profile 的基本操作：

```bash
# 创建新 profile
lark-cli config init --profile client-a

# 切换到指定 profile
lark-cli config use client-a

# 查看当前 profile
lark-cli config show

# 列出所有 profile
lark-cli config list
```

每个 profile 独立 auth login，token 存在各自的配置目录或 keychain 条目里。A profile 的操作不会影响 B profile 的凭证。

隔离的工程实践。第一，每个项目用独立 profile，项目代码里通过环境变量指定 profile 名，避免误用默认 profile。第二，profile 命名要清晰，用项目名或客户名命名，别用 test1、test2 这种模糊名字。第三，定期清理不用的 profile，减少误操作风险。

CI/CD 环境的多应用隔离用环境变量。每个流水线配自己的 app_id、app_secret，脚本里通过环境变量读取，不依赖 profile 文件。这样不同流水线天然隔离。

容器化部署的隔离更彻底。每个应用跑独立的容器，容器里有独立的 CLI 配置和凭证，物理隔离。一个容器出问题不会污染另一个。

身份混用的典型事故：用 A 应用的 token 调 B 应用的数据，报 `open_id cross app` 或 `invalid param`。这类错误信息不明显，容易让人以为是参数拼错。遇到这种诡异的鉴权错误，第一步先确认 profile 对不对、token 是哪个应用的。

社区有人反馈过 profile 不匹配导致的低信号错误问题，CLI 的诊断信息正在改进，新版本会在 doctor 和 auth status 里更明确地区分 bot 和 user 身份状态。

## 93. 怎么让飞书 CLI 在云服务器上常驻服务

本地跑 CLI 适合开发和测试，生产环境的自动化需要在云服务器上常驻运行。

常驻服务的几种形态。

第一种是定时任务型。CLI 本身非常驻，靠 cron 或 systemd timer 定时唤起，执行完退出。适合固定频率的任务（每天推日程、每周发周报）。这种最简单，cron 配好就完事。

第二种是事件消费型。CLI 的 event consume 命令维护 WebSocket 长连接，需要常驻运行。用 systemd 或 supervisor 把它作为服务管理，挂了自动重启。适合实时响应场景（消息自动回复、事件触发通知）。

第三种是 API 服务型。把 CLI 封装一层 HTTP 接口，对外提供 RESTful API，其他系统调你的接口你转调 CLI。这种适合企业内部的中台架构，多个业务系统共用一套飞书集成。

云服务器上的关键配置。

凭证管理。不要把 app_secret 写进代码或配置文件提交到 git。用环境变量或云平台的 secrets 管理（AWS Secrets Manager、Kubernetes Secrets）。

网络配置。云服务器要能访问飞书的 API 域名。国内云服务器访问 feishu.cn 通常没问题，海外服务器访问 larksuite.com 要确认网络通畅。企业内网的服务器要让 IT 放行相关域名。

资源规划。CLI 本身是轻量级工具，常驻一个 event consume 进程占几十 MB 内存够用。但如果你的服务还要跑 Agent（比如 Claude Code），要给 Agent 留足资源。

日志和监控。常驻服务必须有日志，systemd 自带 journald，或者自己写日志文件。配合云平台的监控（CloudWatch、阿里云监控）盯进程状态。

高可用。单点服务总有挂的风险，重要业务上双机热备或 Kubernetes 多副本。但 CLI 的 WebSocket 事件订阅不支持多副本同时消费同一个事件，多副本要做去重或分区。

## 94. 飞书消息历史上限只有 14 天怎么长期存档

飞书群消息默认只能往前翻 14 天（企业版可能更长，但有上限）。对合规审计、知识沉淀、争议追溯来说，这个期限不够，需要自己做长期存档。

存档的基本思路是定期拉取消息存到自己的存储。用 `lark-cli im +chat-messages-list` 或 `im +messages-search` 按时间范围批量拉历史消息，存到数据库或对象存储。

存档的几种实现模式。

模式一：定时增量拉取。每小时或每天跑一次，拉取过去一段时间的增量消息。配合 cron 或调度器，持续追加到存档库。优点是实现简单，缺点是 near-realtime 而非 realtime，且如果某次拉取失败可能漏数据。

模式二：事件订阅实时存档。订阅 `im.message.receive_v1` 事件，每条消息到达时立刻存一份。优点是实时性强、不漏数据，缺点是需要常驻服务、依赖事件订阅稳定性。

模式三：混合。事件订阅做实时存档，定时任务做对账（每小时比对一次存档库和飞书实际消息数，发现缺漏补拉）。

存档内容的完整性。消息不只是文本，还有图片、文件、回复、表情反应、富文本 post。完整存档要把这些附件也下载下来。图片和文件用 `im +messages-image-download` 或 `drive +download` 拉本地，附件和消息本体用 message_id 关联。

存档的可检索性。原始 JSON 存下来后，要建索引才能高效查询。Elasticsearch 适合全文搜索，关系数据库适合结构化查询。按群、按人、按时间、按关键词多维检索，是存档系统的基础能力。

合规场景的额外要求。金融、法律行业对消息存档有监管要求，存档系统要做到不可篡改（append-only 或区块链存证）、可审计（每次访问留痕）、有保留期限策略（保留 7 年还是永久）。这些超出了 CLI 的能力范围，需要专门的合规存档系统，CLI 只负责数据采集这一环。

存档这件事早做早安心。等真的需要某条三个月前的消息发现没存，再补救就来不及了。新群一建立就把存档接入跑起来，是稳妥的做法。
