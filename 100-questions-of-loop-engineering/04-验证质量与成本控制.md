# 四、验证、质量与成本控制

## 53. 为什么 loop engineering 的真正瓶颈已经从生成转向验证

生成这件事，这两年已经没那么稀缺了。模型写东西、补东西、扩东西的能力，大家都见过了。现在真正卡住很多团队的，是另一头：你怎么知道它这次真的做对了。

因为 loop 一旦能连续执行，系统的产出速度会明显快过人类复核速度。你会越来越少遇到“写不出来”，越来越多遇到“写出来了，但我不敢信”。这时候瓶颈自然就从生成挪到了验证。你不把验证补强，前面那一大堆高效产出只会变成更快的风险放大器。

所以今天很多团队最缺的，不是更会生成的 agent，真正缺的是更会拦错的体系。模型负责把东西做出来，验证负责决定这些东西值不值得留下来。

## 54. verifier 到底该检查结果、过程还是中间状态

答案通常是都要看，只是轻重不同。

只看结果，容易漏掉“碰巧答对”的假稳。只看过程，又容易把系统管得太死，最后每一步都得照剧本演。中间状态也很关键，因为很多问题在最终输出里已经看不出来了，偏差是中途长出来的。

比较实用的做法，是把这三层拆开。结果层看交付是否达标，过程层看有没有违规动作，中间状态层看有没有明显跑偏。这样系统既不会完全靠运气过关，也不会因为管得过细而失去灵活性。验证要做的，从来不只是选一个层面死盯，更重要的是知道哪一层最容易出事。

## 55. 什么样的 verifier 才能真正拦住假完成

能拦住假完成的 verifier，通常有三个特征：标准具体、失败明确、反馈可执行。

标准具体，意思是它不会只说“看起来不错”，而是能指出哪条条件没过。失败明确，意思是它真敢判失败，不会一边发现问题一边还是给通过。反馈可执行，意思是它说完问题之后，下一轮能真的照着改，不会只留下一句泛泛的“建议优化”。

很多 verifier 看着存在，其实并没有这三样，所以最后只是做了个检查姿态。真有用的 verifier，不靠气氛感拦错，靠的是一条条能落地的门槛。

## 56. 为什么 done 写不清，整个 loop 就没有可控性

因为只要 done 是糊的，继续、停止、返工这些决策就都会跟着糊。

很多系统写目标时很热闹，写 done 时却很偷懒。结果就是，agent 不知道什么时候该停，verifier 不知道按什么判，人类也不知道这一轮到底算不算完成。最后整个 loop 会滑向一种很熟悉的状态：每个人都感觉差不多了，但没人说得清到底差在哪。

done 写清楚的价值，不只是方便验收，它其实是在给整个 loop 定边界。边界一旦清了，成本、质量、节奏这些东西才开始真正可控。

## 57. loop 的质量标准到底该先写在 prompt 里还是系统里

能进系统的，尽量先进系统。留在 prompt 里的，更适合放那些需要弹性判断的部分。

因为 prompt 里的标准很容易漂。你换个任务、换个上下文、换几轮对话，模型对那句话的理解就可能有偏差。系统层的标准则更硬一点，比如结构校验、测试门槛、审批条件、权限规则，这些写进去以后更稳定，也更容易复用。

所以比较稳的分工是：系统负责守底线，prompt 负责补语气和启发。底线要硬，发挥可以柔。反过来做，loop 往往会越来越像凭感觉在跑。

## 58. 为什么 hard checks 才是 loop 收敛的地板

因为只靠软判断，系统迟早会学会给自己找理由。

hard checks 的价值在于，它不给解释空间。测试没过就是没过，格式不对就是不对，关键路径没走通就是没走通。你可以在上面叠更复杂的主观评估，但底下那层硬门槛如果没有，loop 很容易一边讲道理一边把烂结果放过去。

很多人一开始嫌 hard checks 太笨，觉得限制 creativity。真到生产环境里，才会发现这层“笨”恰恰是收敛的地板。没有地板，你连漂到哪里都不知道。

## 59. 什么情况下要用 rubric，什么情况下要用确定性检查

凡是能清楚算出来、测出来、判出来的，优先用确定性检查。凡是带主观度、带综合判断、带风格权衡的，再上 rubric。

比如接口状态、输出格式、关键字段、测试结果，这些都适合确定性检查。可你让系统判断“这份回答有没有说服力”“这个页面有没有质感”“这个 triage 建议是否够稳”，这时候 rubric 就更有用，因为它能把模糊判断拆成几条可比的维度。

说白了，确定性检查是铁门，rubric 是评分表。两者都重要，只是别把该上铁门的地方拿去打分，也别把明显主观的问题硬凹成二元判断。

## 60. 为什么模型给自己打分在 loop 里几乎总会偏乐观

因为它刚刚参与了生成，对自己那套思路天然有路径依赖。

它知道自己为什么这么写，为什么这么选，于是特别容易把“我能解释”误当成“我做对了”。这跟人写完东西自己审稿很像，脑子里已经有了原始意图，看问题时就会自动帮自己补全。

所以自评分并非完全没用，它可以拿来做轻量预筛。但你要把它当成主验证，那基本就是在邀请乐观偏差上桌。真想要严一点，还是得拉开视角、拉开角色。

## 61. loop 要跑得稳，哪些指标必须持续被记录

至少得持续记这几类：质量、成本、时延、轮次、失败类型、人工介入点。

质量告诉你它有没有真变好，成本告诉你值不值，时延告诉你系统是不是越来越重，轮次告诉你它有没有收敛，失败类型告诉你问题在变少还是在换样子，人工介入点则会暴露出真正的瓶颈到底还在人还是还在系统。

很多团队只记 token 和成功率，结果看起来数字挺好，真实体验却一塌糊涂。因为 loop 稳不稳，从来都不是单指标问题。你得看它是不是同时在质量、成本和可控性上一起站住。

## 62. 为什么 latency 变长，往往说明 verifier 开始变复杂了

因为生成通常是往前冲，验证通常要回来反复看。

你写一段东西可能很快，但你要核对它有没有满足条件、工具返回对不对、页面路径通不通、中间状态有没有漂，这些动作天然就更慢。验证一复杂，latency 往往先涨出来。

所以 latency 变长不一定是坏事，它有时是在提醒你系统正从“先做出来”往“认真确认一下”移动。问题不在延迟本身，而在你有没有意识到这段延迟买来了什么。如果更慢却没更稳，那才是真的亏。

## 63. cost、quality、novelty 三者在 loop 里到底怎么取舍

这三个东西很少能同时拉满。你想要更便宜，通常就得少试几轮、少开分支、少做验证。你想要更新奇，往往就得允许更多探索和更多失败。你想要质量稳，成本和速度大概率都要让一点。

所以取舍的关键，不在于找到一条“通用最优线”，而在于先承认你现在最想保哪一项。生产交付通常先保质量，创意探索可以多给 novelty 空间，批量运营则会更看 cost。

最怕的情况，是嘴上什么都想要，系统里却没写清优先级。那最后通常三头都顾不上。

## 64. 什么情况下你该为更强 verifier 付出更高成本

当错误代价明显高于验证代价的时候，就该花。

比如改真实仓库、批量发用户消息、动工单、动知识库、动生产数据，这些地方一旦放过假完成，后面返工和损失都很大。你在这里省 verifier 的钱，最后多半会从别的地方加倍吐出来。

相反，如果任务本身很轻、可逆、失败也容易修，那就没必要上过重验证。花不花，不取决于你喜不喜欢严谨，取决于错误一旦漏过去到底会有多贵。

## 65. loop 预算到底该按轮次管，还是按结果管

只按轮次管，容易误伤那些慢但必要的任务。只按结果管，又容易放任系统无限膨胀。更稳的方式，是两层都管。

轮次预算适合做第一层护栏，防止系统无限多试。结果预算适合做第二层判断，去看这些轮次换回来的质量提升到底值不值。你把这两层绑在一起，系统才不会一边很节省、一边毫无产出，或者一边产出挺多、一边贵得离谱。

说到底，loop 预算不该只问“跑了几轮”，也不该只问“做完了没”，它得回答“为这个结果花这些资源，到底值不值”。

## 66. 为什么很多人知道 token 贵，却还是管不住 loop 成本

因为大家通常只知道“单次调用很贵”，却没把“系统会怎么持续放大这个贵”算进去。

loop 成本最容易失控的地方，不在一轮，而在累积。多试一次、多开一个分支、多跑一个 verifier、多做一次复盘，看上去都合理。可这些小合理一叠起来，账单就会变得很不合理。

所以控制 loop 成本，核心不在天天盯 token 单价，更在于看哪些结构正在自动制造额外轮次、额外验证、额外上下文。成本失控通常是系统习惯问题，不是算术问题。

## 67. 怎么判断一个 loop 是在收敛，还是在优雅地浪费钱

看三件事：错误是不是在减少，新增轮次是不是还在带来明显收益，系统是不是离 done 更近了。

如果每多跑几轮，问题类型还是老样子，成本继续涨，done 仍然模糊，那大概率就是在优雅地浪费钱。它会给你一种很专业的错觉：日志完整、反馈细致、动作连续。可你把这些表象扒开，里面没有真正的收敛。

收敛的 loop 会越来越短、越来越准、越来越少废动作。花钱不是问题，花钱却没让边界变清楚，那就有问题。

## 68. 遇到高成本 loop，第一步该砍模型还是先砍无效循环

大多数时候，先砍无效循环。

因为很多高成本问题，根本不在模型太贵，而在系统做了太多低价值重复动作。你模型换便宜了，结果 loop 还是继续空转、继续重复验证、继续无意义重试，那只是用便宜模型更便宜地浪费钱。

真正该先看的，是哪些轮次可以删，哪些检查重复了，哪些上下文根本不该塞进去。结构不改，换模型往往只是缓一口气；结构改了，成本曲线才会真的下来。

## 69. 为什么惊喜很多的 loop，往往也是最难控成本的 loop

因为惊喜通常来自探索，而探索天然就意味着更多分叉、更多试错、更多无功而返。

你要系统一直走最稳的路，当然能控成本，可它也更容易变得保守。你要它偶尔跳出套路，去试新的组合、新的工具路径、新的表达方式，成本就很难像流水线那样被卡得死死的。

这不说明惊喜型 loop 没价值，只说明你得先想清楚：你是在买稳定产出，还是在买探索空间。探索空间本来就贵，关键是别把它误装成廉价自动化。

## 70. 为什么 loop 的上限常常卡在评估体系上

因为模型再强，也只能在你定义的“好结果”空间里优化。

如果评估体系很粗，系统就会学着满足那些粗标准；如果评估体系有盲区，系统就会在那些盲区里长出问题。你最后看到的上限，很多时候未必卡在模型能力，更多是卡在你有没有把“什么叫做好”写清楚、测清楚、卡清楚。

所以 loop 做到后面，最值钱的能力往往不在继续换模型，而在继续升级评估。评估体系一抬，系统上限才会跟着抬。
