# 五、工程落地与真实工作流

## 71. 什么任务最适合先从单次 prompting 升级成 loop

最适合升级的，通常是那些重复出现、步骤差不多、结果还得回头验的任务。

更实用一点的判断方法是看四级台阶。第一层，先把单次 prompting 做顺；第二层，做成 turn-based loop，让它自己读代码、动手、跑检查；第三层，如果 done 很清楚，再上 goal-based loop；第四层，只有当任务高频又稳定，才考虑 time-based 或 proactive。顺序别跳。

所以最该先升级的，是那些你已经手工跑了很多次，连提醒它下一步做什么都快背下来了的任务。那种活最容易见效，也最容易把 loop 跑稳。

## 72. 什么任务看起来适合 loop，其实根本不值得自动化

高频不代表值得做 loop，真正不值得的，往往是那些判断代价极高、环境变化太快、或者错误后果非常重的任务。

有些事表面看也重复，比如复杂谈判、敏感审批、模糊战略判断，可它们每次真正难的地方都不一样。你硬做 loop，最后很可能只是自动化了表面动作，没自动化到真正值钱的那层判断。

所以 worth it 这个问题，看的不只是任务出现得多频繁，更要看它里面有多少结构能稳定复用。结构不稳，loop 上得越快，后面越难收。

## 73. 为什么 bug 修复、文档维护和 triage 特别适合做 loop

因为这三类任务都很像：信息要先找齐，动作要一步步做，结果还得回头验。

修 bug 时，你得先复现、再定位、再改、再跑测试。文档维护时，你得先找变更、再对齐事实、再更新、再核对。triage 时，你得先聚合信息、再分类、再判断优先级、再输出处理建议。这些流程都不是一句 prompt 能干净做完的，但它们又足够结构化，很适合闭环推进。

所以这三类任务一旦做成 loop，收益往往特别明显。因为你省下的，不只是一句提示词，更多是整条重复判断链。

## 74. loop 接入真实业务前，第一步该先锁哪几个风险点

先锁权限、数据边界、误触发、假成功、人类兜底这五个点，而且最好写进真实文件里，不要只放在脑子里。

很多团队一开始只顾着让 loop 跑起来，后面才发现真正危险的是它跑得太顺。权限应该写进 `settings.json` 这类配置里，项目边界和禁区要写进 `CLAUDE.md` 这类常驻说明里，验证步骤要落到 hooks、skills、评估脚本这种能反复执行的地方。这样风险控制才不是一句口头提醒。

接业务前先问一句：它如果跑偏，会怎么坏，坏到哪一层，谁来接。这个问题答不清，先别急着自动化。

## 75. 为什么 support loop、SEO loop、product loop 最容易做出复利

因为这三类任务都长期存在，而且每跑一轮都能给下一轮留下资产。

support loop 会留下问题分类、处理模板、升级规则。SEO loop 会留下关键词判断、页面更新规则、内容结构经验。product loop 会留下用户反馈模式、优先级判断框架、复盘结构。它们都不属于一次性任务，反而会越做越能沉淀共用结构。

这就是复利最迷人的地方。你今天不是只完成了一轮动作，还顺手在为后面几十轮铺路。能复用的结构越多，loop 的边际成本就越低。

## 76. 一个 loop 真正接入团队流程，最先撞上的墙是什么

通常不是模型能力，最先撞上的往往是组织摩擦。

谁来信这个结果，谁来接收失败，谁来定义 done，谁来背权限责任，谁来维护规则，这些问题一旦没人拍板，loop 就只能停在 demo 层。个人用的时候靠一个人脑子里的默契还能转，进团队后，这些默契全都要显式化。

所以真正把 loop 接进团队流程，第一堵墙往往是协作结构，不是技术细节。系统想跑得久，先得有人愿意一起按同一套规矩做事。

## 77. 为什么个人能跑通的 loop，一进团队就突然失灵

因为个人 loop 里很多“隐形上下文”，在团队里都会瞬间蒸发。

你自己跑的时候，很多判断根本没写出来。你知道哪个目录危险，知道哪些文件别碰，知道这轮失败了该怎么补。可一到团队，这些东西如果不外显，别人接手就会像进入陌生现场，系统也会因为缺少共同语境而开始失真。

所以个人能跑通，只说明你本人能带着它跑。团队能跑通，才说明这套结构脱离你也能站住。

## 78. loop 想长期跑下去，哪些知识必须沉淀成 skill

凡是会重复出现、又不值得每次重新讲一遍的知识，都应该沉淀成 skill。

最该沉淀的通常有六类：任务分解套路、目录地图、可用命令、验证清单、常见坑、人工升级条件。你去看很多能长期跑的项目，最后都会出现一套很像 `CLAUDE.md + skills/ + hooks/ + MEMORY.md` 的结构。它们做的都是同一件事，把零散经验变成下次还能直接调用的工作资产。

skill 真正值钱的地方，在于让 loop 少走弯路。你把经验搬进系统里，下一轮就不用再靠人临场补洞了。

## 79. 为什么 loop 往往会逼你补齐工程

因为单次 prompting 能糊过去的那些工程烂账，loop 一上来就会全部暴露。

目录不清、状态没存、日志没有、验证太弱、权限混乱、任务定义含糊，这些问题平时可能靠人工经验勉强兜住。可一旦你想让系统持续跑，它们全都会变成显性故障。loop 不会替你遮住工程问题，它会把问题放大到你必须认真处理。

所以很多人刚开始接触 loop，会误以为这是偷懒工具。真做下去才发现，它更像一次工程体检。哪里虚，哪里马上见光。

## 80. 让 loop 写 PR、开 ticket、改文档时，哪些动作必须留人工闸门

凡是会直接改外部事实、影响团队协作节奏、或者可能制造不可逆后果的动作，都应该留人工闸门。

比如发正式 PR、合并改动、关闭 issue、改共享文档、动外部通知、写对外用户信息，这些动作一旦做错，影响已经扩散出系统内部了。你当然可以让 loop 先准备内容、先草拟建议、先跑预检查，但最后那一下是否生效，最好还得有人拍板。

人工闸门留在那里，不是因为人更慢，核心原因是它是责任切换点。越接近真实世界的动作，越值得留这道门。

## 81. 什么情况下 loop 应该只提建议，不能直接执行

高风险、低可逆、责任归属敏感的时候，就该只提建议。

比如安全相关改动、生产数据动作、用户沟通、定价策略、权限变更、法律或合规内容，这些地方就算模型判断八成靠谱，也不值得让它直接拍板。因为漏掉的那两成，很可能就是最贵的两成。

loop 很适合当高质量前置分析器，把信息整好、方案列好、风险点说清。可有些场景，它最好的位置就是到此为止，别再往执行迈那一步。

## 82. 为什么让它后台一直跑听起来方便，实际上最容易出事

因为你一旦看不见它，很多缓慢变形的问题就会有更长时间发酵。

后台 loop 最容易出事的地方，常常出在第十轮、第三十轮。输入变了，依赖挂了，权限漂了，状态文件脏了，定时触发叠在一起了，这些问题单看都不吓人，叠起来就会很烦。所以别一上来就追求 proactive。更稳的做法是先把 turn-based 和 goal-based 跑扎实，再把监控、暂停、报警、人工接管补齐，最后再放到后台常驻。

说白了，后台运行不是“更省心”，只是把问题从眼前挪到了系统里。你没有监控能力，最好别先上这个级别。

## 83. loop 和 GitHub Actions、cron、webhook 结合后会新增哪些坑

最常见的是重复触发、并发打架、状态脏掉、外部依赖抖动，以及失败后自动重试把问题放大。

你单手工跑时，这些问题很多不会显出来。接进调度和事件系统之后，时序才开始真正复杂。一个事件可能被重复收到，一个 workflow 可能在旧状态上继续跑，一个 webhook 可能在你没准备好的时候触发。只要去重、锁、幂等、暂停机制没有设计好，loop 会在自动化层面自己给自己制造事故。

所以 GitHub Actions、cron、webhook 不是单纯的“触发器”，它们会顺手把分布式系统的烦恼也一起带进来。

## 84. 想让多个 loop 彼此协作，最先要统一什么结构

最先要统一的，通常是状态、产物格式、任务边界和交接协议。

多 loop 协作最怕的，不是谁不够聪明，最怕的是谁都听不懂谁。一个 loop 用文件记进度，另一个 loop 用聊天摘要记进度；一个把 done 写成“差不多好了”，另一个要求测试编号和截图，这种情况下，多开几个 agent 只会把混乱放大。先把状态怎么记、产物怎么写、风险怎么标、下一步怎么交，统统定清楚。

所以多 loop 协作更像搭流水线，不像多请几个人来帮忙。结构先统一，信息才流得动。

## 85. loop 跑得越多，为什么共享 artifacts 越容易变成新瓶颈

因为共享产物一开始是资产，规模一大就会变成交通枢纽。

大家都往里写，大家都从里读，版本一多、质量一参差、命名一混乱，artifact 本身就会开始制造摩擦。该信哪份，最新的是哪份，谁有权覆盖，哪些已经过期，这些问题会慢慢取代“有没有产物”本身，变成新瓶颈。

所以共享 artifacts 能带来复用，也会带来治理成本。跑得越多，这个问题越早冒出来。

## 86. 什么样的团队文化，才真的吃得下 loop engineering

得是那种愿意把判断写出来、把流程外显、把失败当系统问题来修的团队。

如果团队文化本来就靠口口相传、靠英雄人物临场救火、靠每个人自己悟，那 loop 很难长起来。因为它最需要的是显式规则、可复用结构、可回溯痕迹。说白了，loop 喜欢制度化环境，不太喜欢“大家心里有数”的环境。

所以一个团队能不能吃下 loop，不只看技术栈，也看它愿不愿意把经验沉淀成公共资产。
