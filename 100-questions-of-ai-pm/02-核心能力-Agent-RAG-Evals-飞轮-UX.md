# 二、核心能力：Agent/RAG/Evals/飞轮/UX 五件套

## 14. 还在用确定性流程图设计 AI 功能？模型只有 80% 把握时你的兜底方案是什么？

传统产品设计的底层假设是确定性，用户点一下按钮，系统必须返回一个确定的结果。AI 产品的底层假设正好反过来，模型输出是概率性的，同一个输入在不同时刻可能给出不同结果，还有一定概率彻底翻车。很多 PM 栽在第一步，原因就是还在用传统流程图的思路设计 AI 功能，每一步都画得很漂亮，但默认每一步都成功，没有任何兜底。这种设计在线下 demo 里很惊艳，一上线，用户稍微问点边缘问题，整个流程就开始雪崩。

兜底要想成三层。最底下是置信度门槛，模型对自己输出的把握程度是可以量化成一个分数的，分数低于某条线就不要硬答，直接走兜底逻辑。中间一层是人工接管路径，模型答不出来或者把握不够的时候，要平滑地把它交给人工，不能让用户卡在报错页面上。最上面一层是用户反馈闭环，答错了要能收集到用户的负反馈，把这些 case 沉淀下来反哺评估集。这三层没有一层是可选项，任何一个缺失，上线后都会以投诉、流失、舆情的方式还回来。

用确定性思路设计 AI 功能，等于在沙地上盖楼，盖得越快塌得越狠。

## 15. 怎么设计 Agent 产品的信任校准，让用户知道什么时候该信、什么时候该介入？

Agent 产品最容易栽在信任这件事上，要么用户不信，要么用户太信，两种极端都会出事。过度信任的典型场面是用户把所有任务都丢给 Agent 做，包括那些高风险的操作，发邮件、转账、删数据，错一次就够喝一壶。信任不足走的是另一头，用户每个动作都怀疑，Agent 的效率优势被消磨殆尽，最后觉得还不如自己手动做。校准信任的核心，是让用户的信任度和 Agent 的实际能力对上，这件事不会自己发生，得靠产品主动设计。

几个设计原则是绕不开的。能力要透明化，用户一开始就得知道这个 Agent 能做什么、不能做什么，别把它卖成全能助手，上线后用户发现做不到反弹会更狠。置信度要可视化，每次输出都带一个信心条，80% 信心和 50% 信心在 UI 上要有明显区别，低信心的回答视觉上就得让用户警觉。错误期望要主动管理，高风险场景必须明确告诉用户这是 AI 输出、请核查，不能让用户误以为是百分百可靠的。

这几个原则落到 UI 上，就是 onboarding 要诚实、输出要带信号、危险操作要二次确认。信任这件事不是越多越好，是越准越好。

## 16. 老板要上多 Agent 协作，你怎么用一份延迟与成本数据劝退他？

多 Agent 协作从 2025 年开始变得特别热，老板听了几场发布会回来就让团队搞多 Agent 系统，这种事现在每周都在发生。作为 PM，你有责任在需求阶段就把这件事的真实代价算清楚，等开发到一半再发现就晚了。

代价其实就三层，每一层都能量化。延迟是最直观的，单 Agent 完成一个任务可能 10 秒，多 Agent 要互相通信、汇总结果，延迟经常翻 3 倍以上。成本是第二层，每个 Agent 都在烧 Token，Agent 之间的通信也在烧 Token，三个 Agent 协作的总成本往往是单 Agent 的 5 到 10 倍。可靠性是第三层，Agent 越多，单点出错导致整个流程崩溃的概率越高，链路一长，一个环节挂掉整条线就断了。

把这些数据做成一张表拿到老板面前，效果比讲道理强得多。单 Agent 方案延迟 10 秒、单次成本 0.05 元、任务成功率 85%，多 Agent 方案延迟 35 秒、单次成本 0.4 元、任务成功率 72%，同样的任务多花 8 倍的钱换来了成功率倒着走，老板自己会判断。绝大多数场景，单 Agent 加好的工具调用比多 Agent 协作更划算。

多 Agent 不是不能用，是要用在真正需要并行、真正需要不同专业能力分工的场景里。为了多而多，就是烧钱。劝退老板不靠嘴，靠数据，数据会替你说话。

## 17. Agent 自主规划的代价是延迟翻 3 倍，什么场景值得，什么场景纯属自嗨？

Agent 和传统工作流的核心区别，落在一个问题上：谁来决定路径。工作流是你提前把路径设计好，模型按路径走；Agent 是模型自己决定走哪条路，每一步都动态规划。自主规划的代价很直接，延迟翻倍、成本翻倍、可靠性下降，这三样都是要真金白银付的。

值得用自主规划的场景，特征也很一致。一类是路径无法预测的任务，改一段代码要动几个文件取决于代码本身，写一份调研报告要查几个方向取决于查到的内容，这些任务没法提前画流程图，只能让模型边走边看。另一类是探索性任务，做创意发散、做开放式研究、做多轮迭代优化，这类任务的价值恰恰在于模型能自己发现意外路径，提前焊死反而扼杀了它的长处。

纯属自嗨的场景也分两类。流程已经固定的，客服分流、订单处理、报表生成，路径是清晰的，用工作流足够，上 Agent 是给自己加成本。延迟敏感的更不能碰，用户等不了 30 秒的场景，用自主规划等于自杀。

判断标准就一条：这个任务的路径能不能提前画出来。能画出来就用工作流，画不出来才考虑 Agent。自主规划不等于高级，它就是昂贵，用对地方是杠杆，用错地方是负担。

## 18. Agent 产品的"可靠性"到底怎么衡量，少能力多可靠是什么意思？

做 Agent 产品的人很快会撞上一个反直觉的现象：能力强的 Agent，用户反馈往往更差。原因是它偶尔犯的错更触目惊心，用户会记更牢。反过来，能力弱的 Agent 只要稳定地犯同样的小错，用户反而能接受，因为他们摸清了它的边界，知道什么时候该用、什么时候不该用。这就是"少能力多可靠"的核心意思，Agent 的瓶颈从来不是能力天花板，是可靠性地板。

用户真正怕的不是 Agent 偶尔做不到，是它偶尔做到、偶尔又犯蠢，而且犯蠢的方式完全不可预测。这种不可预测性摧毁的是用户的判断模型，他们连"什么时候该信它"都建立不起来。

衡量可靠性要看三个东西。任务成功率的下限比平均成功率重要得多，一个平均 90% 但偶尔出灾难性错误的 Agent，比一个平均 80% 但 worst case 也能接受的 Agent 更难用。错误的可预测性也很关键，用户能不能摸清这个 Agent 在什么场景会出错、从而主动避开。再有就是错误的可恢复性，出了错之后用户能不能 easy 地回退、重试、接管。

少能力多可靠，不是说要去做弱 Agent，是要把工程精力从"让它更强"挪到"让它更不蠢"。地板抬高一寸，比天花板拔高一米更值钱。

## 19. 设计 Agent 时，怎么决定哪些步骤必须让人介入？

Human-in-the-loop 是 Agent 产品的标配，没有任何 Agent 能完全无人值守可靠运行。真正的问题不是要不要 HITL，是哪些步骤必须让人介入、哪些可以让 Agent 自己跑。判断这件事要看两个维度，一个是操作的可逆性，一个是错误的代价，两者交叉出一个清晰的优先级。

不可逆的高代价操作必须让人介入，发邮件、转账、删数据、发公告，这些操作一旦执行就没法撤回，代价又高，必须让人点确认。可逆的高代价操作可以让人事后审阅，Agent 先做、做完人 review、不满意可以退回，写邮件草稿、做方案初稿适合这种模式。不可逆的低代价操作可以让 Agent 自己跑但要留好日志，改一个配置、发一条提醒，错了影响不大，事后能追溯就行。可逆的低代价操作就完全交给 Agent，内部文档草稿、数据预处理这种错了也无所谓。

落到产品上，HITL 有四种典型模式可以挑。执行前审批适合高风险操作，Agent 提议、用户点确认、然后执行；先计划后执行适合长流程任务，Agent 列出步骤、用户修改、Agent 再执行；实时引导适合探索性任务，Agent 边做边问、用户随时插话；事后审阅适合低风险但需要质检的任务，Agent 做完、用户接受或退回。模式不是越多越好，一个 Agent 产品选一种主模式做透，比四种都做半套更有效。

## 20. 为什么说分块策略不是 PM 该操心的？PM 在 RAG 项目里到底该管哪三件事？

很多 PM 第一次做 RAG 项目，会一头扎进分块策略、向量模型选型、检索算法调参这些技术细节里。这些事该算法和工程团队操心，不该 PM 操心，PM 去碰这些等于在抢工程师的活，自己的活反而没人干。

PM 在 RAG 项目里真正该管的是三件事。第一件是定义"什么算好"，准确率要多少、召回率要多少、延迟要多少、成本上限划在哪，这些业务约束没人替你定，工程师只能给技术上的参考。第二件是搭评估体系，RAG 效果好不好要有一套可量化的评估集，里面要包含真实用户问题，覆盖常见 case、边界 case、对抗 case，这件事 PM 不做没人会做。第三件是设计兜底和反馈闭环，RAG 答不出来怎么办、答错了怎么收集负反馈，这些是产品决策，落在 PM 这一侧。

分块策略当然会影响效果，但那是工程师的 KPI。PM 的 KPI 是整个产品有没有解决用户问题，不是某一个参数有没有调到最优。把精力花在分块策略上，等于将军去前线挖战壕，挖得再好战役也输了。

## 21. 什么时候该用 RAG，什么时候该让模型直接回答？

RAG 不是万能药，有些场景用 RAG 是雪中送炭，有些场景用 RAG 是画蛇添足。判断要不要上，主要看三件事。

知识的新鲜度是第一个信号。模型训练数据有截止日期，之后的知识它根本不知道。业务一旦涉及频繁更新的内容，比如产品规格、政策、价格，就该用 RAG 把最新知识接进来。知识的私有性是第二个信号，企业内部文档、客户数据、行业情报，这些模型从来没见过，涉及这类内容也得靠 RAG。查询的精确度要求是第三个信号，用户问"公司差旅报销标准是多少"，要的是准确答案，不是模型凭常识猜出来的答案，这种就必须走检索。

反过来，让模型直接回答的场景也有几类。开放性创意任务，写文案、做头脑风暴、做代码 review，用模型的通用能力就够了，加 RAG 反而限制它发挥。通用知识问答也一样，常识问题、历史事件、科学原理，模型本来就训练过，不用再去检索。数据量极小的场景，知识库就那么几十条，直接塞进 prompt 比搭一整套 RAG 系统更高效。

一个常被忽略的维度是成本。RAG 的维护成本不低，文档更新、索引重建、向量库运维都是持续投入，知识库更新频率低于每月一次的，要慎重考虑。RAG 是工具，不是标配，该用的时候用、不该用的时候硬上，只会把简单的产品做复杂。

## 22. RAG 检索召回率低，PM 该怎么排查是哪一环出了问题？

RAG 检索效果不好，团队第一反应往往是换向量模型或者调检索参数，这是最常见的误区。召回率低，根因可能埋在四个环节里，每个环节的解法完全不同。

最靠前的环节是文档解析。PDF 的表格解析丢了，Word 的批注漏了，扫描件 OCR 错了，这些都会让知识从源头就是残缺的，解析环节出问题，后面再怎么调检索都救不回来。往后走是分块质量，分块太大检索时会塞进太多无关信息，分块太小关键上下文被切断，分块位置不对把一句话从中间切开，语义就碎了。再往后是查询理解，用户问"保险怎么弄"，模型不知道用户想问的是车险理赔还是健康险投保，查询没改写好检索方向就错了。排在最后的才是检索算法本身，向量模型选型、混合检索权重、重排序模型，这些是技术环节，但往往是最后才需要动的。

PM 排查这件事的正确顺序是从前向后。先抽样几个召回失败的 case，看看对应的原始文档解析得对不对；解析没问题再看分块是否合理；分块没问题再看查询有没有被正确改写；前三环都没问题，才动检索算法。从后向前调是团队最常见的浪费，调了三周检索参数，最后发现是 PDF 表格根本没解析出来。

## 23. 企业知识库做 RAG，怎么处理"文档互相矛盾"的问题？

这是企业 RAG 项目里最隐蔽也最头疼的问题。文档之间互相矛盾，通常不是因为有人故意搞乱，是因为企业知识本身就在持续演化：去年的政策今年变了，不同部门对同一件事的规定不一样，老板口头说的和写在文档里的有出入。RAG 把这些矛盾文档一股脑检索回来，模型就会在里面随机选一个回答，用户问退换货政策，可能这次答的是去年的、下次答的是今年的、再下次答的是某部门的内部规定。

治理这件事得分三步走。先做文档分层，把文档按权威性分级，最新版的官方政策优先级最高、部门内部规定次之、历史版本最低，检索时带着权威性标签，模型生成时优先采信高权威来源。接着做冲突检测，上线前用脚本扫一遍知识库，自动识别同一主题下相互矛盾的文档，人工裁决保留哪个，这件事不做矛盾就会一直潜伏，直到用户投诉才暴露。最后是回答策略，模型发现多个来源说法不一时，不要硬选一个，应该明确告诉用户"根据 A 文档是这个政策、根据 B 文档是那个政策、建议以哪个为准"。

诚实比假装权威更有用，文档矛盾这件事不是技术问题，是知识治理问题。PM 把它推给算法团队，永远解决不了。

## 24. 为什么团队做了 Evals 但还是上线翻车了？最常见的 5 种失败模式是什么？

做了 Evals 还翻车，这种事在 AI 团队里特别常见。原因通常不是 Evals 没用，是 Evals 做错了，错的方式就那么几种，反复出现。

最常见的坑是只看聚合指标。准确率 85% 听起来很漂亮，但哪 15% 错了、为什么错，不看具体 case 就没法发现那些反复出错的模式。紧跟着的另一个坑是评测集一成不变，业务在变、用户在变、模型在变，评测集却半年没动，跑出来的分数再高也是自欺欺人。评测和业务脱节是第三种，技术指标涨了业务指标没动，说明你评的是模型能力，评的不是用户价值。

还有两种更隐蔽的。没有 Bad Case 库是其中之一，失败的 case 是金矿，能告诉你模型在哪些场景会翻车，大部分团队不收集、不归因、不复盘，等于把金矿当废料倒了。最后一种是把评测当 QA 一个人的事，Evals 是 PM 加算法加业务方共同的责任，PM 尤其要主导评测体系的设计，甩给 QA 评测就会变成走过场。

Evals 不是上线前跑一下就完事的测试脚本，它是贯穿产品全生命周期的生产系统。做了 Evals 还翻车，说明 Evals 的设计本身就有问题，先修 Evals，再谈上线。

## 25. Evals 的 golden set 怎么建，500 条样本够不够？

Golden set 是 Evals 的核心资产，是模型迭代时的刹车片。任何新版本上线前，必须先在 golden set 上跑一遍，分数不退步才能放行。建 golden set 这件事，数量不是关键，质量才是，500 条精选样本比 5000 条粗糙样本有用得多。

几个原则绕不开。样本必须来自真实用户输入，用合成数据建评测集是自欺欺人，真实用户的问题分布和团队拍脑袋想出来的完全不同。样本要覆盖常见 case、边界 case、对抗 case 三层：常见 case 保证基本盘，边界 case 探测能力极限，对抗 case 防御误用和攻击。每个样本必须有明确的标注标准，什么叫好回答、什么叫差回答，要有 example-led 的评分细则，没有细则标注员之间的一致性会很差。样本还要持续更新，业务变了、用户变了、模型升级了，golden set 要跟着扩，三个月没更新的评测集基本已经失效。

500 条够不够，看业务复杂度。垂直场景 500 条精选样本够用，复杂业务可能要 1000 到 2000 条。判断够不够的标准，是看评测集能不能覆盖你担心模型出错的所有典型场景，覆盖不全就继续补。Golden set 是护城河，谁拥有最好的领域评测集，谁就能持续优化出最好的产品。

## 26. 为什么 LLM-as-a-Judge 给你的 Evals 打了 90 分，你却不敢直接信？它在哪 3 类问题上会撒谎？

LLM-as-a-Judge 从 2025 年起成了 Evals 的标配，用大模型当裁判给输出打分，快、便宜、能大规模跑。但直接信它的分数是很危险的事，LLM 裁判在三类问题上会系统性撒谎。

主观偏好类是第一类。问 LLM 裁判"哪个文案更好"，它的判断和你真正的用户可能完全不一致，它有自己的偏好，比如偏好长答案、偏好自信的表述。自我评估类是第二类，让 GPT-4 评估 GPT-4 的输出，分数会系统性偏高，模型对自己的输出有亲近感，这是已经被反复验证的偏见。评估维度模糊类是第三类，什么叫"回答质量"、什么叫"用户友好"，这种没有清晰 rubric 的维度，LLM 裁判基本是在瞎打分。

正确用法是把 LLM-as-a-Judge 当成大规模初筛工具，人工抽检做校准。先用人工标注一批金标准样本，然后让 LLM 裁判评同一批，计算两者的一致性；一致性不够就调 rubric、加 few-shot 例子，直到一致性达标。达标之后，LLM 裁判才能用来大规模打分，但即使如此也要保留 10% 到 20% 的人工抽检持续校准。LLM-as-a-Judge 不是银弹，它是把人工标注成本从一百美元每千条降到十美元每千条的工具，最后那 10% 还得人来兜。

## 27. 模型迭代后业务指标没涨，是 Evals 设计错了还是模型没进步？

这是 AI PM 最常遇到的判断题：技术团队说 Evals 涨了 10 分，业务团队说转化率没动，到底谁错了？两种可能都有，得分开查。

Evals 设计错的情况很常见。评测维度只看模型能力、不看用户价值，BLEU 分上去了用户根本不在乎，准确率涨了但涨的那些 case 用户根本遇不到。模型没进步的情况也有，新版模型在 golden set 上表现好，只是因为 golden set 里有几个 case 恰好被新模型覆盖了，换个真实分布的 live set 效果可能完全没变。

排查要分两步走。先验证 Evals 和业务指标的相关性，把过去几次模型迭代的 Evals 分数和对应的业务指标画成散点图，两者正相关说明 Evals 设计基本对，不相关就一定有问题。再看 Evals 的样本分布，如果评测集里 80% 是简单 case，模型在简单 case 上再涨分也没用，因为真实用户的问题分布不是这样。

修 Evals 的方向是把评测集往业务指标上靠，把真实用户的高价值场景做成评测样本，把业务关心的维度做成评分细则。模型团队和业务团队对不齐，根因往往是 Evals 没设计好，它本来就是技术语言和业务语言之间的翻译层。

## 28. 为什么大部分团队的数据飞轮在第一步（埋点）就卡住了？到底卡在哪？

数据飞轮的概念大家都懂：用户行为产生数据，数据反哺模型，模型变好，用户体验提升，产生更多数据。但大部分团队的飞轮，在第一步埋点就卡死了，卡在埋点这件事没人管。PM 觉得埋点是开发的事，开发觉得埋点是数据团队的事，数据团队觉得埋点的字段定义要 PM 来定，三方互相推，最后上线时埋点要么没做、要么做了但字段不全。更深层的问题是 PM 在 PRD 阶段就没把"要采集哪些信号"写清楚。

飞轮要转起来，至少要采集三类信号。显式信号是用户主动告诉你的偏好，点赞、点踩、评分、重新生成，意图清晰但覆盖率低，通常只有 1% 到 3% 的用户会给反馈。隐式信号是用户没说出来但通过行为告诉你的偏好，复制了哪段、编辑了哪些字、几秒后关掉页面，这类信号覆盖率能到 80% 以上，是飞轮真正的燃料，却被绝大多数团队忽略。结果信号是通过下游业务指标反映模型好坏的信号，用户是否完成购买、工单是否被关闭、代码是否被合并，直接关联业务但延迟长、归因复杂。

PM 不在 PRD 阶段把这三类信号的事件清单写清楚，开发就不会主动埋；开发不埋，飞轮从第一天就死了。飞轮不是算法团队的事，是 PM 的核心职责，这件事不做，产品上线即巅峰。

## 29. 企业知识库做 RAG，业务文档互相矛盾、核心知识在员工脑子里，你该怎么先治理数据？

这是企业 RAG 项目里最容易被低估的坑：技术团队搭好了 RAG 系统，PM 兴冲冲把企业文档灌进去，结果效果一塌糊涂。根因通常不是 RAG 系统不行，是数据本身就烂。

企业文档的典型问题挺集中。文档互相矛盾是一类，去年的政策和今年的不一样、不同部门对同一件事的规定不一样、老板口头说的和文档写的不一样。文档严重过时是另一类，系统早就升级了文档还在讲上一版，流程早就改了文档还在描述老流程。最棘手的是核心知识根本没在文档里，真正怎么做事存在老员工的脑子里、存在微信群里、存在那些从来没被写下来的默契里，这部分才是企业知识的主体。

治理这种数据得分三步走，顺序不能乱。先做文档盘点，把所有文档按主题归档，标注版本、时间、来源、权威性，同主题的文档如果有冲突立刻识别出来。接着做冲突裁决，找业务方的人来做仲裁，决定保留哪个版本、废弃哪个版本，这一步必须业务方参与，技术团队做不了。最后是隐性知识显性化，把核心员工脑子里的流程、规则、经验，通过访谈、复盘、案例整理的方式写下来，工作量大但不做 RAG 永远只能回答表面问题。

数据治理是 RAG 项目里最不性感但最关键的环节。技术上再炫，数据是烂的效果就是烂的，垃圾进、垃圾出，这句话在企业 RAG 里被反复验证。

## 30. 数据飞轮建设该先投标注、先投评测体系，还是先投数据管道？预算有限怎么排序？

这是 PM 在规划飞轮预算时最常遇到的决策题：三件事都要做，钱只够做一件，先做哪个。答案是先投评测体系。

为什么评测体系优先级最高？因为没有评测，你根本不知道后面做的事有没有用。标注了一大堆数据，怎么判断这批数据进训练之后模型变好了还是变差了？没有评测集就是盲飞。数据管道搭得再漂亮，数据流进流出，最终有没有变成产品价值的提升？没有评测集没法回答。评测体系是飞轮的刹车片，它让你敢做迭代、敢换模型、敢调整方向，没有刹车片的车开得越快越危险。

第二优先级是数据管道。评测体系有了，就要让数据能持续流转，埋点采集、信号清洗、样本入库这套管道要跑通。第三优先级才是标注，评测和管道都有了，才开始大规模标注，这时候标注的每一份数据都能被评测集验证有没有用。

预算有限的团队，最常见的错误就是先冲标注。招一批标注员标了几万条数据，结果发现评测集没建、没法验证这些数据有没有用，数据堆在那里，飞轮还是转不起来。先评测、后管道、再标注，这个顺序不能乱。

## 31. 什么时候该微调，什么时候继续用 RAG 就够了？

微调和 RAG 不是替代关系，是解决不同问题的工具，选错了白花钱还做不出效果。判断标准就一条：问题出在知识，还是出在风格。

知识问题用 RAG。模型不知道企业内部政策、不知道最新产品规格、不知道行业专属信息，这些都是知识缺失，RAG 把外部知识检索回来注入 prompt 就能解决。风格问题用微调，模型回答不够专业、语气不符合品牌调性、输出格式不统一、某些垂直任务的表现差一口气，这些都是模型行为需要调整，RAG 解决不了，只能微调。

更具体地看，微调适合几类典型场景。输出格式需要严格统一的时候，比如要求模型永远输出特定结构的 JSON、永远按某个模板写报告，RAG 给的是知识、改不了模型的输出习惯。垂直领域需要深度对齐的时候，法律、医疗、金融这些领域，模型需要用专业术语、按行业惯例组织回答，通用模型直接用语气和深度都不对。风格和品牌调性需要固化的时候，客服要有客服的语气、销售要有销售的语气，通用模型给不了这种差异化。

最常见的误区是把微调当万能药。模型不知道公司政策去微调，模型不知道最新价格去微调，这些都是知识问题，微调成本高、周期长，而且知识一更新又要重训。规则很朴素：风格问题微调，知识问题 RAG，记错这条规则预算就白烧。

## 32. HITL 标注的三档怎么选，纯人工、LLM-as-a-Judge、主动学习各值在哪？

HITL 标注是飞轮的关键齿轮，纯自动标注不可能成功，必须有人的参与。人怎么参与，大致分三档，选哪档看飞轮处在什么阶段。

飞轮刚启动时适合纯人工标注。标注员给每条输入输出对打分，1 到 5 分，成本高质量也高，目标是用最短时间积累 500 到 2000 条高质量种子数据。这批种子数据是用来校准后面的 LLM 裁判的，没有这批人工金标准，LLM 裁判就没法用。

飞轮进入成长期，可以切到 LLM-as-a-Judge 加人工抽检这一档。用大模型当裁判给每条输出打分，人工只抽检 10% 到 20% 的样本用来校准 LLM 裁判有没有跑偏。这一档的关键是给 LLM 裁判明确的评分细则，没有细则它就乱打分，有细则加 few-shot 例子，它的判断和人工的一致性能到 80% 以上，标注效率能提升 10 倍。

飞轮到了成熟期，可以上主动学习加人工兜底。模型主动挑出它最不确定的样本送人工标注，90% 的数据由 LLM 裁判处理，10% 不确定的送人工，目标是让人工只做最有价值的判断，把人力从重复劳动里解放出来。三档之间不是非此即彼，成熟的飞轮会同时跑三档，根据数据类型和重要性灵活切换。

## 33. 为什么说 AI 产品的护城河是 UX 不是模型？

"AI 产品的护城河是 UX 不是模型"，这个判断从 2023 年被提出，到 2026 年依然成立。

模型从来不是护城河，原因很简单：所有人都能调用同一个 GPT、同一个 Claude、同一个开源模型，你能做的事别人也能做。最有说服力的例子是 ChatGPT 本身。它爆火之前，类似的模型 Davinci 已经通过 API 开放将近一年，没有任何团队做出 ChatGPT。真正的护城河是两个 UX 创新：一个是流式输出，token 边生成边显示，模拟阅读和聆听的体验，比等全部生成完再显示友好得多；另一个是抽象掉状态管理，用户不用再把历史问答复制粘贴进新 prompt，系统自动维护对话状态，这才是 Chat 的真正含义。模型没变，UX 变了，产品就爆了。

判断一个 UX 改动算不算护城河，有一个朴素的标准：这次改进是不是把工作从用户身上转移到了产品身上，而且是直观地转移。如果答案是肯定的，这就是护城河；如果只是界面更好看，那只是装修。按这个标准，UX 护城河可以分三层来想。最浅的一层是让 AI 在场，产品有 AI 比没有强，这是入场券不是护城河，多数 AI 产品卡在这一层。中间一层是让 AI 实用，同样的目标用户用你的产品更容易拿到结果，这是体验差异化。最深的一层是让 AI 强大，产品让用户探索出模型自己都想不到的用法，这是效用差异化，也是最深的护城河。

AI 现在整体还停留在第一层为主、略带一点第二层，第三层远未展开。PM 的精力应该重点投在第二层和第三层，在第一层内卷是最低效的竞争。

## 34. Agentic UX 和传统 GUI 的根本差异在哪？

传统 GUI 是设计师预定义的界面，按钮在哪里、流程怎么走、错误怎么处理，都是设计阶段定死的。Agentic UX 走的是另一条路，界面由 Agent 动态生成，同一个入口、不同任务可能渲染出完全不同的界面。这个差异是根本性的，直接决定了产品设计的方法论得换一套。

几条差异是连在一起的。界面生成方式上，传统 GUI 是设计师画好让用户填表单、点按钮，Agentic UX 是 Agent 根据任务动态生成卡片、组件、画布。用户操作方式跟着变了，从点按钮、填表单、走固定流程，变成自然语言加多模态输入，用户说一句话 Agent 理解意图。流程控制也从固定流程树变成了 Agent 自主规划，每一步都可能是新的路径。错误处理从防御式设计变成了 HITL 加自愈，Agent 出错后人来兜底。信任建立的方式也不一样，传统 GUI 功能可用即可用，Agentic UX 必须显式校准，用户对 Agent 的信任度要和 Agent 的实际能力匹配。

这些差异落到产品设计上就是完全不同的工作方式。PM 设计传统 GUI，核心是画流程图、定状态机、设计组件；PM 设计 Agentic UX，核心是定义 Agent 的能力边界、设计 HITL 控制流、校准用户信任度、设计生成式 UI 的 schema。还在用传统 GUI 的思路设计 Agent 产品，等于用马车的思路设计汽车，看着都叫车，开起来完全是两回事。

## 35. 怎么校准用户对 Agent 的信任度，避免过度信任或不敢用？

信任校准这件事，从留存角度看比从设计原则角度看更刺骨，因为它直接决定用户用多久会走。

用户对 Agent 的信任度会自然滑向两个极端，不会自发停在中间，这是 Agent 产品特有的现象。过度信任的用户什么任务都丢给 Agent 做，包括那些 Agent 明显做不好的高风险操作，而 Agent 错一次，这类用户会立刻反转成彻底不信任，反转的速度比渐进流失快得多，几乎是一夜走人。信任不足的用户走的是另一条路，每个动作都怀疑，Agent 给的答案都要自己再核对一遍，Agent 的效率优势完全被消磨，最后觉得还不如自己手动做。两条路径终点都是流失，只是节奏不同。

校准的难处在于，它要对抗的不是用户的理性判断，是用户的情绪惯性。能力透明化是第一道关，onboarding 阶段就要让用户知道这个 Agent 能做什么、不能做什么，别为了卖货把它包装成全能助手，上线后用户发现做不到反弹会更狠。置信度可视化是第二道关，每次输出都带一个信心指标，80% 信心和 50% 信心在 UI 上要有明显区别，低信心的回答视觉上就得让用户警觉。错误期望管理是第三道关，高风险场景必须明确告诉用户这是 AI 输出、请核查，不能让用户误以为百分百可靠。

这三道关落到 UI 上，就是 onboarding 要诚实、输出要带信号、危险操作要二次确认。Agent 产品最大的 UX 红利不在让 Agent 看起来更强，在让用户准确理解 Agent 有多强，校准做好了留存才稳得住。
