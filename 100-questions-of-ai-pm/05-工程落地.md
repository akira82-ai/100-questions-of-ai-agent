# 五、工程落地：从 PRD 到商业化

## 71. 为什么说 AI 产品的 PRD 正在被 Prompt + Eval Spec 取代？三种新型需求文档分别长什么样？

传统软件的 PRD 核心是描述功能，用户点哪个按钮、系统返回什么结果、流程怎么走、状态怎么变。AI 产品的 PRD 这套描述方式开始失效，因为 AI 产品的核心不是功能，是行为，而行为是概率性的，没法用确定性的功能规格来描述。

取代传统 PRD 的是三种新型需求文档。Prompt Spec 描述的是模型在不同场景下应该怎么表现，包括角色设定、输入约束、输出格式、异常处理，说白了就是把模型当成一个新员工写一份工作手册。Eval Spec 描述的是怎么判断模型表现好不好，包括评估维度、评分细则、测试集构成、通过标准，就是把模型当成一个考生写一份评分标准。Behavior Spec 描述的是模型在不同边界场景下应该怎么做，包括已知边界的处理、未知输入的兜底、高风险操作的确认逻辑，就是把模型当成一个决策者写一份授权边界。这三份文档加起来才是 AI 产品的完整需求。

传统 PRD 描述的是"系统应该做什么"，新型文档描述的是"模型应该怎么做、怎么判断做得对、遇到没预料的情况怎么办"。PM 不会写这三份文档，需求就传递不清，开发按自己理解做，上线后发现和预期完全不一样。

## 72. 怎么在 PRD 里写清楚一个 Agent 的"行为边界"？

Agent 产品的 PRD，最难写也最重要的部分是行为边界。它定义的是 Agent 能做什么、不能做什么、在什么情况下必须让人介入，边界写不清楚 Agent 上线后就会乱来。

写行为边界要分三层，从外到里收紧。最外层是能力边界，这个 Agent 被授权处理哪些类型的任务，比如客服 Agent 只处理咨询类问题不处理退款操作，退款类任务要转给专门的 Agent 或人工。中间层是权限边界，这个 Agent 能调用哪些工具、每个工具的权限范围是什么，比如客服 Agent 可以查询订单状态但不能修改订单，可以发起退款流程但不能直接执行退款。最里层是风险边界，什么情况下 Agent 必须停下来等人确认，通常是涉及资金、数据、对外发送的高风险操作。

这三层要在 PRD 里写清楚，而且要写得具体不能只写原则。不要写"Agent 要避免危险操作"，要写"Agent 在执行以下操作前必须获得用户确认：金额超过 100 元的退款、对外发送邮件、删除用户数据"。边界写得越具体，开发和测试就越有依据，写得越模糊上线后越容易出事。

## 73. PRD 里该不该写 Prompt，写了开发还要不要做设计？

PRD 里写不写 Prompt，是 AI PM 最纠结的问题之一。写了怕开发照搬、不再做设计层面的思考，不写又怕开发完全按自己理解来、做出来的效果和预期差很远。折中的做法是 PRD 里写 Prompt 的设计意图，不写具体文案。

设计意图包括几个部分：这个场景下模型扮演什么角色、要完成什么任务、输出要符合什么约束、异常情况怎么处理。具体文案交给开发或算法团队去写和迭代，PM 把意图说清楚，执行交给专业的人。这个分工的原因是 Prompt 要持续迭代，上线后要根据效果数据反复调整，PM 把 Prompt 写死在 PRD 里反而限制了迭代空间。PM 该做的是定义"好的标准"，不是规定"怎么做"。

还有一个实操要点：PRD 里要附上几个示例对话，包括典型输入和期望输出。这些示例不是 Prompt 本身，是 Prompt 要达到的效果。示例对话比文字描述更有信息量，开发看到示例就知道 PM 想要什么风格、什么深度、什么格式的输出。

## 74. AI 功能的需求文档该写到多细，写粗了开发乱做怎么办？

AI 功能的需求文档，写粗了开发乱做、写细了又限制死没迭代空间，这个度很难拿。原则是确定的部分写细，不确定的部分写粗。

确定的部分是 PM 该拍板的，包括业务约束、用户场景、成功标准、风险边界，这些都要写得很细。业务约束包括延迟上限、成本上限、准确率底线，不写清楚开发按技术最优做可能完全不符合业务要求。用户场景包括典型用户画像、高频使用路径、已知边界 case，不写清楚开发做的产品可能完全偏离用户需求。成功标准包括怎么算成功、怎么算失败、评估维度是什么，不写清楚上线后没法判断做得好不好。风险边界包括哪些操作必须人确认、哪些场景必须兜底，不写清楚上线后出事就是大事。

不确定的部分是要持续迭代的，包括 Prompt 的具体写法、模型选型、检索参数、微调策略，这些写粗一点给开发留空间。PM 把确定的部分写细，开发的执行就有了边界；把不确定的部分写粗，迭代的空间就留出来了。写粗了开发乱做，根因通常是 PM 没把确定的部分写细。

## 75. AI 产品该怎么选定价模式？Token 计费、订阅、按结果，5 种模式的决策树怎么画？

AI 产品的定价，是 2025 到 2026 年最被低估的产品决策。传统 SaaS 的 per-seat 定价在 AI 产品上经常翻车，一个重度用户可能消耗 100 倍于轻度用户的算力，per-seat 定价会让重度用户变成成本黑洞。选定价模式要看三个维度：成本结构、用户使用频率、结果可度量性。

成本结构决定大方向，边际成本高的产品不能用 per-seat，边际成本低的还能撑。用户使用频率影响接受度，高频产品用户能接受订阅或套餐，低频产品用户更愿意按次付费。结果可度量性决定能不能走到最理想的按结果计费，结果容易度量的可以尝试，结果难度量的只能退回订阅或按次。

五种主流模式各有适用场景。Per-seat 适合协作型 SaaS，用户主要价值在多人协作；按 Token 计费适合 API 类产品，用户是开发者能理解 Token 成本；订阅制适合高频 C 端产品，用户希望不限次使用；按次计费适合低频工具型产品，用户偶尔用一次；按结果计费适合任务明确的产品，用户只为完成的任务付费。决策树的逻辑是先看结果能不能度量，能度量优先按结果；不能度量看使用频率，高频用订阅、低频用按次；都不是最后退回 per-seat。定价不是拍脑袋决定的，是根据成本结构、用户行为、结果可度量性推导出来的。

## 76. 为什么按 Token 收费的产品，用户越用越想跑？

按 Token 收费听起来很合理，用户用多少付多少、成本和价值对齐。实际上这种定价方式会让用户越用越想跑。根因是 Token 是用户看不懂的单位，用户不知道一次对话消耗多少 Token、不知道一个月会用多少、不知道下个月的账单会是多少。这种不确定性让用户焦虑，焦虑的用户会减少使用、会寻找替代品、会跑到竞争对手那里，按 Token 收费的产品用户留存天然就低。

解决这个问题有几个方向。最根本的是把 Token 翻译成用户能理解的单位，不要让用户看 Token 数，让用户看"消息数"、"任务数"、"文档数"，用户能理解"这个月发了 500 条消息"，看不懂"这个月消耗了 50 万 Token"。再者是给用户明确的额度上限，订阅制加额度、超出再按量收费，用户知道自己最多花多少钱焦虑感就降下来了。还有一层是给用户实时的成本反馈，用户每用一次立刻看到这次消耗了多少额度、剩余多少，透明的反馈能让用户有掌控感。

按 Token 收费不是不能用，是不能让用户直接面对 Token，要把 Token 包装成用户能理解、能掌控的东西。用户愿意为价值付费，不愿意为不确定性付费。

## 77. 企业客户的"模型能不能私有化部署"是真需求还是谈判筹码？

做企业级 AI 产品的 PM，一定会遇到客户问"模型能不能私有化部署"。这个问题有时是真需求，有时是谈判筹码，要分辨清楚才能给出对的回应。

真需求的情况有几种。监管要求高的行业是真需求，金融、医疗、政务这些行业的数据合规要求极严，数据出域是被禁止的。数据敏感度高的场景也是真需求，客户的核心数据、商业机密、用户隐私不能发给第三方模型。超大规模使用同样是真需求，客户调用量极大，按 API 计费成本远高于自建，这是经济考虑。

谈判筹码的情况也分几种。采购流程需要的是一种，有些企业的采购流程要求至少询问私有化选项，不代表真要买。压价手段是另一种，客户用私有化作为谈判筹码逼你给 API 方案打折。还有一种是技术好奇，客户的技术团队想了解私有化的可能性，不代表业务方真的需要。

分辨的方法是问客户的具体场景和数据量。能说清楚具体场景和数据量的通常是真需求，说不清楚的大概率是谈判筹码。真需求的客户值得投入做私有化，谈判筹码的客户解释清楚两种方案的差异就好，不要被带偏。

## 78. AI 产品的毛利率能做到多少，Token 成本的底线在哪？

AI 产品的毛利率，是创业公司和投资人都特别关心的数字。传统 SaaS 的毛利率能做到 70% 到 80%，AI 产品的毛利率通常只有 40% 到 60%，差距主要在 Token 成本。

Token 成本的底线取决于三个因素。产品类型是最基础的，工具型产品的 Token 成本占营收比例通常在 15% 到 20%，订阅型产品在 25% 到 30%，Agent 类产品在 35% 到 45%。用户结构影响很大，重度用户和轻度用户的成本差异极大，20% 的重度用户可能消耗 80% 的 Token 成本，用户结构不健康毛利率就撑不住。模型选型决定效果和成本的平衡，用最贵的模型 Token 成本高但效果好，用便宜的模型成本低但效果可能差。

提升毛利率有几个方向。模型路由能把平均 Token 成本降下来，简单任务用小模型、复杂任务用大模型。缓存策略能让高频问题不再烧 Token，重复调用直接返回缓存。Prompt 压缩能把 prompt 写得更精炼，减少不必要的 Token 消耗。用户分层定价让重度用户收更高的费用，覆盖更高的成本。AI 产品的毛利率要做到 60% 以上，需要在这几个方向上持续优化，不做优化 40% 都撑不住。

## 79. 多模型路由：PM 怎么决定一个请求用 GPT-5 还是 Qwen-7B？

多模型路由听起来很美，简单请求用小模型省钱、复杂请求用大模型保质量。实际操作起来，PM 要面对的核心问题是怎么判断一个请求是简单还是复杂。判断错了，简单请求用了大模型白烧钱，复杂请求用了小模型质量崩盘。

判断请求复杂度有几条路径，各有取舍。基于规则是最简单的，请求长度低于某个阈值、关键词命中某些模式、用户画像属于低复杂度群体就用小模型，准确率不高但实现快。基于小模型预判准确率高一些，先用一个极小的模型给请求打一个复杂度分，分数低的走小模型、分数高的走大模型，代价是增加延迟。基于历史数据是最准的，同类请求历史上用小模型的效果如何、用大模型的效果如何，根据数据决定，但需要先积累足够的数据。

不管哪条路径，路由错误的代价都要算清楚。路由错误率超过 15%，节省的 Token 成本就被质量损失抵消；超过 30%，模型路由就是负优化。PM 决定要不要做多模型路由，要看产品的路由错误率能不能控制在 15% 以内，做不到不如不做。模型路由不是必选项，对很多产品来说，单一模型加好的 prompt 工程比多模型路由更划算。

## 80. 缓存策略能省多少 Token，什么场景下缓存反而是负优化？

缓存是 AI 产品省 Token 成本最直接的手段。原理很简单，用户问的问题有大量重复，把高频问题的答案缓存起来，下次直接返回缓存不再调用模型。缓存能省多少 Token 取决于问题的重复率，客服场景重复率高缓存能省 40% 到 60%，创意写作场景重复率低缓存可能只能省 5% 到 10%。

但缓存用不好反而是负优化，常见的坑有四种。缓存过期是最常见的，知识库更新了缓存的答案还是旧的，用户拿到过期信息比没有缓存还糟，解法是给缓存设过期时间、知识库一更新立刻清缓存。命中率低还硬缓存是第二种，缓存系统本身有成本，存储、查询、维护都要花钱，命中率低于 20% 缓存系统的成本可能超过节省的 Token，解法是监控命中率、命中率低的场景不要硬上。缓存污染是第三种，错误答案被缓存了之后所有用户都拿到这个错误答案，解法是缓存前验证答案质量、错误答案绝不进缓存。延迟增加是第四种，缓存查询本身要时间，缓存没命中还要回源调用，总延迟可能比直接调用模型还高，解法是缓存查询要做得很轻、缓存系统本身的响应时间要远低于模型调用。

缓存是省钱的利器，但不是免费午餐，用得好能省大量成本，用不好反而添乱。

## 81. 每加一个客户开一个模型分支，你什么时候该说"不"？

做企业级 AI 产品的 PM，一定会遇到这种情况：每来一个大客户就要求定制化，定制化的极端形式是单独开一个模型分支、用客户自己的数据微调。一个客户开一个分支，短期看是贴心服务，长期看是工程灾难。

问题会从三个方向同时压过来。维护成本爆炸是最直接的，每个分支都要单独维护、单独迭代、单独部署，10 个客户就是 10 套系统，研发团队根本忙不过来。版本同步地狱是第二种，主模型升级了每个分支都要同步，同步过程会引入新的 bug，每个分支的 bug 还不一样。数据隔离失效是第三种，不同客户的数据混在同一个代码库里，一旦出问题就是合规灾难。

PM 要学会说"不"，但说"不"要有策略不能硬拒绝。说"不"的时机是定制化需求的通用性极低、只有这一个客户需要，这种需求做了对其他客户没价值，维护成本却要所有人承担。说"不"的方式是提供替代方案，比如用 prompt 定制代替模型微调、用配置参数代替代码修改、用插件机制代替核心功能改造。说"不"的底气是产品有清晰的产品边界和能力地图，客户要的东西超出了边界，可以引导客户用标准能力组合实现。不会说"不"的 PM，产品会被一个个客户带歪，最后变成一堆定制项目的集合，没法维护也没法演进。

## 82. PM 该对 AI 安全负什么责？Red Teaming 和 Guardrails 该怎么写进产品设计清单？

AI 安全这件事，不是 QA 的事，也不是算法团队的事，是 PM 的核心职责之一。PM 对安全的责任，体现在把安全写进产品设计的每一个环节，让它从"事后补救"变成"事前内置"。

这件事要贯穿产品全生命周期。需求阶段，PRD 里要有专门的安全章节，明确产品的安全边界，哪些内容不能输出、哪些操作不能执行、哪些数据不能收集。设计阶段，产品设计要内置 Guardrails，输入层做安全过滤、输出层做安全审查、高风险操作必须人确认。测试阶段，上线前要做 Red Teaming，主动找一批人从各个角度攻击产品看哪里会崩。上线后，持续监控安全事件、建立应急响应机制，出事能快速定位、下架、修复、通报。

把 Red Teaming 和 Guardrails 写进产品设计清单，关键是具体不要抽象。不要写"产品要做好安全防护"，要写"输入层用 X 分类器过滤违规内容，输出层用 Y 分类器审查敏感话题，高风险操作必须经用户二次确认"。不要写"产品要防 Prompt 注入"，要写"system prompt 不对用户可见，输出层识别'repeat the above'等注入模式并拦截"。安全设计越具体落地越有效，写得太抽象开发不知道该怎么做，最后就是没做。

## 83. AI 产品团队该按功能分还是按模型分？

团队组织架构是 AI 产品落地的重要决策，按功能分还是按模型分是两种主流思路。

按功能分是传统互联网产品的思路，前端、后端、算法、产品、设计各司其职。这种分法的好处是职责清晰，每个团队专精自己的领域；坏处是跨团队协作成本高，AI 产品的迭代节奏快，跨团队协调经常跟不上。按模型分是 AI 原生团队的思路，每个核心模型或核心场景配一个跨职能的小组，里面有 PM、算法、工程、设计，端到端负责。这种分法的好处是迭代快、小组内能快速决策，坏处是资源可能重复配置、不同小组之间缺乏协同。

选择哪种分法，要看产品阶段和规模。早期产品场景少模型少，按模型分更高效，小组制能让小团队快速迭代、找到 PMF。成熟产品场景多模型多，按功能分更合理，职能化能避免资源浪费、提升专业深度。还有一种混合分法值得考虑：核心场景用小组制，基础设施用职能制，小组制负责快速迭代，职能制负责底层能力建设。没有标准答案，团队架构要跟着产品形态走，不能反过来。

## 84. 一个 AI 产品从 0 到 1，最少需要哪几个角色？

做 AI 产品从 0 到 1，团队最小化的配置是什么？这是很多创业者和大厂内部创新团队都问的问题。最小化不等于最少，是少到不能再少还能把产品做出来。最精简的配置是四个角色。

AI PM 负责定义产品、定义评估体系、定义用户场景、做跨团队协调，这个角色不可或缺，没有 PM 团队就失去方向。算法工程师负责模型选型、prompt 工程、RAG 搭建、效果调优，是 AI 产品的技术核心。全栈工程师负责产品开发、API 对接、前端 UI、后端服务，早期产品不需要细分前端后端，一个全栈能搞定。设计师负责 UX 设计、交互设计、信任校准设计，AI 产品的 UX 比传统产品更重要，不能省。这四个角色是底线，少一个产品就做不出来，或者做出来不能用。

随着产品发展，角色会逐渐细分：算法团队会分出数据工程师、ML 工程师，工程团队会分出前端、后端、运维，产品团队会分出增长 PM、商业 PM。但从 0 到 1 阶段，四个角色足够，人多反而会拖慢决策、增加协调成本。创业团队最常见的错误是人招太多，以为多招人能加快进度，实际是增加管理成本、拖慢迭代速度。

## 85. 老板说"我们也搞个 Agent"，PM 该怎么把这句话变成可执行需求？

老板说"我们也搞个 Agent"，是 AI PM 最常听到的"需求"之一。这句话信息量极低，但引发的连锁反应极大。如果直接开始做，大概率会做出一个四不像的产品，老板自己也不知道要什么，做出来肯定不满意。PM 的职责是把这句话翻译成可执行需求，办法是把四个问题逐个问清楚。

最前面要搞清楚的是背景：老板为什么突然要搞 Agent？是看了竞品发布，是听了投资人建议，是客户在问，还是内部有真实痛点。背景不同要的东西完全不同，竞品压力要的是快，投资人建议要的是故事，客户在问要的是能卖，内部痛点要的是能解决真问题。接着要搞清楚场景：Agent 要解决什么具体问题、服务什么用户、在什么环节介入，场景说不清楚就不要开始做，做了也是白做。再往后是约束：预算多少、周期多长、团队能力覆盖到哪、能接受多大程度的失败，约束清楚了才知道做多大的事，不清楚约束就开工，做到一半发现资源不够项目就废了。最后是成功标准：什么算做成、什么算没做成，是上线就算，还是要达到某个指标，还是要产生收入，标准不清楚项目永远没有终点，团队一直做老板一直不满意。

把这四步搞清楚，写一份一页纸的需求文档给老板确认。确认了再开工，不确认就一直追问。直接开工是最大的浪费，先确认再开工，是 PM 保护团队的基本功。
