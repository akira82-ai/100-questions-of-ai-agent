# 第四章 · 本地能跑的东西，怎么让别人也用上？

## 45. demo 在我电脑能跑，怎么让全世界都能访问到

直答：把代码从你电脑搬到一台一直开着的公共服务器上，再给它一个网址，别人就能点了。

你本地跑的那个地址，比如「localhost:5173」，只有你这台电脑认得。它像你在家厨房炒好的一盘菜，闻着香，但邻居进不来。要让别人吃到，得把菜放到一个对外营业的窗口。

现在的工具把这件事压得很扁。用 Lovable、Bolt、Replit 这类浏览器工具，做完点一下「Deploy」就能直接拿到网址，全程不用碰命令行。用 VS Code 加 Copilot 这类方式，常见做法是先把代码推到 GitHub，再连到 Vercel 这种托管平台，一两分钟生成「xxxx.vercel.app」这样的网址。拿到链接发朋友圈，朋友点开就是你的应用。

先别纠结"服务器"三个字吓人。对绝大多数个人小工具，你不需要自己买机器、装系统。平台替你管着那台一直开着的电脑，你只管把做好的东西交上去。等真有上千人在用、要定制复杂逻辑了，再考虑自己租服务器也来得及。

这一步的核心动作就一句：让代码离开你的电脑，住进一个别人也能访问的地址。

## 46. 部署要租服务器吗，有没有免费的让我先试水

直答：对个人试水，绝大多数情况不用租服务器，免费托管平台够你跑通第一个能分享的应用。

「服务器」听起来像要买台机器放家里。其实现在的主流做法是「托管」：别人养好了一堆一直开着的电脑，你按使用量蹭一点。Vercel、Netlify 都给免费额度，部署纯前端小工具基本零成本。用浏览器类工具（Lovable、Bolt、Replit）时，部署按钮背后就是这些平台，你连它们叫什么都不知道也能上线。

什么时候才需要自己租？两种：做微信小程序得按微信规则走它的发布流程，和网页不是一回事；或者你的东西变复杂了，要跑自定义后端、接很多第三方、流量大到免费额度兜不住。即便到那步，也是免费起步、不够再升级。

建议很直接：第一个能分享的应用，坚决走免费托管。先把「别人能打开我的东西」这个正反馈拿到手，钱的事等它真有用户了再想。

## 47. "上线"和"能用的上线"区别在哪，别被自己骗了

直答：能打开只是一个开头。真正"能用的上线"，还得有真实数据、用户登录、容错和备份，缺一样都不算交差。

新手最容易的自我欺骗，是 localhost 跑通、部署上去朋友点开截图说"哇能用了"，然后就觉得"我做完了"。这顶多算「能打开」。一个只存在你自己脑子里的假数据、刷新就清零、谁都能看谁的数据的"上线"，离能用差得很远。

这里有一道「最后一公里墙」：做 demo 那 70% 很爽，剩下 30%（真实数据库、用户认证、支付、生产环境部署）才是多数人的滑铁卢。做完一个记账小工具，本地看着完美，一部署发现数据存不住、多人用就串号，这才意识到前面只是壳。

判断你到没到"能用的上线"，问自己四句：数据存在哪、刷新还在吗？别人登录能看到彼此的信息吗？涉及钱或隐私的功能跑通且安全吗？万一我电脑坏了，东西还能找回吗？四句有一句答不上，就还没上线，只是摆了个样子。

别被自己骗。能打开值得高兴，但高兴完，那 30% 的功课得老老实实补上，东西才真算做出来了。

## 48. 数据存哪，Excel 不够用了怎么办，数据库到底是个啥

直答：小工具先用浏览器自带的 localStorage 顶着，要多人共享、长期保存，再上云数据库（比如 Supabase），不用一上来就懂 SQL。

你做的第一个应用，数据可以存在用户自己的浏览器里，这种方式叫 localStorage。好处是零配置、不用后端，刷新页面数据还在。代价是换台设备、清了缓存，数据就没了，而且只有那台设备看得到。做单机小工具，它够用。

当你需要「用户 A 存的东西，用户 B 看不到，但 A 换手机还能看见」，本地存储就不够了，得有个独立、永远在线的地方统一存。那个地方就是数据库，可以想成一个永远不关、谁都按权限存取的超大表格仓库。

对非技术人最友好的云数据库是 Supabase，它把数据库包装成能看懂的界面，很多工具也原生支持。你不用先学 SQL，告诉 AI「帮我把任务存到 Supabase，每个用户只能看自己的」，它就能接上。

一句话：先 localStorage 跑通，要多人共享再上 Supabase。数据库不是高深物，就是数据住的、大家按规矩进出的房子。

## 49. 用户要注册登录，账号密码怎么存才不泄露

直答：别自己写登录逻辑，直接用现成的认证服务（Supabase Auth、微信登录等），并且绝不把密码明文存进数据库。

新手最容易踩的坑，是让 AI「帮我做个登录功能」，AI 偷懒把账号密码原封不动写进表，谁都能读。这等于把用户家门钥匙挂在门口。正确做法是：密码在存进去之前就做哈希处理（变成一串不可逆的乱码），哪怕数据库被拖走，黑客也还原不出原密码。

更稳的路子是不碰这块。Supabase、Firebase 这类服务自带「认证」模块，注册、登录、找回密码、权限隔离全帮你做好了。你只需告诉 AI「用 Supabase Auth 做登录，每个用户只能看自己的数据」，剩下的它接好。微信小程序更是直接调微信的登录能力，连密码都不用存。

还有一个要命的开关叫「行级权限」（RLS）。它决定用户 A 能不能看到用户 B 的数据。不打开它，所有人看到的是同一份数据，隐私就裸奔了。上线前必须确认它开着，并且规则是「每人只看自己的」。

账号安全，新手最佳策略是：用大厂现成的认证，别自己造轮子。你省下的是风险，不是麻烦。

## 50. 什么是后端，为什么光有个界面远远不够

直答：前端是你眼睛看到的界面，后端是藏在服务器里、真正干活的"后厨"。只有店面没有后厨，客人点单只能看菜单。

你用 AI 做出来的页面、按钮、颜色，都属于「前端」。它负责把东西漂亮地摆在用户面前。但一个能记账、能发消息、能按账号区分数据的应用，光有界面不够。谁在算"这个月花了多少"？谁在记住"这条数据属于哪个用户"？谁在发那条短信？干这些活的，是「后端」。

打个比方：前端是餐厅的堂食区和菜单，后端是后厨、仓库和账本。客人（用户）只看见前厅，但没后厨，点任何菜都做不出来。很多新手做出来的东西"看着像那么回事，一用就空"，就是因为只有前厅、没有后厨。

好消息是，vibe coding 里后端也能让 AI 生成。你可以说「帮我做一个后端，负责保存任务、按用户隔离数据、提供接口给前端调用」，AI 会写出服务端代码。对简单工具，甚至可以用云数据库（Supabase）部分替代自己写后端，省掉一堆活。

记住这个区分：你看到的是前端，你看不到但让它"真能干活"的是后端。评估一个应用靠不靠谱，别只看脸，要看后厨在不在、干不干净。

## 51. 想接支付、地图、短信，AI 能帮我接上吗

直答：能接，而且 AI 很擅长写"对接第三方能力"的代码。但这类活牵扯密钥和钱，你得在旁边把安全关。

支付、地图、短信、天气，AI 不会现造一套。它们是大厂早就做好的"公共能力"，靠一种叫接口（API）的方式对外提供，AI 的长项就是读懂接口说明书、写出调用代码。你说"接入微信支付，用户下单后调起付款"，它真能接上。

这类对接有个危险共性：都要"钥匙"（API 密钥），往往连着你的钱或用户隐私。一个真实教训：有人让 AI 接第三方服务，AI 图省事把密钥写死在代码里、还开了调试模式，密钥和内部链路全暴露。接支付短信时，这种疏忽可能让你半夜被扣费、或让用户数据外泄。

所以原则是：让 AI 写对接代码可以，但"钥匙"放环境变量、不写进会公开的前端代码；涉及钱的功能上线前用小金额真测一遍。

一句话：AI 能帮你把支付地图短信接上，这部分大胆交给它。钥匙怎么保管、上线前怎么验，这部分你亲自盯。

## 52. 接口（API）是什么，为什么做真东西它绕不开

直答：API 就是两个程序之间的"点菜单"。前端通过它向服务器或第三方"点单"，拿回要的数据或能力，做真东西基本绕不开。

你看到的页面不会自己变出数据。当你刷新一个待办列表，是页面悄悄发了个请求给后面：「把用户 A 的任务给我」。后面回一句：「这是，三条规定」。这套"你点单、我出餐"的约定，就是 API。前端是顾客，后端或第三方是厨房，API 是厨房窗口的菜单和取餐口。

为什么绕不开？因为现代应用是拼起来的。你的页面负责好看，数据存在别处，支付地图在另一家。把它们缝在一起的线就是 API。你让 AI 做"任务列表从后端取数"，本质就是在写"按菜单点单"的代码。

新手在这里最常翻车的是"地址写死"。比如做一个积分商城，AI 把接口地址直接写死成开发环境的，上线后全失效。正确做法是把地址放进环境变量，开发、测试、生产各用各的。这一处没交代清，部署上去就是一片空白。

你不用懂 API 底层协议，但得建立这个画面感：东西之间靠"菜单窗口"传话。下次 AI 说"我调了 API"，你知道它是在按菜单点单，而不是变魔术。

## 53. 上线后网址没小锁（https），会有什么事

直答：没小锁意味着网址还是 http，用户填的账号密码在路上裸奔，浏览器还会给访客亮"不安全"红灯，很多人直接关掉。

你注意过地址栏左边的小锁吗？带锁的网址以 https 开头，没锁的是 http。差别在于：https 给数据传输加了密，像给用户和服务器之间修了一条只有他俩有钥匙的管道；http 则是明信片，路上谁都能偷看。

对做应用的人，没 https 有两笔实打实的损失。一是安全：用户在你这登录、填手机号、输地址，明文传输可能被中间人截走。二是信任：现在浏览器对 http 网站直接标"不安全"，朋友点开你分享的链接，第一眼看到红色警告，八成以为遇着钓鱼站，扭头就走。

好消息是，正规托管平台（Vercel 等）部署时默认就给你配好 https，你几乎不用管。自己租服务器的人，才需要手动申请证书、配置加密，这一步常是新手部署卡点之一。

检查清单就一句：把链接发别人之前，先确认地址栏是带锁的 https。没锁，先别发，回去把加密打开。这不是锦上添花，是上线的最低门槛。

## 54. 部署失败红字说"构建错误"，新手怎么自救别慌

直答：构建错误九成是依赖或配置问题，别自己瞎改，把完整红字贴给 AI，让它定位，比你看懂每一行快得多。

部署到线上时，平台要先"构建"一遍你的项目（把零散代码打包成能跑的产物）。这步挂了，就会甩一堆红字，新手一看就慌。先稳住，红字是机器在报告卡在哪步，不是你人废了。

最常见的原因有两类。一是依赖缺失或版本不对，比如本地有某个包、线上没装，或者 Node 版本差一截。二是配置文件写错，有个真实案例：有人让 AI 生成项目配置，AI 默认开了调试模式、把环境数值全写死，没区分开发和生产，一部署就启动失败，每次迭代还得人工核对好几份配置。

自救标准动作：复制终端里从错误开头到结尾的完整文本，连同"我想实现的效果""我试过啥"一起贴给 AI，让它修。你不需要先搞懂每一行。但有个前提：别为了修一个错，手动去改那好几份散落的配置文件，那是给自己挖更深的坑。让 AI 统一从环境变量读配置、自动切环境，一次根治。

记住：构建错误是部署的常态，不是你能力问题。贴给 AI，它比你先慌之前就修好了。

## 55. 我一个人能维护上线后的东西吗，要花多少精力

直答：简单工具一个人完全能维护，精力主要花在"定期更新、看错误、做备份"三件事上，复杂度决定负担大小。

有个做微信小程序的开发者算过一笔账：他那个完整产品，约八成代码是 AI 生成的，两成是他的判断和修正。上线后他一个人维护，靠的是把 AI 当"虚拟开发团队"用，而不是自己扛所有技术细节。所以人力上你不是孤身一人，AI 一直在。

真正消耗精力的不是写，是这三件：一是更新，平台或依赖升级时跟着调，不然某天突然跑不起来；二是看错误，用户报"打不开"你得能查日志、把报错贴给 AI；三是备份，代码推到 GitHub、数据定期导出，别一夜回到解放前。

精力多少，取决于你那 30% 有多重。纯前端小工具，一个月可能就花几小时。一旦上了真实数据库、支付、自定义后端，运维面变宽，要操心的事成倍涨。这也是为什么前面一直劝你：先做个轻的、能用的，别一上来堆复杂。

结论：一个人能维护，但别指望"上线就完事"。把它当养了盆植物，偶尔得浇水（更新）、除草（查错）、挪窝（备份）。轻的应用，浇水频率很低，你扛得住。

## 56. 用户多了网站变慢，是我的锅还是 AI 的锅

直答：慢的锅大概率在架构和数据库，不在 AI 写代码的"手艺"。多数托管平台会自动扩容，真正卡的是你没设计好的数据层。

先分清两层。一层是"算力够不够"：用 Vercel 这类平台，访问量涨了它会自动加机器，这一层你基本不用管，慢不到这。另一层是"你的数据怎么存的、逻辑怎么写的"：比如所有请求都去查同一个没加索引的大表，或者每次刷新都重算一遍复杂统计，人一多这就崩。这层是设计问题，AI 只是按你的需求把设计落地了。

有个参照：有人在架构设计上让 AI 规划"可扩展"的结构，从一开始就留好余地，后来用户涨了没太慌。反过来，若你需求描述就含糊、AI 为了凑功能堆了一堆低效查询，人一多自然卡。所以慢的锅，源头常在你的需求清不清晰、架构有没有留扩展余量，不在 AI 写代码的"手艺"本身。

排查顺序：先看平台后台的流量和响应时间曲线，确认是不是真到了扩容边界；再看是不是某个具体接口特别慢。把慢的那个接口贴给 AI，让它查"有没有没必要的全表扫描、能不能加缓存"。

一句话：运行时快慢主要看你设计的地基，AI 不背这个锅。

## 57. 怎么把本地 demo 安全推到线上，别把密钥一起传上去

直答：密钥放环境变量、用忽略文件排除敏感文件、绝不写进会公开的前端代码，这三道关守住，demo 推上线就不会变"泄密现场"。

部署往往要把代码推到 GitHub 再连平台，这一步最常见的事故，是把本该保密的东西一起传上去了。有个数据触目惊心：安全机构扫了一批平台做出来的应用，约一成五的公开代码里能直接看到第三方密钥，等于把付费接口密码贴在自家门口。

正确做法就三道。密钥（API key、数据库密码）不要写死在代码里，放进环境变量，代码只留「去环境变量取钥匙」的占位。用忽略文件（.gitignore）把本地配置和密钥文件排除，确保 git 推送时它们根本不会出门。前端代码是用户浏览器里能看到的，任何密钥都只能放服务器端，绝不能出现在前端。

给 AI 的指令写清：「所有密钥从环境变量读取，配置区分开发和生产，密钥文件加入忽略列表，前端不含任何密钥」。

推上线前花两分钟查：代码里搜得到明文密钥吗？忽略文件配了吗？两问答"否"，你才安全。

## 58. 上线前有没有"检查清单"让我照着过一遍再发布

直答：有。一份十分钟能过的清单，覆盖功能、数据、密钥、加密、备份五块，勾完再发，能挡掉多数新手翻车。

功能块：每个按钮、每条流程亲自点一遍，别信"AI 说做好了"。很多人上线才发现某个功能只在假数据下跑通过，真数据一接就崩。验收标准就一条：用真实会使用的方式走一遍，而不是看它"看起来对"。

数据块：确认用户密码不是明文存的，行级权限（每人只看自己）开着，换设备登录数据还在。这三样缺一个，用户隐私就裸奔。

密钥块：搜一遍代码，确认没有明文密钥；密钥走环境变量；前端代码里搜不到任何钥匙。那成一成五泄露的应用，全栽在这块没过。

加密块：链接是带锁的 https 吗？没锁先别发，浏览器会吓退你的用户。

备份块：代码推到 GitHub 了吗？数据库有导出或自动备份吗？万一哪天服务器没了，你能从哪找回？

把这五块写成你自己的上线清单，每次发布前勾一遍。它不保证万无一失，但能拦下绝大多数"我明明测过了"的低级事故。清单的价值不在复杂，在于你真的照着做。

---

本地能跑只是拿到入场券。那 30% 的上线功课——真实数据、登录、加密、备份——才是决定你到底算不算真的"做出来"的东西。
