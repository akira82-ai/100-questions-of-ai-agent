# 四、进阶：Skill·MCP·插件·自动化·操控电脑

## 52. Skill、MCP、插件到底有什么区别

这三个词是新手绕不过去的坎，混在一起根本分不清谁是谁。用大白话讲一遍。

插件是能力安装包。你装一个插件，Codex 就多了一个新能力。比如装了 Presentations 插件，Codex 就会做 PPT；装了 Spreadsheets 插件，Codex 就会处理表格；装了 Chrome 插件，Codex 就能操作你已登录的浏览器。插件解决的是"能不能做某类事"的问题。

Skill 是工作流说明书。Skill 的作用不是给 Codex 加新能力，Skill 把"某件事怎么做"的方法固化下来。比如你有一套自己整理数据的标准流程，写成 Skill 之后，下次只要一句话 Codex 就按你的流程来，不用每次重新解释。Skill 解决的是"同类任务怎么做"的问题。

MCP 是连接外部工具的通道。MCP 的作用是让 Codex 能读到外面的数据和工具——你的 GitHub、你的飞书、你的 Notion、你的数据库。MCP 解决的是"连接什么外部资源"的问题。

一句话记忆：**"怎么做"用 Skill，"连什么"用 MCP，"打包的能力"用插件**。

普通人的上手顺序应该是：先玩明白内置能力和插件（这一步就够你做大部分事了），等你有重复性的工作流想固化了，再去碰 Skill；等你需要让 Codex 读外部数据了，再去研究 MCP。判断标准很简单：普通写代码不一定需要 MCP；需要让 Codex 访问外部工具或外部数据时才考虑 MCP。

## 53. 怎么找到并安装一个好用的插件

插件不是越多越好，按需装才对。

Codex 里有个插件页，打开能看到所有可用的插件，分类很清楚。装的方式很简单，找到你想要的，点 Add to Codex 或者 Install，按提示走完授权就行。常见的插件有几大类：浏览器类（Chrome、内置浏览器）、办公类（Documents、Presentations、Spreadsheets）、开发类（GitHub、Computer Use）、内容创作类（HyperFrames、Remotion）。

新手最容易犯的错是"看到就装"。刚开始只用官方已经内置好的插件，不要急着装一堆，后续熟练了再慢慢添加。装太多你用不上的插件，只会让界面变乱、增加上下文消耗（每个插件都占一点），还可能引入兼容性问题。

正确的做法是按需装。你明确知道"我要让 Codex 做 PPT"，那就去装 Presentations；你明确知道"我要让 Codex 读我的 GitHub"，那就装 GitHub。没有明确需求的时候，别为了"以后可能用得上"而装。

还有一个判断标准：先想清楚你要做什么事，再去找对应的插件。而不是反过来——看到一堆插件不知道装哪个。等你明确知道"我要让 Codex 连接某个工具"，再去插件页找。这种"需求驱动"的安装方式，能让你装的每个插件都用得上。

装完插件之后，调用方式是在对话框里用 `@` 命令。比如 `@Chrome` 调用 Chrome 插件，`@Computer` 调用电脑操控。打一个 `@` 就能看到所有已安装的插件列表，选你要用的就行。

## 54. 自己怎么创建一个 Skill

创建 Skill 不需要懂技术，会用自然语言说话就行。

最简单的创建方式是这样的：你先让 Codex 完整跑一遍你想固化的流程，跑出来的结果你满意，然后跟 Codex 说"把这个流程变成一个 Skill"。Codex 会自动把你刚才的流程整理成一个 SKILL.md 文件保存下来，下次你就能快速调用。

更直接的方式是用 skill-creator。在对话框里输入 `$skill-creator`，然后像跟同事说话一样，用自然语言描述你的流程：第一步做什么、第二步做什么、最后输出什么。Codex 会帮你把这个描述整理成结构化的 Skill 文件。这种方式不需要知道技术细节，直接把自己的流程讲出来就行，不需要说得严丝合缝，Codex 能理解，理解不了的地方再交互几轮。

很多人觉得 Skill 这个词听起来很高级、很复杂，不敢碰。其实 Skill 就是个 markdown 文件，里面写清楚"什么场景用、目标是什么、按什么步骤做、输出什么格式"。Skill 的价值在于：把只在"你脑子里"的经验，变成"Codex 能反复执行"的标准流程。

做一件事的流程很重要——先做 A 再做 B，还是先做 B 再做 A，结果差很多。做饭的时候，炒完菜立刻擦灶台，趁油没凝结，比吃完饭再收拾省力得多；做饭的过程中，趁着间隙顺带把后面不再用的餐具清洗了，会减轻饭后的清理负担。Skill 就是把你做事的"流程"沉淀下来，让 Codex 按你这个高效流程来，而不是 Codex 自己随便发挥。

什么时候该做 Skill？标准很明确：**只做一次的任务用提示词，经常重复做的任务用 Skill**。如果你每个月都要做一次月度汇报，每次都重复解释一遍流程，那就把流程做成 Skill；如果只是临时做一次的事，直接说就行，不用专门做 Skill。

## 55. Skill 怎么写才能被自动触发

很多人做了 Skill，发现 Skill 从来没有被自动调用过，每次都得手动 `$skill-name` 去喊，觉得 Skill 不好用。问题出在 Skill 的描述上。

Skill 能不能被自动触发，关键在 Skill 的 description（描述）字段。Codex 决定要不要用某个 Skill，是看你的任务描述和 Skill 的描述匹不匹配。描述写得模糊，永远匹配不上；描述写得清楚，Codex 就能在你说话的时候自动判断"这个任务该用那个 Skill"。

一句话总结：**Codex 仅根据描述文本来匹配你的任务，以便隐式调用。把关键用例和触发词放在前面；模糊的描述永远不会触发。**

什么样的描述是好的？举个对比。坏的描述："这是一个数据分析的 Skill。"——太模糊，Codex 不知道什么场景该用。好的描述："通过读取飞书多维表格里的回款数据，按产品和渠道分类汇总，标记异常值，输出一份带图表的 Excel 周报。每当我要求'回款对账'或'周报'时触发。"——场景、动作、输出、触发词都写清楚了。

写描述有几个要点。先把最常见的触发场景放开头，让 Codex 一眼看到这个 Skill 是干什么的；再把触发词写清楚，明确说"每当我说 X 或 Y 时触发"；输入什么、输出什么也要交代明白，Codex 才知道要读什么、给什么。还有一点容易忽略：避免笼统的词。"数据分析""内容创作"这种太宽泛，要具体到"飞书回款对账""公众号文章生成"这种窄场景。

自动匹配之外，还有两种方式可以兜底。一种是显式调用，在对话里输入 `$skill-name` 直接把 Skill 喊出来，不依赖自动匹配；另一种是 `@` 提及，和引用文件一个语法。这两种方式确保即使自动匹配没生效，你也能手动调用。但前提还是描述写清楚，不然你连该喊哪个 Skill 都不知道。

## 56. MCP 什么时候才需要碰

普通人大部分场景都不需要碰 MCP。但有几个特定场景，MCP 是绕不过去的。

先说什么时候不需要。如果你只是让 Codex 处理本地文件、做网页、做 PPT、剪视频、分析 PDF，这些用内置能力和插件就够了，不用 MCP。MCP 没有"用了更高级"这一说，它就是个"特定场景才需要"的工具。

什么时候需要？当你要让 Codex 读外面的数据或者用外面的工具的时候。最常见的是几类活：比如你要让 Codex 读你的 GitHub 仓库、创建分支、发起 PR、评论 issue，用 GitHub MCP，Codex 就能直接操作，不用你手动搬运；再比如你要让 Codex 读 Notion、飞书、Linear 里的文档和任务，对应接上 MCP，Codex 就能直接读到，省得你复制粘贴；还有一类是让 Codex 读某个数据库、内部 API 或某个 SaaS 工具的数据，只要对应服务有 MCP，接上就能用。

MCP 干的事就是"给 Codex 接外部工具的插座"。生活化的类比：Codex 是会干活的人，MCP 是给 Codex 接工具的插座，MCP Server 是工具箱，Tool 是具体工具。你要让 Codex 干的活只在 Codex 自己脑子里和本地文件里，不用接插座；你要让 Codex 干的活涉及外面的数据和工具，就得接插座。

如果判断下来你确实需要 MCP，配置不算难。Codex 里有 MCP 管理界面，按提示添加 MCP Server，填好启动命令和凭证就行。常用的 MCP Server 都有现成的，比如 GitHub、Notion、飞书、各种数据库。装完之后，Codex 在对话里就能直接调用这些外部工具。

记住一条：MCP 是"按需启用"的东西，不是"装了显得专业"的东西。没需求别碰，有需求再研究。

## 57. AGENTS.md 到底要不要写

要写，而且这是把 Codex 用得稳的关键配置之一。

AGENTS.md 是写给 Codex 看的项目规则说明书。每次你在这个项目里开新对话，Codex 都会先读这个文件，知道你是谁、项目是什么、有什么规矩、什么不能动。AGENTS.md 是 Codex 中最被低估的文件。

不写 AGENTS.md 会怎样？每次开新对话，Codex 都"重新认识"你的项目，每次给你的答案都不一样，每次都得你重新解释一遍背景。"没有 AGENTS.md"是让 Codex 只发挥 3% 潜力的坏习惯之首。

写 AGENTS.md 不难，也不用从头写。聪明的做法：用简单的话告诉 Codex 你的项目目标，让 Codex 帮你起草一个 AGENTS.md。你会得到一个结构完整的文件，然后再根据自己的需要修改。这样比从头写快得多、完整得多、不容易遗漏。

一个好的 AGENTS.md 应该包含什么？大致四块内容。开头是背景，一段话讲清你是谁、为什么做这个项目；接着是目标，一段话描述你要的最终状态；然后是约束条件，列成清单，写明硬性规则，比如用什么技术栈、不能用什么、安全边界、输出格式；最后是工作习惯，也是清单，比如"改代码前先确认计划""不要顺手重构""把失败经验记录下来"。

放哪里？项目根目录放一个 AGENTS.md，整个项目通用；子目录放 AGENTS.md，那个子模块的专属规则；用户级（`~/.codex/AGENTS.md`）放一个，是你所有项目通用的个人规则，比如全局禁止某些危险命令。这几层叠加生效。

对普通人来说，最小的 AGENTS.md 只要写清楚三件事就够用：这个项目是做什么的、用什么语言/风格回答你、哪些东西绝对不能动。写一次，后续每次开对话都省事。再往里的高阶玩法，放到 Q68「AGENTS.md 有哪些高价值的进阶套路」里专门讲。

## 58. 任务干预和任务排队怎么用

这两个功能是让你在任务跑的过程中保持控制权的关键，被叫做"人机合一的掌控感"。

任务干预（Steering）是中途打断 Codex。当你发现 Codex 跑偏了、要撞南墙了，不用等 Codex 跑完，直接开口打断 Codex，给 Codex 新方向。比如 Codex 正在审查一个网页，你看着侧边栏发现某个元素间距不对，直接说"这两个元素间距看着不对，调小一点"。Codex 会立刻收到你的指令，调整当前的工作。

任务排队（Queuing）是给 Codex 排后面的活。Codex 正在干 A，你想让 Codex 干完 A 接着干 B，不用等 A 完成再说，现在就可以告诉 Codex"等这个干完之后，把预览链接发到 Slack"。Codex 会把 B 排在 A 后面，A 一完成自动接 B。

两者的区别很清楚：**"干预"是改变 Codex 眼下正在做的事，"排队"是安排 Codex 接下来要做的事。**

这两个功能为什么重要？因为不掌握这两个功能，你用 Codex 的方式就是"丢任务→等→看结果→再丢任务"，每一步都得等 Codex 完成才能做下一步，效率很低。掌握这两个功能之后，你可以在 Codex 跑的过程中持续介入、持续指挥，像一个真正在协作的同事，而不是一个你扔了任务就够的黑盒。

什么时候用干预？看到 Codex 方向错了立刻用，别等。什么时候用排队？你脑子里已经想好下一步了，但当前这一步还没完，用排队把下一步排上。

还有个小技巧：默认情况下，你在 Codex 跑的时候发新指令，新指令是排队的（按顺序执行）。如果你想插队，让 Codex 立刻处理新指令而不是等当前任务完成，要明确用"引导"功能，告诉 Codex"这个优先"。

## 59. 自动化任务适合干什么

自动化是把"需要定期做的事"交给 Codex 自己跑，相当于给项目请一个 AI 值班工程师。

适合做成自动化的，是那些有固定节奏、有明确流程、不需要你每次重新想的事。最典型的是定期汇总类——每天早上让 Codex 整理一下项目状态、每周生成一次周报、每天从某个数据源拉数据做汇总，有人就每天让 Codex 给自己准备一份 AI 行业日报，全网值得关注的资讯第一时间就能看到。还有定期检查类，比如每周检查一次仓库有没有问题、每天查一下回款数据有没有异常、定期监控某个网页或邮箱有没有更新。另外续跑类也很实用，半小时后继续当前线程、每天定时回到同一个对话接着推进工作，对话流自动化会按设定的时间表定期回到同一个对话里继续干活，甚至能根据情况自己调整查岗的频率。

设置自动化不难。跑通一个复杂任务之后，直接告诉 Codex"每天早上 9 点自动执行"或者"每周一检查一次"，Codex 会自动创建自动化任务。

但有一个特别重要的提醒：**自动化任务的模型选择器不会继承你活跃对话的设置**。新建的自动化任务会使用面板的默认值，那个默认值可能比你实际想用的模型更慢更便宜。如果你不手动指定模型，原本 7 分钟的任务可能突然跑 40 分钟。所以创建自动化的时候，一定要显式指定用什么模型、什么推理强度。

自动化是把双刃剑。用得好，自动化让你睡觉的时候活也在干；用得不好，自动化在你看不见的地方跑偏、烧额度、产出垃圾。设置的时候把流程写清楚、把模型指定对、定期检查 Codex 的产出，才能真省心。

## 60. 操控电脑是怎么回事，安全吗

操控电脑（Computer Use）是 Codex 最强、也最需要谨慎的功能。这是 Codex 最被夸的能力，但同时要非常小心。

Computer Use 能干什么？打开你电脑上的应用、点击按钮、读取页面、填写表单——像一个人坐在你电脑前帮你操作。Codex 能操作 Xcode、飞书、Figma、系统设置、iOS 模拟器，任何你装了并且授权的桌面应用，Codex 都能碰。有人用 Computer Use 从零写了一份法律检索报告，自己只完成了登录几个网站的操作，搜索、探究网站结构、点击、整理资料全是 Codex 一次性完成。

为什么这么强？因为 Codex"看"得到你的屏幕。Computer Use 的工作方式是：看屏幕 → 决定点哪里 → 点击 → 等应用响应 → 再看。每一步都基于 Codex 看到的画面，所以 Codex 能操作任何有图形界面的应用，不需要那个应用提供 API。

但这里有个安全问题必须讲清楚。开了 Computer Use，相当于把你电脑钥匙交了一部分给 Codex。Codex 能碰你授权的所有桌面应用，包括微信、邮箱、浏览器、付费软件、公司工具、私人资料。涉及付款、账号设置、删除数据的操作，最好别让 Codex 碰；不用的敏感应用先关掉，不要让 Codex 接触到不相关的东西；每次只给 Codex 一个描述清楚的任务。

新手用 Computer Use 的最低安全标准：第一次用，只让 Codex 操作无风险的应用，特别记住不要让 Codex 操作社媒账号和微信。等你对 Codex 的行为模式有信心了，再逐步放开。

还有三种"操控"方式的区别，普通人最容易搞混。**Computer Use** 权限最大，能碰桌面上所有授权应用，但最慢。**Chrome 扩展** 权限中等，进入你已登录的浏览器，带着你的账号和 Cookie，网站会把 Codex 的操作当成你本人的操作。**应用内浏览器** 权限最小，完全隔离，不带你的任何账号信息，最适合本地开发调试。

选择原则：桌面应用相关的事用 Computer Use，需要你登录状态的网页任务用 Chrome 扩展，本地开发和公开页面调试用应用内浏览器。涉及付款、发布、提交、删除的操作，不管用哪种方式，都务必你来确认，不要让 Codex 自动执行。

## 61. 三种浏览器方式到底用哪个

三种方式有重叠，新手特别容易搞混。这里把选择逻辑讲清楚。

应用内浏览器——最隔离、最安全。应用内浏览器是住在 Codex 里的一个全新浏览器，没有你的任何 Cookie、账号、扩展、历史记录，完全干净。适合的场景：打开本地开发服务器（localhost）、调试前端样式、检查不同屏幕尺寸的布局、在页面上直接标注反馈。应用内浏览器做不了需要登录的页面，但本地开发和公开页面调试最合适。

Chrome 扩展——带着你的登录状态。Chrome 扩展不会新开浏览器，Chrome 扩展会直接进入你已经登录的 Chrome，带着你的账号、Cookie、已打开的标签页。适合的场景：需要你登录状态才能进行的任务、操作多个标签页、读取和对比不同页面的信息。典型例子：你已经在浏览器登了 Vercel，让 Codex 去 Vercel Dashboard 看部署状态；你登了飞书云文档，让 Codex 去读某个文档。网站会把 Codex 的操作当成你本人在操作，所以涉及发送、提交、付款的操作，务必你来确认。

Computer Use 操作浏览器——最重、最慢、最广。Codex 用 Computer Use 操作浏览器的时候，看到的是像素和坐标，像在屏幕上盲点，不理解网页结构。适合的场景：必须操作桌面应用、需要在多个应用之间切换、完成某个"最后一步"（比如上传文件、确认弹窗）。对比很清楚：任务全程在浏览器里 → 用 Chrome 扩展，不要用 Computer Use；需要操作桌面应用 → 只能用 Computer Use。

简单记忆：**权限从小到大、隔离从强到弱：应用内浏览器 → Chrome 扩展 → Computer Use**。能用小权限完成的，别用大权限。本地调试用应用内浏览器，已登录的网页任务用 Chrome 扩展，桌面应用相关的事才动用 Computer Use。

还有一个工具叫 Appshots，Appshots 算不上第四种方式，Appshots 是你"指"给 Codex 看的手段。在 Mac 上快速按两下 Command 键（左 CMD + 右 CMD），截取当前最前面的窗口，Codex 会收到截图和页面文字。然后你说"这个报错是什么意思"或者"这个按钮我想移到右边，怎么改"。Appshots 是你用来"指"给 Codex 看的手指，Computer Use / Chrome / 浏览器是 Codex 用来"动手"的工具。

不过上面三种方式都是"看屏幕操作"。有一类网站特别难搞——拼多多后台、电商管理后台、有反爬检测的 SaaS 系统，你用 Computer Use 去操作，会被网站识别为自动化操作直接拒绝访问。这时候就该换第四条路：Playwright。Playwright 可以理解成"让代码直接操控浏览器内核"，它不靠看屏幕、模拟键鼠，而是直接通过 Chrome 的底层协议去找按钮、填表单、点页面，行为更接近真人。配合"复用你已经登录过的浏览器"这一招，能绕过大部分登录态检测。分工原则很清楚：开放的网站（Google 搜索、文档编辑、X 发帖、Vercel 这类），用 Computer Use 或 Chrome 扩展就够；有反爬保护的后台系统（电商后台、有验证码的页面、检测 WebDriver 的网站），用 Playwright 更稳。

有人用这套思路做过真实的电商批量上架：表格里几千行商品数据，要批量填进拼多多后台。Computer Use 直接被拒，换 Playwright 加复用已登录浏览器，整条链路跑通——读 Excel、分析字段、整理成上架表、自动填写、截图暂停让人确认、批量运行、记录成功失败日志。最后把流程沉淀成 Skill，从"一次性帮我填表"升级成"公司可复用的自动化流程"。这条偏进阶，新手先用三种基础方式，等碰到反爬墙了再研究 Playwright。同样要提醒：Playwright 也是有风险的操作，涉及支付、提交、删除的步骤，务必你确认了再让 Codex 执行。

## 62. Worktree 模式什么时候该用

Worktree 是给同一个项目额外开一个独立工作副本，相当于给 Codex 一本草稿本，让 Codex 大胆改而不影响你的主项目。

为什么需要 Worktree？因为本地模式（Local）下，Codex 直接在你当前的项目目录里改，每次改动都影响实际的工作文件。小修小补没问题，但如果 Codex 要大胆尝试、要重构、要试一个不确定的方案，直接在主项目上改风险太高——改坏了你的主项目就乱了。

Worktree 模式解决了这个问题。Worktree 在 Git worktree（和同一个仓库绑定的独立工作目录）里隔离改动，Codex 在单独的分支里工作，不影响你的主分支。如果跑错了，直接删除 worktree，零风险，你的主项目纹丝不动。

经验法则是：**Worktree 用于重要工作，Local 用于小修小补，Cloud 用于长时间运行的自动化任务。三个信任级别，按任务选择。**

什么时候该用 Worktree？凡是牵连面广、结果不确定的活都建议开。比如让 Codex 做比较大的改动、试一个不确定的方案，或者要重构、加新功能；再比如你想同时做多任务、对比多个方案，每个方案各开一个 worktree 互不干扰。说到底，只要你希望 Codex"放心大胆地试"、最坏了能直接丢掉，就用 Worktree。

什么时候用 Local 就行？改个文案、调个颜色、修个小 bug，这种改动小、可控、你盯着就能完成的，用 Local 最快最直接，没必要专门开 worktree。

Worktree 还有一个隐形好处：比较方案。你可以对同一个任务开两个 worktree，让 Codex 用两种不同的思路分别跑，跑完对比哪个好，留下好的，丢掉差的。这种"并行试错"在 Local 模式下做不到，因为两个改动会互相覆盖。

普通人刚开始用 Codex，可能觉得 Worktree 听起来很高级、很复杂。其实 Worktree 就是个"草稿本"概念，用了 Git 的人会觉得很自然。就算你不用 Git，把 Worktree 理解成"给 Codex 一个独立的副本去改，改好了再并回主项目"就行。任何稍微大点的改动，都建议用 Worktree，这是最低成本的安全保障。

## 63. 多方案预览到底看哪个

这是 Codex 一个挺有用的功能：让 Codex 针对同一个任务，一次给你几个不同的实现方案，你挑一个。

新版 Codex 引入了一个预览系统，针对每个任务会生成 2 到 4 个不同的实现选项。每个方案可能用不同的技术选型、不同的结构、不同的取舍。你可以对比之后选一个最合适的，而不是 Codex 给你什么你就接受什么。

看哪个？几个判断维度可以参考。先看哪个更符合你项目的现有风格——如果你的项目用的是 A 技术栈，Codex 给的方案里有一个用 A、一个用 B，选 A 的那个，集成进去最顺。再看改动范围，同样的功能，一个改 5 个文件、一个改 15 个文件，优先选改动小的，引入风险低。另外依赖越少越好，一个不引入新依赖、一个要装新库，选前者，项目更干净。最关键的一条：选你看得懂的，看不懂的方案后续维护不了。

但多方案预览也有坑。一是别陷入选择困难。Codex 给你 4 个方案，你比来比去半小时选不出来，这比直接用第一个方案还浪费时间；多数情况下，第一个或第二个方案就够用，纠结边际效益不大。

还有一点，别都试一遍。有人想"我把 4 个方案都跑了看哪个好"，听着合理，实际很费额度。多方案预览的意义是让你在执行前对比思路，而不是让你每个都实际执行一遍。

正确的用法是：看几个方案的思路描述和关键差异，选一个你认为最合适的，执行 Codex。如果执行结果不满意，再换另一个方案。这种"先看思路选一个，不行再换"的方式，比"每个都跑一遍"高效得多。

## 64. Goal 模式和普通任务有什么区别

Goal 模式是给一个任务画一条明确的终点线，让 Codex 在一段时间内持续朝终点冲刺。

普通任务是这样的：你说一个需求，Codex 干完就结束，干没干成、干到什么程度，由 Codex 的判断决定。Goal 模式不一样，你除了给需求，还给一个明确的"成功标准"，Codex 会持续朝这个标准努力，直到达成或者明确达不到。

对比很清楚。一个糟糕的 Goal："把这个 Markdown 文件里的计划实现一下。"——没有终点，Codex 什么时候算完成全凭 Codex 自己判断，大概率做到一半就"觉得"完成了。

一个优秀的 Goal："直到所有单元测试全部通过，这个新版本的开发才算完成。"——终点明确，Codex 知道要跑到什么状态才算完，跑到那个状态之前都不算成功。

Goal 模式干的事是把"持续执行"和"验证器"结合在一起。你来定义想要的结果、何时停止的条件、以及判断 Codex 有没有离终点更近的信号。好用的验证器包括：一套完整的测试用例、一项基准性能测试、一个能稳定复现的 bug、一个验证矩阵、一个必须始终跑通的端到端工作流。

适合用 Goal 模式的场景：代码迁移（从一种语言迁到另一种语言，目标是所有测试通过）、bug 修复（目标是某个 bug 不再复现）、性能优化（目标是某个基准跑得更快）、重构（目标是功能不变但结构更好，所有测试还得过）。

不适合 Goal 模式的场景：需求模糊的事、没有客观验收标准的事、纯创意性的事。这些没有明确的"终点线"，Goal 模式用不上。

有一句话特别值得记住：**有野心固然重要，但没有验证机制的野心，就只是在许愿而已。** Goal 模式的前提是你能给一个可验证的终点。给得了，Goal 模式能帮你把复杂任务推进到位；给不了，用普通任务模式更合适。

## 65. 持久对话流和置顶怎么用

持久对话流（durable threads）是 Codex 里一个被低估的功能。

普通对话是"聊完即焚"的：你开一个对话，聊完任务完成，关掉，下次开新的又得重新解释背景。持久对话流不一样，持久对话流能长时间运行、跨多次使用保留工作上下文。Codex 记得你之前做过的决定、你的偏好、当前的进度，下次你回到这个对话，Codex 还在那个状态。

为什么要用持久对话流？因为很多工作是持续推进的，不是一次性完事。你做一个产品，可能要持续几周；你做内容创作，可能每天都有新想法往里加；你监控某些数据，可能要持续跟进。这些场景下，每次开新对话重新解释背景太累，持久对话流让你"接着上次的继续"。

推荐的用法是"置顶"。把那些需要反复推进的持久对话置顶，这些对话就随叫随到。比如一个专属的"幕僚长"对话流，帮 Codex 处理日常杂务；一个负责产品发布的对话流；一个负责审查文档的对话流；一个专门盯外部数据的监控对话流。这些对话跟聊完即焚的闲聊框不一样，这些对话是持久的工作空间。

置顶之后还能用快捷键。按下 Command-1 到 Command-9，你能瞬间穿越回这些保存好的专属对话流里继续工作。这种"一键回到某个工作上下文"的体验，比每次翻历史记录找对话快得多。

持久对话流配合共享记忆，威力更大。把这些持久的对话"锚定"在一个知识库里——说白了就是建一个存放纯文本文件的文件夹。在最外层放一个 AGENTS.md，给 Codex 定规矩：当 Codex 了解到新情况时，如何更新这个知识库。这样重要的上下文就不会锁死在某一次聊天里，而是写下来，放在下一个对话流能立刻接手的地方。

普通人的起步用法：为你最常做的几类事，各开一个持久对话流，置顶起来。比如一个"日常杂事"、一个"我的项目"、一个"学习笔记"。下次有事，直接去对应的对话流里说，不用每次重新解释你是谁、在做什么。

## 66. 共享记忆怎么搭最稳

共享记忆是让 Codex 跨多次对话保留上下文的机制，有一种最稳的搭法。

为什么要搭共享记忆？因为 Codex 默认只有短期记忆——Codex 在本次对话里学到的东西，明天就忘。每次新对话，Codex 都"重新认识"你和你的项目。如果你反复做类似的事，每次都要重新解释，效率很低。共享记忆就是把这些重复需要的上下文，存在对话之外的地方，让未来的工作能基于这些信息继续推进。

最稳的搭法，是用一个纯文本文件夹当知识库。说白了就是建一个文件夹，里面放 markdown 文件。示例结构：vault 文件夹下分 TODO、people、projects、agent、notes 几个子目录。在最外层放一个 AGENTS.md，给 Codex 定规矩：当 Codex 了解到关于人员、项目、决策、待办的新情况时，如何更新这个知识库。

为什么用纯文本文件夹？纯文本文件夹简单直白，方便你随时查看、修改、移动，能保存很久。团队可以把这个文件夹放在任何云盘里——Git、Dropbox、Google Drive 都行。

还有一点特别重要：**别死板地照抄某一种知识库结构**。你需要做的是"教"Codex：那些持久的上下文该放在哪、哪些上下文需要保留、什么时候不要瞎折腾去改文件。每个人的工作方式不一样，知识库结构应该贴合你的实际，而不是抄一个"标准模板"。

一份实用的 AGENTS.md 指南里可以写：把这个文件夹当作长期工作记忆区；把笔记整理得有条理，别搞得到处都是碎片记录；准确地把待办、人员、项目、每日总结分类放好；把做过的决定、遇到的卡点、负责人、日期、有用的链接保存下来；如果没有实质性的新进展，不要随意修改文件。

Codex 自己也提供官方的记忆功能，在设置 > 个性化 > 记忆里。这个功能就像系统自带的本地记事本，用来记住你的个人偏好、常用的工作流、经常踩的坑。但这个功能是用来辅助你清晰写下来的上下文的，不是取代它。

等你用得更深，想让记忆系统更聪明，可以往"带索引和语义检索"的方向升级。社区里有进阶玩家用 Obsidian 当记忆库的外壳，加一层本地数据库做关键词检索，再加一层语义检索（按意思找东西，而不是只按原词匹配），配合 Git 做版本留痕、closeout 脚本做自动收尾、audit 脚本做定期体检。这种架构能把记忆库从"一堆文本文件"升级成"可搜索、可审计、能自运转的系统"。不过这套偏极客，搭建成本高，对普通用户来说，先把基础的纯文本知识库用好，等你规矩多得管不过来了，再考虑往这个方向升级。

普通人起步：建一个文件夹，放一个 AGENTS.md，写清楚你最常需要 Codex 记住的几件事。然后用起来，缺什么补什么，慢慢长出自己的知识库。不用一开始就追求完美结构。

## 67. 语音输入为什么值得用

语音输入听起来像个小功能，但语音输入解决了一个真问题：把脑子里最原始、最粗糙的念头先捕捉下来。

打字的问题是什么？你脑子里有个模糊的想法，但还没完全成型。要把它打成文字，你会本能地"字斟句酌"——边想边组织语言、边过滤边精简。这个组织的过程会丢失很多信息，原本丰富的上下文被你压缩成几句干巴巴的话，AI 拿到的信息量变少了。

语音输入绕过了这个过滤。你对着 Codex"碎碎念"两三分钟，把脑子里的想法一股脑倒出来，效果出奇的好。举个例子："我记得有个叫 Ben 的人在 Slack 上提过这事儿，细节我忘了，你去帮我找找看。"对于一个会自己搜索、收集上下文并向你汇报的 agent 来说，这种模糊的话就足够 Codex 干活了。

录音转写也是同样的道理。一份未经修饰的会议记录，或者一段口述的计划草案，往往比一份简短的总结更有价值。因为那些粗糙的记录里，保留了你犹豫的语气、强调的重点、还没讲完的灵光一现。这些细节在打字的时候会被你"精简掉"，但恰恰是 AI 理解你真实意图的关键信号。

语音输入配合任务干预，威力更大。一个任务正在跑，你看着侧边栏发现某个元素不对，不用停下来打字，直接开口说"这个调小一点""这句文案写错了"。这种"边看边说"的交互，比"打字描述问题"流畅得多。

什么时候用语音？想法还很模糊、没完全成型的时候最该用，把它一股脑倒出来再整理；需求复杂、打字要打很长的时候也划算；任务跑的过程中你想随时插话，开口比打字快得多；还有就是你正走动、不方便打字的时候。

Codex 的语音输入可以设成全局键，打开之后不止在 Codex 内部，可以在电脑上任意一个对话框调用。如果你习惯了别的语音输入法（比如豆包），不切换也行，重点是养成"用嘴说"的习惯，而不是事必打字。

## 68. AGENTS.md 有哪些高价值的进阶套路

Q57 讲了 AGENTS.md 的基础写法。写多了你会发现，有几类规则特别管用，是社区里反复验证过的"高效套路"。这四个套路各有针对性的痛点，可以按需挑着用，不用全上。

第一个套路叫 Confidence Gate（置信度闸门），专治"半成品"。Codex 经常在任务没真正完成时就宣称做完了。你在 AGENTS.md 里写一段"完成定义"，让 Codex 在宣布完成前先自评打分，分数不够就得继续干。模板长这样：

> 宣布任何任务完成之前，先在三个维度上给自己打分（每项 0-10）：1. 测试证据——有没有跑测试，把输出贴出来；2. 行为等价——有没有验证公开接口和行为没变；3. 清理——有没有删掉死代码、没留下注释掉的代码块。三项都达到 8 分以上才算完成。任何一项低于 8 分就继续干，不要来问我。

这个套路对治半成品特别有效。Codex 为了凑够分数，会主动跑测试、主动验证、主动清理，而不是写完核心功能就停。

第二个套路专治"频繁打断"。长任务里 Codex 会时不时停下来问"要不要继续""下一步做什么"，把节奏打断。在 AGENTS.md 里明确写：

> 持续执行直到任务完成，不要中途停下来问我"要不要继续"。遇到必须确认的事情，先按最合理的方式做，把决策记录下来，等我回来统一审。

这条能让长任务从"一小时停三次"变成"连续跑数小时"，省心得多。

第三个套路专治"上下文爆炸"。Codex 跑长任务时，会把命令输出的整页日志塞回上下文，把额度吃光。在 AGENTS.md 里约束命令输出的处理方式：

> 命令输出超过 50 行的，只保留尾部和关键报错行，不要把整页日志塞回上下文。读大文件这种重活，开一个新的子对话去读，主对话只收结论。

有人实测这条规则把额度消耗砍掉了一半左右。

第四个套路有点反直觉：**别让 AGENTS.md 太长**。社区里有人做了跨多个 agent 的对照研究，发现过长的 AGENTS.md 反而降低表现——Codex 要花更多注意力去遵守所有规则，反而抓不住重点。正确做法是把 AGENTS.md 当"入口"，只放最核心的规则和指引；架构细节、实现约定、repo 结构这些，拆成单独的 markdown 文件，从 AGENTS.md 里引用。AGENTS.md 控制在你能一眼扫完的长度，Codex 反而更听话。

第五个套路是进阶玩法：**把习惯拆成多个文件，按场景路由**。一个 AGENTS.md 装不下所有规矩的时候，别硬塞，按场景拆成几个文件，让 Codex 工作前先读对应的那份。

举例，你可以这样拆。一份放全局使用习惯：用什么系统（比如 Windows 默认跑 PowerShell）、文件操作用 `-LiteralPath`、不混用 Bash 的 `&&`、执行命令前先确认项目目录。这些看着碎，但决定了 Codex 给的命令能不能直接在你电脑上跑——很多时候 Codex 不是不会做，是不知道你本机的工作环境。另一份放写作和沟通偏好：中文要直接、不要套话、不要硬升华、未实测的工具必须标注"未实测"，让 Codex 写东西时不会写出那种很标准的 AI 腔。再一份放安全边界：不删用户文件、不自动 commit 或 push、发布前先检查 Token 和密钥、不把私有代码发给公网搜索工具。

这种"多文件路由"的好处是：每个文件短、专注、好维护；Codex 按需读对应那份，不会被无关规则干扰。坏处是搭建成本比单个 AGENTS.md 高，适合你已经用了一阵、规矩越来越多的时候再上。新手先把 Q57 的基础 AGENTS.md 写好，等规则塞不下再来这套。

## 69. 怎么让 Codex 改代码前先讲思路

这是个特别实用的防翻车技巧，比计划模式更轻量，适合"不想走完整 plan 流程、但又不放心让 Codex 直接干"的场景。

正常情况下，你给个任务 Codex 直接开干。Codex 干得很快，等你想介入的时候已经改完了一大半。如果你能在 Codex 动手之前先听听 Codex 的思路，很多翻车都能提前挡住。

怎么让 Codex 先讲思路？最简单的就是直接说："先告诉我你打算怎么做，我确认了你再开始改。"Codex 会返回一份思路描述——打算改哪些文件、用什么方法、分几步。你看完觉得没问题，再说"开始执行"；觉得有问题，直接在思路阶段提出来改。

复杂一点的任务可以用计划模式，打 `/plan`，Codex 会返回一份结构化的计划文档，比单纯讲思路更完整。

更省事的做法是把它写进 AGENTS.md，作为 Codex 的默认行为。比如"对于任何涉及多文件的改动，先告诉我思路，我确认了再执行"。这样不用每次都嘱咐，Codex 默认就先讲思路。

跳过计划阶段是项目出错的最大原因。一句话的误解，可能让 Codex 改了四十个文件。先让 Codex 讲思路，你能一句话纠正，比写完代码再改省事十倍。

什么时候该用这个技巧？凡是任务涉及多文件改动、做法上有多种选择而你又想参与决策、或者错了代价比较大的，都必须先讲思路。

什么时候不用？改个变量名、调个样式、问个简单问题，这种确定性强、错了也好回退的事，直接让 Codex 干就行，先讲思路反而啰嗦。

记住一个原则：**复杂度越高、不可逆性越强，越要让 Codex 先讲思路**。简单可逆的事可以放手让 Codex 干，复杂或者影响面大的事必须先对齐思路。这个习惯能挡住大部分翻车。

## 70. 怎么让 Codex 记住我项目的约定

让 Codex 记住约定，最稳的办法就是写下来，而不是指望 Codex 自己"记忆"。

Codex 默认只有短期记忆，Codex 在本次对话里学到的东西，新对话就忘。很多人抱怨"我跟 Codex 说过好几次别动那个文件，Codex 还是动"，根本原因就是 Codex 真的"忘"了——你上次说的，存在上次的对话里，新对话 Codex 不知道。

让约定真正被记住，要靠几种"写下来"的机制。

最常用的是项目级 AGENTS.md，把项目的硬规矩写在里面，每次开新对话 Codex 都会读。比如"不要修改 config 目录下的任何文件""所有新增组件放在 components 目录""使用我们项目自己的代码风格"。这些写一次，长期生效。

再往上一层是用户级 AGENTS.md（在 `~/.codex/AGENTS.md`），把你个人的通用偏好写在这里，对你所有项目都生效。比如"我偏好简洁的代码""修改前先解释影响""涉及隐私文件先提醒我"。

如果你有一套固定的工作方式，比如"写文档按这个结构来""做代码审查按这个清单查"，那就把这套方式做成 Skill。Skill 是显式的、可调用的，比纯记忆可靠。

Codex 还自带一套记忆功能（设置 > 个性化 > 记忆），算系统级的本地记事本，用来存你的个人偏好、常用工作流、踩过的坑。但这个功能是辅助性的，不能取代你清晰写下来的上下文。

还有一个特别好的建议：**遇到值得记录的经验教训时，告诉 Codex 更新 AGENTS.md 或者项目记忆**。比如 Codex 试了一种方法失败了，你跟 Codex 说"把这次失败的原因记到 AGENTS.md 里，下次别再试这个方法"。系统会越用越聪明，因为 Codex 把这些教训写下来了。

一句话总结：**别指望 Codex"记住"，要让 Codex"读到"**。写在 AGENTS.md、Skill、知识库里的东西，Codex 每次都能读到；只存在对话里的东西，下次就没了。重要约定一定要写下来。

## 71. 哪些小习惯能让结果稳定性翻倍

很多小习惯，单独看都不起眼，叠加起来效果惊人。这里集中讲几个最有效的。

最值得养成的一条：所有复杂任务先开计划模式。计划阶段消耗少，能挡住绝大多数跑偏。养成"但凡复杂点的事先 `/plan`"的习惯，翻车率会大幅下降。

每次跑完别只看 Codex 的文字总结，去 diff 里看 Codex 到底动了什么。看到不对的改动立刻问、立刻回退。这个习惯能把"顺手重构""过度工程化"挡在合并之前。

任务要分清楚边界——一个明确任务一个 thread，不要把多件事塞进一个对话。任务越独立、上下文越干净，Codex 越稳定。

遇到失败先停再想。发一句"先停一下，总结失败原因"，比无脑重试有用得多。让 Codex 给你失败分析，你再决定下一步。

还有几条顺手就能做的：好的结果立刻保存，agent 的输出是一次性的，跑出来不保存就没了，满意的成果马上复制到项目文件夹、提交到 git；任务描述带验收标准，明确说"完成的标准是 XXX，缺一样都不算完"，半成品的概率会大幅降低；任务描述也别忘了带约束，"不要动 X""不要引入新依赖""不要顺手重构"，约束越明确，Codex 越收敛。

最后是两个跟额度和成本有关的小规矩。经常查额度，发现掉得快立刻排查，不用的时候彻底退出 Codex，不让后台任务偷跑；按任务配推理强度，小任务用低档省额度，复杂任务才用高档，开任务前花两秒判断该用哪档。

还有一条特别要紧：遇到值得记录的经验立刻写进 AGENTS.md，让 Codex 下次别再犯同样的错。系统会越用越聪明。

这些习惯没有任何一条是高深的技巧，全是日常操作层面的小规矩。那些把 Codex 用得特别好的人，靠的并不是什么秘密，靠的就是这些小习惯每天都在做。叠加起来，他们的结果稳定性就是比一般人高得多。
