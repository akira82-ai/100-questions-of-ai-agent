# 二、第一次上手与核心机制

## 16. 第一次用 Codex 该从哪一步开始

别急着给 Codex 派活，先做两件事：装好、体检。

下载的渠道只有一条，就是 OpenAI 官方入口。Mac 用户去官网下 dmg，注意分清 Apple Silicon（M 系芯片）和 Intel 版本，选错了装不上。Windows 用户去 Microsoft Store 搜 Codex，发布者认准 OpenAI；Codex 的 Windows 版是 2026 年 3 月才上线的，对 Windows 11 体验最好，Windows 10 也支持。不要从奇奇怪怪的第三方站点下，那种渠道质量参差不齐。

装完第一次打开，会走一个登录流程。三个选项：ChatGPT 账号登录、API Key 登录、第三方中转站。普通人选第一个，用你的 ChatGPT 账号登进去就行，走的是套餐额度，最省心。API Key 是给开发者和自动化场景用的，新手别碰。

登完 Codex 会问你职业身份（Engineer / PM / HR 之类），随便选一个就行，这个只是给 OpenAI 做统计，不影响功能。接着会问要不要开 Sandbox 沙箱模式，一定点 Set up 开启，这样 Codex 跑命令时不会乱动你项目目录之外的东西。

做完这些，别马上丢任务。先随便发一句"你好"，确认 Codex 能正常回复，说明全套配通了。然后再开始真正的项目工作。

第一次接触的最低目标就一条：确认"装对了、登对了、能跑了"。能不能干活先放一边，这一步顺过去，后面才不会埋雷。

## 17. 登录方式选哪个，ChatGPT 账号还是 API Key

普通用户选 ChatGPT 账号登录，别碰 API Key。这是新手最容易选错的岔路口。

ChatGPT 账号登录走的是你的套餐额度，Plus、Pro 这些会员每个月给你一定的使用量，用完了等下个周期恢复。你不用管花了多少 token、多少钱，OpenAI 都帮你算好了。这种模式对普通人是透明的、可预期的，最省心。

API Key 登录走的是 OpenAI Platform 的按量计费，用多少 token 扣多少钱，和充电话费一个道理。好处是没有固定月费、用多少算多少，坏处是你得盯着消耗，跑偏的任务可能瞬间烧掉一笔钱。这种模式适合开发者、自动化场景、CI/CD 流水线，不适合普通人日常用。

一个判断标准：如果你听到"token""按量计费"这两个词还得愣一下想是什么意思，那就老老实实用 ChatGPT 账号登录。等你用熟了、想精细控制成本了，再去研究 API Key 那条路。

还有第三条路，就是通过 CC Switch 这种工具接国产模型，这条路其实也是走 API Key，但用的是国内厂商的额度。这条路对中国用户更便宜，但属于非官方玩法，留到第六章细讲。

## 18. 项目到底是什么，为什么一定要先建项目

项目，就是你电脑上的一个文件夹。

Codex 不是单纯的聊天工具，Codex 需要进入一个具体的"项目目录"才能干活。你新建一个项目的时候，让 Codex 选一个文件夹，选完之后这个文件夹里的所有文件，Codex 都能读、能写、能改。文件夹就是 Codex 的工作边界，也是 Codex 的信任边界。

为什么非要这一步？因为 Codex 的核心能力是"动手"，动手就得有个动手的地方。你让 Codex"帮我改这个网页"，Codex 得知道网页在哪个文件夹、有哪些文件、依赖装在哪。没有一个明确的项目目录，Codex 就像一个被蒙着眼睛扔进陌生办公室的人，不知道从哪下手。

很多人跳过建项目，直接在普通对话框里让 Codex 干活，结果遇到一堆怪事：Codex 说找不到文件、Codex 改的东西不知道存哪了、Codex 跑的命令报莫名其妙的错。这些问题的根子都在"没建项目"。

建项目还有一个好处，是结果可追溯。项目模式下，Codex 生成的所有文件都存在同一个文件夹里，你打开文件夹就能看到全部产物，不会像对话模式那样生成的东西散落在各处、找不到。日常用基本都开项目模式，除非问个临时小问题才用普通对话。

所以正确的打开方式是：先建一个专门的文件夹当工作区，再把每个具体的活当成这个文件夹里的一个项目。

## 19. 项目文件夹该选哪个，有哪些坑

这个选择比你想的重要得多，选错了后面全是麻烦。

先说该选什么。专门新建一个干净的文件夹，专门给 Codex 用，里面再按项目分子文件夹。比如 `D:\AI-Codex-Projects\我的网页项目`，或者 Mac 上的 `~/AI-Codex-Projects/我的网页项目`。路径用英文最稳，中文路径和空格在 Windows 上特别容易出毛病。

再说千万别选什么。C 盘根目录、桌面、下载文件夹、系统目录、重要资料文件夹、中文路径、带空格的路径，这些都不能用。

C 盘根目录和系统目录权限敏感，Codex 跑命令容易撞墙。桌面和下载文件夹太乱，里面什么都有，Codex 分不清哪些是你要处理的、哪些是你的私人文件。重要资料文件夹风险在于，万一 Codex 理解错了任务，把你的重要文件改了或删了，恢复起来很麻烦。中文路径和空格则是技术层面的坑，很多命令行工具对这两种路径处理不好，容易报莫名其妙的错。

还有一个隐形坑：不要选一个正在被别的程序占用的文件夹。比如你正在 VS Code 里编辑这个项目，同时又让 Codex 去改 Codex，两边可能冲突，文件状态会乱。要么关掉编辑器，要么给 Codex 用 worktree 模式（这个后面讲）。

记住一条铁律：建一个干净、专用、英文路径的工作文件夹，把它当作你和 Codex 共同的工作区，所有项目都在这里面发生。

## 20. Sandbox 权限那三档到底怎么选

Sandbox 是 Codex 的安全围栏，分三档，选错要么束缚手脚，要么风险失控。

第一档叫只读，Codex 只能看你项目里的文件，不改东西。适合你只想让 Codex 分析、解释、给出建议，但不想让 Codex 动文件的场景。比如"帮我看看这个项目的架构合不合理"，这种用只读档最安全。

第二档叫项目可读写，Codex 能改你当前项目文件夹里的文件，但出不去这个文件夹。这是日常推荐档位，绝大多数活都用这档。Codex 能读、能写、能跑命令、能装依赖，但所有动作都被限制在你的项目目录里，你电脑其他地方的文件 Codex 碰不到。

第三档叫完全访问，整个电脑 Codex 都能碰。这一档新手不要碰，只有在你非常清楚自己在干什么、并且完全信任当前任务的时候才考虑。开了这档相当于把电脑钥匙全交给 Codex，Codex 要是理解错了任务，能搞出大麻烦。

Codex 里这三档对应的是"请求批准 / 自动审核 / 完全访问权限"三个选项。新手建议保持"请求批准"，意思是任何命令执行前都先问过你，你确认了才执行。这样慢一点，但安全。中间的"自动审核"档，系统会替你判断某些安全操作要不要放行，比"请求批准"省心，又不至于像"完全访问"那样放任。

权限给多大，说到底是"信任 vs 效率"的权衡。权限小，安全但繁琐；权限大，丝滑但风险高。刚开始小权限，等你熟了、信任建立起来了，再逐步放开。一上来就开最大权限是最容易翻车的做法。

## 21. 推理强度那几档有什么区别

推理强度控制的是 Codex 在回答你之前思考多久。档位越高，Codex 想得越深，但越慢、越烧额度。

通常是四档。低档适合改文案、改颜色、回答小问题这种轻活，几秒就完事。中档适合普通网页、简单 bug、日常开发，这是最常用的档位。高档适合多文件修改、复杂 bug、重构这种需要深思考的活。最高档适合很难的问题、架构分析、反复修不好的 bug，Codex 会想很久，但往往能想到你意想不到的解法。

很多人犯的错是：所有任务都开最高推理，觉得"越高越聪明"。这是双重浪费。小任务根本不需要那么深的思考，高档和低档在改个变量名这种事上结果一样，你白等时间。高档消耗的额度远多于低档，你白烧额度。

正确的用法是按任务匹配。改个文字、调个样式，用低档；做个普通功能、修个常规 bug，用中档；只有遇到真正复杂、需要 Codex 深入分析的问题，才上高档。省额度最核心的原则就是这一条：小任务不开最高推理。

如果你拿不准该用哪档，从中档开始最稳。中档是"够用且不浪费"的甜点位，跑完看结果，要么调高要么调低，几次之后就摸清规律了。

## 22. 计划模式是什么，什么时候该开

计划模式是 Codex 最被低估的功能，也是防翻车最重要的一道闸。

正常模式下，你给个任务 Codex 直接开干。计划模式下，Codex 不立即动手，而是先停下来，整理出一份工作计划：打算怎么做、分几步、会动哪些文件、有什么风险，然后等你确认了再开始。

为什么这个模式这么重要？因为 agent 干活快，快到你来不及反应。你一句话丢进去，Codex 可能已经改了十几个文件，等你发现方向错了，已经改完了一大半，回退都麻烦。计划模式就是给你一个"刹车的机会"，在 Codex 真正动手之前，先看看 Codex 的思路对不对。

跳过计划模式是项目出错的最大原因。一句话的误解，可能让 Codex 改了四十个文件，而如果你先看计划，一句话就能纠正。

什么时候该开？所有稍微复杂的任务都先开计划模式。改个变量名、调个颜色这种小事不用，但凡涉及多文件、新功能、重构、修复杂 bug，先开计划模式让 Codex 出方案，你审完再让 Codex 执行。

用法很简单，在对话框里打 `/plan`，或者描述任务时说"先给我一个计划"。Codex 会返回一份分步方案，你看完觉得没问题，再说"开始执行"。觉得有问题，直接在计划阶段提出来改，比写完代码再改省事十倍。

## 23. 侧边栏是干什么的，为什么这么重要

侧边栏是 Codex 的"作业检查区"，也是 Codex 和普通聊天工具拉开差距的关键。

普通对话里，AI 给你一段文字或者一段代码，你看完就完了。Codex 不一样，Codex 干活的时候，右边那个侧边栏会实时展示 Codex 正在做什么、生成了什么、改了哪里。这个侧边栏就是 Codex 的"工作现场"。

侧边栏能展示的东西很多：Codex 生成的文件、搜索来源、网页预览、图片预览、PDF 预览、内置浏览器、代码改动对比。中间告诉你"Codex 做了什么"，右边让你看"Codex 产出了什么"。

侧边栏最有价值的几个用法里，最该盯紧的是看代码改动。Codex 改完文件后，侧边栏会显示一个 diff，绿色是新增、红色是删除，你能清楚看到 Codex 到底动了哪些行。除此之外，Codex 做了个网页，侧边栏直接打开就能看效果，省得你自己去启动服务器。更妙的是，网页预览时你可以直接在页面上圈出某个按钮，说"这个颜色改深一点"，Codex 能精确收到你的反馈，只改你指的那一处。

很多人刚用 Codex 的时候，眼睛只盯着中间的对话框，把 Codex 当聊天工具用，完全忽略侧边栏。这是巨大的浪费。真正会用的人，眼睛是盯着侧边栏的，因为那才是 Codex 干活的现场。中间的对话只是你指挥 Codex 的渠道，右边的侧边栏才是 Codex 交作业的地方。

## 24. Diff 是什么，为什么每次都要看

Diff 就是代码改动的对比，绿色是新增、红色是删除。每次 Codex 完成任务后，第一件事就是看 diff。

为什么这么强调看 diff？因为 Codex 改得很快，快到你不知道 Codex 到底改了什么。如果只看 Codex 给你写的文字总结——"我修改了登录逻辑，优化了样式"——你根本不知道具体动了哪些地方。文字总结可以糊弄，diff 不会骗人。

看 diff 能帮你抓出几种问题。最常见的，Codex 改了不该改的地方——你以为让 Codex改登录页，结果它顺手把首页的样式也动了，这种"顺手改"在 diff 里一眼就能看出来。还有就是删了不该删的东西，有些代码 Codex 觉得"没用"就删了，但其实是你的业务逻辑；再就是莫名其妙多了个新依赖、多了个新文件，这些在 diff 里都很显眼。

有一条硬规矩值得反复说：**每次 Codex 完成任务后，先看 diff，再决定要不要接受**。不要只看 Codex 的文字总结就放心。看到不理解的改动，先停下来问 Codex"这里为什么这么改"，而不是默认接受。

Diff 还能用来迭代。你在 diff 某一行的旁边留个评论，说"这里改得不对，换种方式"，Codex 能收到你的反馈，只针对那一处再改。这比把整个任务重新描述一遍高效得多。

养成"先看 diff"的习惯，是用 Codex 用得稳的最低门槛。这个习惯能把绝大多数翻车挡在合并之前。

## 25. Thread 是什么，一个项目里能开几个

Thread 就是一个项目里的一个具体任务对话。一个项目能开很多个 thread。

为什么要分 thread？因为一个项目里你要干的事通常不止一件。做网页的过程中，你可能一边让 Codex 修首页的 bug，一边让 Codex 加个新功能，一边让 Codex 改样式。这些是三个独立的任务，塞进一个对话里会乱——Codex 分不清你新说的话是接着上一个任务，还是开新任务，上下文会搅在一起。

正确做法是一个明确的任务开一个 thread。修 bug 是一个 thread，加功能是另一个 thread，改样式又是一个。每个 thread 专心干一件事，上下文干净，结果也清晰。项目像一个公司，thread 像公司里的员工，每个员工负责一摊事。

thread 之间可以并行。你布置完一个任务，不用等 Codex 完成，可以新建一个对话直接布置另一个。多个任务只要不是改同一个文件，就不会互相干扰。每个 thread 的进度是独立的，你能分别看到哪个完成了、哪个还在跑。

完成的 thread 可以归档，归档不是删除代码，是把任务列表收起来，保持界面清爽。归档了还能找回来，在设置里取消归档就能还原。如果一个 thread 跑得不好，想换个思路重来，可以 fork 一份去试新方案，原来的还在。

记住一条：不要把所有事情塞进同一个 thread。任务越清楚、thread 越独立，Codex 越稳定。

## 26. 怎么知道一个任务还要跑多久

这个问题没有精确答案，但有几个信号可以判断。

Codex 跑任务的时候，左边对话列表里那个任务的状态会变化。如果是转圈的图标，说明 Codex 还在干，没结束。如果变成了一个静止的蓝色点，说明任务已经跑完了，你可以去验收。

至于具体还要多久，Codex 不会给你一个倒计时。一个任务快可能几十秒，慢可能十几分钟，差距很大。影响因素有几个：任务本身的复杂度、Codex 要不要装依赖、Codex 要跑几轮试错、当前模型版本的状态。改个变量名这种事，几秒就完；让 Codex 从零做一个完整应用，跑半小时也不奇怪。

如果你觉得 Codex 跑得太久，可以中途去看 Codex 在干什么。点开那个 thread，能看到 Codex 当前正在执行的动作——在读文件、在跑命令、在等什么。如果 Codex 卡在某个动作上很久不动，可能是遇到了问题；如果 Codex 在正常地一步步推进，那就是任务本身需要这么久。

有一种常见现象要警惕：Codex 跑着跑着会进入"原地打转"的状态，反复尝试同一种解法但一直失败。这种情况下等再久也没用，应该手动叫停，换个思路重来。怎么判断 Codex 是不是在打转，看 Codex 的动作日志，如果连续几轮都在做差不多的事，基本就是在绕圈子了。

## 27. 任务跑着跑着没动静了怎么办

先别慌，分情况处理。

最常见的情况，Codex 真的还在跑，只是慢。有些任务本来就耗时，比如装一堆依赖、跑完整的测试套件、处理大文件。点进 thread 看 Codex 的动作日志，如果还在正常推进，那就是等的事，别打断。

还有一种情况，Codex 在等你批准什么。Codex 在"请求批准"模式下，遇到敏感操作会停下来等你确认，比如要装个新依赖、要改某个文件、要跑某条命令。这时候 Codex 是在问你要不要让 Codex继续，没卡住。检查一下界面有没有待批准的提示，处理了 Codex 就能往下走。

再有一种，Codex 真的卡住了或者原地打转。判断方法是看动作日志，如果连续几轮都在重复类似的操作、反复报同样的错、或者干脆没有任何动作更新，那就是卡了。这种时候等再久也没用。

卡住了怎么办？最直接的办法是手动叫停，重新描述任务，或者缩小任务范围。也可以直接停掉这个 thread，开一个新的，把上下文重新整理一遍再给 Codex。要是环境问题（比如依赖装不上、网络不通），就先暂停任务，去把环境收拾干净，解决完再让 Codex 继续。

连续失败时要叫停，不要让 Codex 无限乱试。Codex 有时候会陷入"改一版跑一遍发现还不对再改一版"的死循环，每多一轮都在烧额度，最后大概率还是不行。及时止损比硬等明智。

## 28. 任务跑完了结果在哪看

结果要在三个地方看，各有各的用处。

中间的对话框最快——Codex 跑完会给你一段文字总结，说做了什么、改了哪里、结果如何。这是最快速的概览，但也是最不可信的部分，文字总结可以糊弄，你得结合别的地方验证。

真正要看的是右边的侧边栏。如果 Codex 改了代码，侧边栏会显示 diff，你能逐行看 Codex 到底动了什么；如果生成了新文件，侧边栏能预览那个文件；如果做了网页，侧边栏能直接打开网页看效果。侧边栏就是 Codex"交作业的地方"。

最实在的是你项目文件夹本身。Codex 改的所有文件、生成的所有产物，最终都落在你建项目时选的那个文件夹里。打开 Finder 或者资源管理器，去那个文件夹看，所有东西都在那。不管 Codex 文字总结怎么写，文件夹里的真实状态是骗不了人的。

验收的标准流程是这样的：先扫一眼文字总结知道大概，然后认真看 diff 确认改动符合预期，再打开实际产物（网页、文件、图片）确认效果，最后跑一遍你自己的测试或者验证流程。这四步走完，才能说这个任务真的完成了。

千万不要只看文字总结就放心。每次完成后先看 diff，再决定要不要接受。文字总结 + diff + 实际产物 + 你的验证，四重检查一个都不能少。

## 29. 5 小时滚动窗口到底怎么算的

这是 Plus 用户最绕不清的一件事，搞不懂就总在莫名其妙烧额度。

先讲一个最容易误解的点：Plus 的 5 小时窗口看的不是"5 小时内你能发多少条消息"，看的是"5 小时窗口里你有多少推理计算量"。OpenAI 官方文档的原话是，本地消息和云端任务共享一个 5 小时窗口，窗口里的额度按照推理计算量（reasoning compute）来算，跟消息条数没有直接关系。

社区里有实测数据：Plus 大概是每个 5 小时窗口里有相当于 40 分钟左右的推理时间。Pro 5x 档大概是这个数字的 5 倍。注意，这个"40 分钟"指的是你跑的任务折算成推理计算量，大约等于 40 分钟的推理消耗，跟你实际坐在电脑前的挂钟时间没关系。一个简单的改文字可能几秒就完，几乎不消耗；一个复杂的多文件重构可能一条消息就吃掉好几分钟的推理额度。

这就是为什么很多人觉得"我没发几条消息额度就没了"——你发的几条都是重活，每条消耗大。同样，也有人觉得"我发了挺多条还有额度"——因为都是轻活。

这里还有个"滚动"的坑。窗口并不会"每 5 小时整点刷新一次"，而是滚动计算——你 5 小时前用的那部分额度，要到 5 小时之后才"释放"回来。所以你不会感觉到一个明确的"刷新时刻"，额度是慢慢恢复的。

除了 5 小时窗口，还有一个周限额。社区里有人实测，连续重用几天就会触顶。触顶之后只能等下周恢复。

## 30. 怎么查自己还剩多少额度

Codex 里能直接看，不用去网页查。

打开 Codex，找到设置里的额度查看入口，能看到当前账号的使用情况——还剩多少、什么时候刷新、速率限制状态。入口一般在左下角设置里，点开就能看到剩余用量。用完了就只能等下个周期自动刷新。

为什么要经常查？因为 Codex 不会在你快用完的时候主动提醒你。Codex 只会等你真的用完了，下一个任务发出去直接报错，告诉你额度不够。这种"突然断电"的体验很糟，尤其是你正跑着一个长任务的时候。

养成一个习惯：开一个稍微大点的任务之前，先扫一眼额度。够就跑，不够就等，或者把任务拆小。这种习惯能避免一半以上的"额度焦虑"。

如果你走的是 API Key 计费（不是普通用户的首选），查的就是余额和消耗，在 OpenAI Platform 的后台看。如果是接的国产模型，在对应厂商的后台看余额。这些非套餐路线的消耗更透明，但也更需要你主动盯着，因为按量计费跑偏了真的会爆账单。

还有一个细节：额度恢复不是瞬时的。有时候你看到"还有 1 小时恢复"，以为到点就能满血，其实恢复是渐进的，刚恢复那会儿额度很少，重活还是跑不动。安排任务的时候留点余量，别卡着点用。

## 31. Plus 和 Pro 的额度差得远吗

差得远，但大多数人不需要 Pro。

Plus 每月二十刀，给的是一个 5 小时滚动窗口里的推理额度，小任务够用。Pro 要贵得多，给的是 Plus 的好几倍额度。Pro 有 5x 档（Plus 的 5 倍）甚至更高档，数字看着吓人，但价格也吓人。Pro 5x 经常会有促销活动，期间额度再翻倍，但这种促销是临时的，过了就回到正常档位。

谁需要 Pro？只有那种把 Codex 当成生产力工具、每天重度使用、Plus 频繁触顶的人。普通人做做网页、改改文件、跑跑小任务，Plus 完全够，上 Pro 是浪费钱。

判断自己要不要升 Pro，最简单的方法是先用 Plus 跑一两周，看你会不会频繁撞额度墙。如果 Plus 的额度对你来说绑着用都跑不完，那 Pro 是浪费；如果你每隔几天就因为额度不够得歇着，那 Pro 才有意义。

还有一个隐形选项：买额外额度。Plus 和 Pro 用户额度用完了，可以单独买额外的 credits 继续用，不用升级套餐。这种适合"偶尔需要猛用一次"的人，比长期订阅 Pro 划算。

对中国用户还有一个特别的省钱路子：接国产模型。通过 CC Switch 接智谱、Kimi、DeepSeek，月费比 Plus 便宜一大截，还不用梯子。这条路是非官方的，有兼容性和稳定性风险，但对预算敏感又想持续用的人来说，是很实在的选项。

## 32. 额度烧完了任务会立刻停吗

不会立刻停，但会停。

额度检查通常发生在任务开始或者某些关键节点上。你正在跑的任务，大多数情况下会跑完当前这一轮，然后在你发起下一个任务、或者 Codex 要进入下一轮执行的时候，报错告诉你额度不够。

这意味着两件事。第一，正在跑的任务不用太担心被腰斩，大概率能跑完当前阶段。第二，你不能指望"刚好用完最后一滴额度把任务跑完"，因为额度耗尽的时机你控制不了，可能正好卡在你任务的中段。

更麻烦的是后台任务。你开了一个任务没盯着，关掉浏览器去干别的，任务在后台继续跑、继续消耗。等你回来发现额度早烧光了，任务可能跑了一半挂在中间，既没完成又赔了额度。这种"隐形消耗"是 Plus 用户提前烧光的主要原因之一。

怎么避免？长任务跑之前先看额度，预估一下够不够。拿不准的，拆成小段跑，每段跑完看一眼额度还剩多少。不要同时开好几个后台任务，那样消耗是叠加的，烧得特别快。

万一任务跑到一半额度真没了，别硬重启。先让 Codex 停在那，等额度恢复一部分，再决定是接着跑还是重新来。强行用仅剩的一点点额度去续跑一个需要大量消耗的任务，往往跑不完还白烧。

## 33. 手机上能用 Codex 吗

Codex 本身是电脑端的，手机上用的是另一个东西——ChatGPT 手机端的 Codex 里的 Codex 功能。

这两者能力不一样。Codex 跑在你电脑上，能读本地文件、操控你电脑上的应用、用本地环境。手机端的 Codex 里的 Codex 走的是云端，能干的事更接近 Codex Web 那一套，处理 GitHub 项目、跑云端任务，但碰不到你电脑上的本地文件。

不过手机端有一个特别实用的场景：续跑和远程控制。2026 年 5 月 OpenAI 还专门更新过，让 ChatGPT 手机版能直接发起 Codex 任务，包括在 Windows 上跑的 Codex 任务。你在电脑上开了一个长任务，出门了，路上用手机查看进度；Codex 有问题问你，你直接在手机上回复、批准 Codex 的下一步、或者指派新方向。你电脑上的本地环境安安静静在那干活，你人可以自由移动。

这种体验是：任务在你电脑上跑，但你的自由不受限制。手机版和电脑端能完全同步过去的所有会话，比 ChatGPT 的手机版稳定得多。

所以答案是：手机不能跑 Codex 本身，但能用手机端继续参与 Codex 上启动的任务。对长任务、后台任务特别有用。如果你只是想随时随地问点小问题，手机端的 Codex 也能用，只是能力比桌面版弱一截，毕竟碰不到你电脑的东西。

## 34. 关掉 Codex 任务还在跑吗

分情况，看你跑的是哪种任务。

如果跑的是本地任务（Local 模式或者 Worktree 模式），任务直接依赖你电脑上的 Codex 进程。你把 Codex 完全退出（注意是完全退出，不是最小化），任务就停了。这是本地任务的特点：任务在你的机器上跑，机器或者 Codex 关了，任务就没了。

这里有个坑：很多人以为关掉窗口就是退出 Codex，其实 Codex 默认是最小化到系统托盘（Windows）或者菜单栏（Mac），Codex 还在后台跑。你以为关了，其实没关，任务还在继续，额度还在烧。要真正退出，得去托盘图标右键选 Quit，或者菜单栏里选退出。

如果跑的是云端任务（Cloud 模式），那就不一样了。云端任务跑在 OpenAI 的服务器上，你电脑关机、Codex 退出都不影响，任务照样在云端推进。这是云端模式最大的价值——不依赖你的机器在线。

所以判断"关掉 Codex 任务还在不在跑"，先看你跑的是哪种模式。本地模式，关 Codex 等于停任务；云端模式，关 Codex 任务继续。拿不准的话，最稳的做法是：跑长任务或者要离开电脑的时候，用云端模式；日常短任务用本地模式，用完就退。

养成一个习惯：不用的时候彻底退出 Codex，别让 Codex 最小化在后台偷偷消耗。尤其是 Plus 用户，额度宝贵，被后台任务悄无声息地烧掉最亏。

## 35. Codex Credits 值得买吗

看你是什么用法，分两种人。

一类人，平时 Plus 额度够用，只是偶尔某几天需要猛用一次。这种人值得买 credits。Plus 的月度额度是固定的，超额了只能等下个月。如果你这个月临时有个大项目要赶，买点 credits 顶一下，比为了这几天升级到 Pro 划算得多。用完就停，下个月回到 Plus 节奏，灵活。

另一类人，几乎每个月都把 Plus 额度用光，经常被额度卡住。这种人不该买 credits，该升 Pro。如果你每个月都要补 credits，累计下来可能比 Pro 还贵，而且体验断断续续——额度一断任务就停，补了 credits 又得重接上下文，效率低。这种情况直接上 Pro，额度池子大，连续干活不卡壳，长期更省心。

判断自己是哪种人，跑一两个月就有数。如果你发现自己这个月买了 credits，下个月又买，连续两三个月都这样，那就是该升 Pro 的信号。如果只是偶发性的需求，credits 是最灵活的补充。

还有一个隐性成本要算：买 credits 之后心理上容易"放开用"，本来该省着用的也开始大手大脚，结果 credits 也烧得快。买之前先问问自己，是真有那么多活要干，还是只是用得更随意了。前者值，后者是白烧钱。
