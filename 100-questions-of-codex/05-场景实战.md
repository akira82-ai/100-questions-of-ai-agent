# 五、场景实战

## 72. 第一次该拿什么任务练手

别拿重要项目练手，别拿复杂需求练手，拿一个"成了挺好、废了也不亏"的小事开始。

具体拿什么任务练手？推荐让 Codex 做一个简单的 Hello 网页：黑色背景、页面中间显示大字、字体白色、整体水平和垂直居中、只用 HTML 和 CSS。这种任务特别适合练手——它清晰、简单、可验证，Codex 做完了你一眼就能看出对不对；涉及的文件少，不会出乱七八糟的依赖问题；做错了也没成本，改改再来就行。

要明确一点：第一次练手的核心目标是走通"建项目→提任务→看 Codex 执行→看 diff→验收"这整套流程，而不是真的做出什么有用的东西。你要感受的是 Codex 怎么工作、你怎么跟它交互、结果在哪里看。这些流程性的体验，用一个最简单的任务就能跑通，没必要用复杂任务给自己增加认知负担。

第一次还有几个小规矩得守住。先用普通对话问个简单问题，确认 Codex 能正常回复、整套环境配通了；然后建一个干净的项目文件夹，别用你已有的重要项目；打开计划模式，让 Codex 先出方案你过一眼；跑完之后认真看 diff、看实际产物，验收习惯从第一次就要养成。整个过程都别开完全访问权限，保持"请求批准"档位。

有一个观点值得记住：**很多 Codex 任务失败，原因不在 Codex 不会写代码，而在它还没理解项目就被要求直接动手**。所以第一次练手，重点是让自己熟悉"怎么和 Codex 合作"，而不是检验它的能力。等你把这套流程跑顺了，再上真正的活。

## 73. 让 Codex 做网页该怎么提需求

做网页是 Codex 最经典的场景，也是最容易看出"提需求方式决定结果质量"的场景。

新手最常犯的错是需求太模糊。一句"帮我做个网站"，Codex 会给你一个完全凭 Codex 自己想象的东西，技术栈、风格、结构全是 Codex 定的，和你想的差很远。你看着这个东西不满意，又不知道怎么改，就卡住了。

正确的提法是把需求具体化到 Codex 能照着做。可以套用一个结构。背景：一段话，你是谁、为什么要做这个网页。目标：一段话，描述最终状态，比如"一个展示我个人作品的 portfolio 页面，单页应用，响应式"。约束条件：列表，硬性规则，比如"只用 HTML 和 CSS，不用框架""配色用深色系""字体用思源黑体"。完成标准：怎么算完成，比如"在浏览器里打开能看到内容，移动端也能正常显示"。

还有几个能直接提升结果质量的技巧。先说用途和受众——"给我自己用的 portfolio""给客户看的展示页""做演示用的原型"，受众不同，Codex 给的结构和风格也差很远，提前讲清楚省得返工。更关键的是，先让 Codex 用图像功能生成概念图。在写 UI 代码之前，先让 Codex用 gpt-image 生成一两张图模拟页面外观，保存到项目里，然后看着图来写代码——最终的视觉效果，比让 Codex凭文字描述去设计好得多。还有一点值得一提：做完之后用应用内浏览器打开看效果。让 Codex 在侧边栏的内置浏览器里打开，你能直接看到效果，圈出不对的地方让 Codex改，比反复打字描述问题高效多了。

做完之后，标准验收流程：看 diff 确认改动、在内置浏览器看实际效果、在移动端尺寸下检查布局、点开每个链接看通不通。这些都过了，这个网页才算真做好。

## 74. 让 Codex 做 PPT 该怎么玩

做 PPT 是非程序员最能直接受益的场景之一。

开始之前先装 PPT 插件。Codex 默认不一定有 PPT 能力，需要装 Presentations 插件。装完之后，Codex 就能生成可编辑的 PPT 文件——是真的能打开改的演示文稿，不是图片或 PDF。

装好插件，接下来最要紧的是给 Codex 足够的素材。做 PPT 这件事，素材的质量直接决定结果的质量。做年中汇报 PPT，把相关的文件、数据、截图、PDF 都丢给 Codex，它能理解所有这些内容，整合成一份 PPT。你给的素材越丰富、越具体，做出来的东西就越贴合你的需求。

光有素材还不够，需求也得说清楚，这点跟做网页一样。模糊的需求只会得到模糊的结果。讲明白主题、受众、风格、要包含哪些部分、大致几页，尤其要明确风格——商务正式、活泼简洁、科技感强，这些风格词会直接影响 Codex 的设计。

另外得心里有数：AI 一次做出来的 PPT，肯定不会 100% 贴合你的要求。Codex 做完之后，你直接上手改就行；如果改动比较大，继续给 Codex 提要求让 Codex改，比你自己从头改快。

要是你做 PPT 比较频繁，还可以用 Skill 沉淀风格。网上有别人做好的 PPT Skill，把 Skill 地址发给 Codex，让 Codex 帮你安装，之后做 PPT 就按那个 Skill 的风格来。如果你自己有固定的 PPT 风格，也可以做成自己的 Skill，每次都按你的风格做，不用重新解释。

装插件只是最基础的路。社区里现在还流行两条更进阶的路线，效果往往比插件生成的 PPT 更好看。一条叫图片型 PPT：先让 Codex 用图像能力生成一张张 16:9 的视觉图片，再把图片铺进 PPT，每一页就是一张完整图片。这条路视觉效果好，但每一页是图片，编辑性差。另一条叫 HTML 型 PPT：不用传统 PPT 软件，直接用 HTML、CSS、JS 写幻灯片，能在浏览器预览、能做动画、能做交互、能用代码控布局，特别适合和 Codex 这种代码执行型工具配合。网上有不少开源的 HTML PPT Skill，搜一下让 Codex 装上就能用。

除了"写 Skill"，还有一种更新、更省事的 Skill 创建方式，叫 Record & Replay（录制回放）。你在 Mac 上把一套操作亲自演示一遍，Codex 在旁边看一遍，自动把流程整理成一个可复用的 Skill。比如你录一个"文章多平台分发"的流程——打开文章、改标题、整理标签、检查封面、切换平台、再处理一遍——Codex 观察后生成 Skill，下次你只要说"用我的文章分发 Skill，原文是这篇，平台是小红书和公众号"，Codex 就按你录的规则处理。

这条路特别适合那些"步骤固定但你写不清楚、做一遍反而清楚"的重复流程：内容分发、电商选品、报表整理、后台录入。有几个限制要先知道：目前主要是 macOS 支持，Windows 还得等；需要开启 Computer Use 权限；高风险操作（支付、转账、删库）不建议交给它录；页面经常变的网站录出来不稳定。最适合的是步骤稳、页面稳、成功标准明确、风险可控的流程。

做 PPT 最容易踩的坑是期待 Codex 一次完美。Codex 的价值在于给你一个 80% 完成的底稿，你再加工，指望它一次给你能直接用的成品多半会失望；指望它给你一个改起来比自己从头做快得多的底稿，这个期待就靠谱多了。

## 75. 让 Codex 做视频靠谱吗

能做，而且效果可以很惊艳，但要装对插件、给对素材。

Codex 自己不会直接剪视频，Codex 的视频能力靠插件。两条路：一条是 HyperFrames by HeyGen 插件，专门做动效视频；另一条是 Remotion Studio，用代码生成视频。这两条路的逻辑都是"用代码描述视频"，然后由对应的引擎渲染成真正的视频文件。

有一个特别生动的电商案例。一个做电商的人用 Codex 做了一个"电商剪辑工作流"：拆解爆款、识别和选择素材、自动配音加 BGM、自动对齐字幕。整个过程做成一个 Skill 之后，每次只要输入一个斜杠命令，Codex 就按提前设计好的流程跑，跑完在剪映草稿里能看到剪好的视频。就靠这一个工作流，他说电商素材产出效率翻了 5 倍，单日订单涨了 60%。

这个案例的启示是：视频这件事，"一句话丢给 Codex 就完事"做不成，视频是一套流程。拆解爆款、选素材、配音、字幕，每一步都是独立的任务，组合起来才是完整的视频工作流。把这个工作流固化成 Skill，下次一键调用，比每次重新解释流程高效得多。

让视频工作稳定输出，有几个要点得守住。素材一定要给齐，视频质量很大程度取决于你给的素材质量。流程也要拆细，别一句"帮我做个视频"就完事，要拆成"先拆解爆款结构""再选这些素材""按这个节奏配音""字幕按这个格式对齐"。最要紧的是结果一定要验证——Codex 做的视频你一定要看一遍，发现不对的地方让 Codex改，别默认接受。等这套流程跑通一次，就把它做成 Skill 固化下来，下次直接调用。

视频是 Codex 能力上限很高的一个场景，但门槛也比做网页、做 PPT 高。新手不要一上来就碰视频，先把基础流程跑顺，再往视频这种复杂场景延伸。

## 76. 让 Codex 处理 PDF 和文档怎么用

这是非程序员用 Codex 最高频的场景之一。

Codex 能干哪些事？分析体检报告、把 PDF 转成 markdown、提取 PDF 里的关键信息、整理一堆文档的内容、对比几份文档的异同。有人直接把家人的体检报告 PDF 扔给 Codex，让 Codex 分析之后梳理一份建议放到飞书文档上，全程不用自己读那些医学数据。还有人经常要把 PDF 转成 MD 文件，这事也交给 Codex 批量处理。

为什么这件事 Codex 做得特别好？因为文档处理是典型的"输入明确、输出可验"的任务。输入是一份或者几份文档，输出是结构化的整理结果，对不对你一眼能看出来。这种任务最能发挥 Codex 的执行能力，又不涉及高风险操作。

提需求的关键是把"输出格式"说清楚。不要只说"帮我分析这份报告"，要说"帮我分析这份报告，按以下结构输出：1. 总体健康状况，2. 异常指标列表，3. 每个异常指标的解释和建议，4. 后续要关注的事项"。输出格式越明确，Codex 给你的东西越有用。

处理多份文档的时候，把所有文档放在同一个项目文件夹里，让 Codex 一次性读完所有，然后做整合。比如对比三份合同的条款差异、把五份会议纪要整理成一份汇总、从十份简历里筛选出符合某个条件的候选人。这些任务手工做要花几小时，交给 Codex 几分钟就完。

文档处理有一个特别适合的场景：批量格式转换和清理。把一堆 PDF 转成 markdown、把一堆 word 文档提取出纯文本、把一堆乱糟糟的文件按某种规则重命名归类。这种重复性高、规则明确的活，Codex 做得又快又准，你手工做反而容易出错。

唯一要注意的是隐私。如果文档里有敏感信息（身份证号、银行卡号、公司机密），让 Codex 处理之前先想清楚。Codex 处理的内容会经过 Codex 的模型，虽然 OpenAI 有隐私政策，但涉及真正机密的内容还是要谨慎。

## 77. 让 Codex 分析数据和对账怎么做

数据分析和对账是另一个普通人高频受益的场景。

有一个特别典型的对账场景。一个做业务的人，每天上午要去检查昨天的回款数据，看几张表的数据能不能对得上。以前这事人工干，容易忘、容易漏。这个人把这个流程做成了一个 Skill：告诉 Codex 对哪里的账、怎么对、什么算异常。然后设了个定时任务，每天自动跑一次，发现异常就提醒。这个流程从他脑子里挪到了系统里，再也不会忘，再也不会漏。

这个案例揭示了数据对账类任务的关键：**你要做的，是把脑子里的对账规则写成 Codex 能照着执行的检查步骤，而不是指望它替你判断什么算异常**。你不说，Codex 不知道你要对什么、怎么算异常，自然帮不上忙。

提这类需求，有几个要素得讲清楚。先告诉 Codex 数据在哪里——飞书多维表格、Excel 文件、数据库、还是某个 SaaS 工具，让 Codex知道去哪读。然后说明规则：哪些字段要对、按什么 key 对、差异在什么范围内算正常、超过多少算异常。再讲清楚发现异常怎么办——是发邮件给你、推送到飞书、写进报告、还是只标记出来等你来看。最后别忘了约定正常情况输出什么，日报、周报、汇总表，什么格式。

涉及外部数据源的时候，可能需要 MCP。比如数据在飞书多维表格里，给 Codex 装飞书的 CLI 或者飞书 MCP，Codex 就能直接读飞书的数据。数据在 Notion 里，装 Notion MCP。数据在本地 Excel，那直接放项目文件夹里就行，不用 MCP。

对账这类任务特别适合做成 Skill 加自动化。Skill 把对账规则固化下来，自动化让 Codex 定时跑。组合起来，就是一个"AI 值班会计"，每天自己查、自己比对、发现异常主动汇报。用了这套之后，那些最耗时耗力的"收集背景资料"的工作往往在你回来之前就做完了，你只需要做最后拍板的决定。

## 78. 让 Codex 帮我清理电脑靠谱吗

靠谱，而且效果出乎意料地好，但要谨慎。

有个人用 Codex 检查最近装的应用是不是都卸载干净了。卸载软件这件事，很多人都遇到过：明明卸载了，系统里还残留一堆配置文件、缓存、依赖，慢慢把磁盘塞满。让 Codex 检查某个软件卸载干不干净，Codex 会去找这个软件相关的文件和依赖，看看有没有残留。如果有，直接让 Codex 清理。这个人说他在 macOS 上靠这个方法腾出了 20G 空间。

为什么 Codex 适合做这个？因为"找残留文件"这种事，规则明确（找特定名称、特定路径、特定后缀的文件）、可验证（找到的就是找到、没找到就是没找到）、又不涉及危险操作（删除前给你看清单，你确认才删）。这种任务 Codex 能稳定完成，你验收也容易。

但这里有安全边界必须讲清楚。首先，始终用项目可读写档，别开完全访问，把 Codex 限制在它能处理的范围里，别让 Codex整个硬盘乱扫。其次，删除前一定要看清单——让 Codex 列出来要删什么，你看完确认才让 Codex动手。还有，重要文件先备份，万一 Codex 判断错了要删的东西，你有备份能恢复。最要紧的是，碰到不确定的文件，别盲信 Codex 说它"没用"，多问一句"这是什么文件，删了会有什么影响"，等它给你解释完你再决定。

清理电脑这个场景，涉及删除数据的操作，最好别让 Codex 自动执行，务必你来确认。Codex 特别擅长"找出候选删除项"，但"删不删"这个决策要你来做。

这种"Codex 找、你定、Codex 执行"的协作模式，是处理有风险任务的标准做法。Codex 把最耗时的"找"做了，你做最关键的"判断"，Codex 再执行你确认过的操作。既发挥了 Codex 的效率，又保住了你的控制权。

## 79. 让 Codex 写文章和做内容怎么用

内容创作是 Codex 的强项之一。

有一个内容创作者的例子特别典型。这个人目前的内容输出基本靠 Codex 搞定。前几天发的一篇公众号文章，新号跑出 1000 浏览量，算是个小爆款。这篇文章从大纲到初稿，再到封面、配图，全部由 Codex 完成。这个人只是在完成之后做了一些细节上的调整。

这个人还有一套自动化的内容生产流程。每天 Codex 会给这个人准备一份 AI 行业日报，全网有哪些值得关注的资讯第一时间看到。然后还有一个选题 Brief 的自动化任务：Codex 每天读取日报内容，找出其中可以写的选题，改成需求 Brief 放在这个人的知识库里。那篇爆款文章，选题就是 Codex 给这个人做的。

这种用法揭示了内容创作的关键：**真正省力气的做法，是让 Codex 把"找选题、做调研、出初稿、配图"这些耗时的体力活扛下来，你只做"选哪个选题、改哪里、发不发"的判断**。那位作者的做法就是这样，只在完成以后做一些细节调整，效果还不错。

让内容创作稳定输出，有几个要点。最要紧的是给 Codex 一份风格指引——你的写作风格、目标读者、内容调性，写进 AGENTS.md 或者做成 Skill，Codex 每次都按你的风格来；网上也有别人做好的写作 Skill，可以直接用。光有风格还不够，输入也得给足，Codex 写得好不好很大程度取决于你给的素材和背景，先把选题、调研、素材都准备好（这些也可以让 Codex 做），再让 Codex写正文。还有一点心态得摆正：AI 一次出来的东西不会 100% 贴合你的要求，你要期待的是它给你 80% 的底稿，你再精修，而不是指望一次完美。

内容创作还有一个适合做成自动化的点：定期监控和选题。每天让 Codex 扫一遍你关注的信息源，找出值得写的内容，整理成选题清单放在你的知识库。等你坐下要写的时候，选题已经有了，调研也做了一半，你直接进入"写和改"的环节，效率高得多。

## 80. 做完整的 side project 怎么用最划算

side project（个人小项目）是 Codex 的甜区。

为什么 side project 特别适合？因为 side project 通常是一个人做、规模可控、需求自己定、错了也没大代价。这种场景下，Codex 的能力上限完全够用，Codex 的额度消耗也在可控范围，你能体会到"AI 替我把大部分活干了"的爽感。

最划算的用法是按"六步法"推进。完整流程是这样的：需求拆解→制定计划→小步实现→测试→代码审查→提交与复盘。每一步都有对应的提示词模板，逐步推进，比一句话丢个大需求靠谱得多。

第一步需求拆解。把你想做的东西写清楚，七个要素：背景、要解决什么、相关文件、不能动什么、完成标准、需要哪些测试、有哪些风险。第二步制定计划。用计划模式让 Codex 出方案，你审完确认。第三步小步实现。不要让 Codex 一口气把整个项目做完，分模块、分功能，一步一步来，每步完成立刻验证。第四步测试。让 Codex 写测试用例，跑通了才算这一步完成。第五步代码审查。推荐两轮审查：第一轮让 Codex 自审，把明显问题先暴露出来；第二轮你人工审，重点看边界条件、安全问题、是否误删、业务逻辑。第六步提交和复盘。改完提交到 git，记录这次遇到的问题、总结好用的 Prompt、更新 AGENTS.md。

side project 还有一个特别适合的工作模式：用 ChatGPT + Codex 组合。先用 ChatGPT 想清楚，再用 Codex 进项目执行。ChatGPT 是顾问，帮你理清思路、做技术选型、讨论方案；Codex 是实习生，进项目把活落地。两者配合，比单独用任何一个都强。

省额度，有几个习惯值得养成。复杂任务先开计划模式，规划阶段消耗少，规划对了执行阶段不浪费；小任务用低推理强度就行，复杂任务才上高档；不要一次让 Codex 扫整个项目，能指定文件就指定文件；另外别反复大范围重构，那种操作最烧额度。还有一条经验：拿不准的时候先让 Codex 讲思路，你确认了再让 Codex执行，比一头扎进去撞墙划算。

side project 是用 Codex 最好的"练习场"。你用 side project 把整套流程跑顺，熟悉了 Codex 的工作方式，再上真正的生产项目，心里就有底了。想让 Codex 半自动跑长项目，Q87 会专门讲那套"半自动工作流"怎么做。

## 81. 改 bug 该怎么交给 Codex 最有效

改 bug 是 Codex 最能体现价值的场景之一，但提需求的方式决定成败。

最有效的提法是这样的：把完整的报错信息给 Codex，加上能复现这个 bug 的步骤，加上相关的代码文件。标准做法是把报错信息直接发给 Codex，让 Codex 自己分析。Codex 能读到完整的报错栈、能跑到你的项目里复现、能改完再跑一遍验证。这种"Codex 在自己环境里真的跑通"的能力，是改 bug 场景里 Codex 比普通对话强得多的地方。

新手最常犯的错是只描述 bug 不给信息。一句"我的登录页报错了"，Codex 只能凭空猜。Codex 不知道报什么错、不知道你的代码长什么样、不知道怎么复现，给的建议也是泛泛的，解决不了你的实际问题。

正确的提法是把这几样东西凑齐。先把完整报错给它——控制台输出、错误信息、报错栈直接复制，或者截图（Codex 支持图片输入）。再讲清楚复现步骤：你是怎么触发这个 bug 的，点哪里、输什么、做了什么操作。还要指一下相关文件，告诉 Codex bug 大概在哪些文件、哪些函数里，缩小它的搜索范围。最后别漏了环境信息，什么操作系统、什么浏览器、什么版本，这些有时候很关键。

改 bug 还有一个特别适合的模式：让 Codex 先复现，再修。让 Codex 先在你的项目里跑一遍，触发那个 bug，确认 Codex 能复现。然后再让 Codex 修。修完再跑一遍，确认 bug 没了。这种"先复现、再修、再验证"的循环，比"Codex 给你一段代码你拿去试"靠谱得多——因为 Codex 在 Codex 自己的环境里已经验证过了。

遇到难修的 bug，推荐用 Goal 模式。给一个明确的终点："直到这个 bug 不再复现，修复才算完成。"Codex 会持续朝这个目标努力，试各种方法，每试一种都验证一遍，直到 bug 真的消失。

改 bug 最容易踩的坑是"修了一个引入两个"。Codex 改完 bug，你可能没注意 Codex 顺手改了别的东西，结果 bug 修了但引入了新问题。防御办法还是那条：每次跑完看 diff，发现 Codex 改了和 bug 无关的文件，立刻问、立刻回退。

## 82. 加新功能该怎么拆给 Codex

加新功能是 side project 里最常见的需求，标准做法是"小步实现"。

新手最容易犯的错是一句话丢个大功能。"帮我的网站加个用户系统"，Codex 听到这种需求，会按 Codex 自己的理解从零给你搭一套，登录、注册、权限、数据库、session 管理，全都做。结果大概率和你现有的项目结构对不上，引入一堆新依赖，还可能动到你不想动的地方。

正确的拆法是按"用户能感知的功能单元"拆，不按"技术模块"拆。比如"加用户系统"这个大功能，应该拆成：第一步加注册页（用户能填表单提交），第二步加登录页（用户能登录退出），第三步加权限控制（登录用户能看到特定内容），第四步加用户资料页（用户能查看和编辑自己的信息）。每一步都是一个"用户能感知的功能单元"，每一步都能独立验收。

每一步怎么执行？先用计划模式让 Codex 出方案，你审完确认。然后明确告诉它"本次只实现这一步，不要顺手做下一步"，把它框在当前这个功能单元里。做完立刻验收，确认这一步真的完成了，再进入下一步。每步完成提交一次 git，留个版本节点，后面想回退也方便。

有一个反模式要警惕：**让 Codex 顺手重构无关代码**。Codex 在加新功能的时候，可能觉得现有代码"不够优雅"，顺手重构一下，结果引入各种意外。防御句式写进任务描述里："本次只实现当前功能点，不要顺手重构无关代码，不要修改命名、目录结构、依赖版本和全局配置。"

加功能还要注意一个点：回归测试。改了首页按钮，也可能影响复用同一组件的其他页面。新功能加进去之后，不光要测新功能本身，还要测原有的功能有没有被影响。这种"回归测试"是最容易被忽略的——你盯着新功能，忘了确认 Codex 有没有顺手搞坏别的。让 Codex 跑一遍完整的测试套件，或者你手动点一遍核心流程，确认没引入新问题。

## 83. 老项目敢不敢交给 Codex 重构

敢，但要极其谨慎，而且一定要用 Worktree 模式。

老项目重构是 Codex 能力上限很高的场景，也是最容易翻车的场景。老项目通常代码多、历史包袱重、耦合复杂、没准还有一堆历史遗留问题。让 Codex 在上面做大改动，风险天然就高。

老项目有几个特有的坑得提前知道。首先，老项目本身就可能有 lint 或类型问题，别让 Codex 顺手重构——它一看到这些"问题"就想"修一下"，一修就牵连出一堆改动，把本来能跑的项目改崩。其次，老项目的历史包袱要尊重，有些代码看着"烂"，却是当年为了某个特殊原因写的，改了可能引入隐蔽的 bug。最要紧的是，老项目的耦合比你想的复杂，改一个地方，可能影响你完全没想到的别的地方。

正确的做法有一套。一定要用 Worktree 模式，给 Codex 一个独立的工作副本去改，你的主项目纹丝不动；它改完了你审，审过了再并回主项目，改坏了直接丢掉 worktree，零风险。每次只让 Codex改一小块，别"重构整个模块"，而是"只把这一个函数拆清楚""只把这一个组件抽出来"，改动范围越小，引入风险越低。每步都得跑完整测试，老项目能改的前提是有测试，没有测试的老项目，每改一步都得你手动验证核心流程。另外，看着 diff——Codex 改的每一行你都要看，发现它顺手改了无关代码立刻回退。

老项目重构还有一个适合用 Goal 模式的场景：代码迁移。把一个内部工具从 Python 迁移到 Rust。建好新目录，设定目标："直到所有单元测试全部通过，这个新版本的开发才算完成。"Codex 持续往这个目标努力，每改一处都跑测试，直到所有测试通过。

老项目重构最重要的心态调整：**慢就是快**。不要急着让 Codex 一次重构一大块，宁可慢一点、稳一点，每步都验证。有一句话值得记住——交付不是结束，复盘才是下一次效率提升的开始。重构完一块，复盘一下：哪里翻车了、哪里 Codex 做得好、下次怎么拆更合理。这种复盘能让后续的重构越来越顺。

## 84. 写测试用例怎么交给 Codex

写测试是 Codex 的强项，写测试被推荐为稳定输出的关键技巧之一。

为什么写测试特别适合交给 Codex？因为测试的特点就是"明确输入、明确预期、验证输出"，这种规则清晰、可自动验证的任务，正是 agent 最擅长的。Codex 能读懂你的代码、推断边界条件、生成各种测试用例，然后自己跑一遍验证哪些通过哪些失败。

最有效的提法是：先让 Codex 读懂要测的代码，再让 Codex 生成测试。不要直接说"给这个函数写测试"，而是说"读这个函数的实现，找出所有边界条件，为每个边界条件写一个测试用例，跑一遍确认都能通过"。这种提法能让 Codex 真正理解代码、找全边界，而不是机械地生成几个常规用例。

还有一个反向用法：先写测试，再让 Codex 实现。这种思路特别适合新功能。你先告诉 Codex"这个功能要满足这些测试用例"，让 Codex 看着测试去实现功能。测试是明确的、可验证的，Codex 实现的代码只要能让所有测试通过，就算完成。这种"测试驱动"的方式，比"实现完再补测试"稳定得多，Superpowers Skill 就是这种思路——先澄清需求、写规格、做计划、用 TDD 推进。

写测试有几个坑要避开。别让 Codex 只写"happy path"测试——它倾向于写"正常情况能跑通"的测试，可真正有价值的是边界测试，比如空输入、超长输入、非法输入、并发情况。明确让 Codex"重点测边界条件"，不然给你的测试覆盖不了真实场景。还有一个隐蔽的坑：测试和实现可能一起跑偏。有时候 Codex 写测试按自己的理解，写实现又按另一个理解，结果测试通过了但功能是错的——防御办法是先确认测试用例本身对不对，再让 Codex实现。另外，测试也要看 diff，你不看就接受，可能引入一些没意义的测试、漏掉关键测试，扫一眼它写的测试，确认覆盖了你想覆盖的场景就行。

写测试是那种"前期投入、长期受益"的事。第一次让 Codex 写测试可能比直接让 Codex 实现慢，但有了测试之后，后续每次改动都能跑测试验证，回归问题大幅减少。写测试被列为稳定输出的关键技巧之一，理由就在这里。

## 85. 让 Codex 做代码审查靠谱吗

靠谱，而且有些方面比人审更细，但要会用。

Codex 做代码审查，干的事就是让 Codex 读一组代码改动，找出其中的问题。推荐两轮审查的做法。第一轮让 Codex 自审，Codex 输出一份问题清单，把明显的 bug、边界问题、安全隐患先暴露出来。这一步相当于"让写代码的 Codex 自己挑自己的错"，往往能挡住低级问题。第二轮人工审，你看着 Codex 列出来的清单和 diff，重点判断那些 Codex 发现的问题是不是真问题、有没有遗漏。

这种"一个写、一个挑错"的模式，特别推荐用于高风险场景：涉及登录、支付、权限、数据库、鉴权的改动，建议用第二个模型做交叉审查。第一个模型写代码，第二个模型审代码，两个模型从不同角度看，比一个模型自己写自己审靠谱得多。

Codex 做代码审查擅长什么？边界条件它能想到各种边界输入，提醒你哪里没处理；安全问题，像常见的注入、越权、敏感信息泄露，它能识别出来；明显 bug，比如空指针、数组越界、逻辑错误，这些它也能发现；还有风格和一致性问题，命名不一致、不符合项目约定，它也能指出。

Codex 做代码审查的局限也要清楚。业务逻辑它看不出——你的代码是不是符合业务需求，它没法判断，因为它不懂你的业务，这部分得你来审。架构合理性它的判断也有限，能看出局部问题，但对整体架构是不是合理，它说的不一定靠谱。还有一点，它可能误报，说"这里有问题"，其实是你故意这么写的，它不知道上下文。所以它的报告你要判断，不能全信。

代码审查还有一个特别适合的场景：在合并前审一遍。让 Codex 读 diff，列出问题清单，你逐条判断。这比你自己逐行读 diff 高效，因为人的注意力有限，盯着 diff 看久了会漏掉问题，而 Codex 能从头到尾保持一致的细致度。

但永远记住那条硬规矩：**不要让 AI 自动合并 PR**。Codex 能审，能发现问题，但"改不改、合不合并"的决策永远是你做的。Codex 的审查是辅助，不是替代你的判断。

## 86. 团队多人协作能用吗

能用，但和单人用法不一样，有一些特别注意点。

先说项目文件夹的协作。Codex 工作在一个本地文件夹里，如果团队多人都要用它处理同一个项目，最稳的方式是 Git。每个人在自己电脑上跑 Codex，改完提交到 Git，互相通过 Git 协作。千万别让多个人同时用 Codex 改同一个文件夹——文件状态会冲突，改动会互相覆盖。

更关键的是 AGENTS.md 在团队里的价值。团队场景下，把团队的代码规范、技术栈约定、不能动的文件、提交规范都写进 AGENTS.md，放在项目根目录。这样不管团队里谁打开这个项目用 Codex，Codex 都按同一套规矩来，输出风格统一。AGENTS.md 本身也提交到 Git，团队所有人共享同一份规则。

Skill 同样可以团队复用。团队里某个成员把某个工作流做成了 Skill，共享给其他人就行——Skill 就是文件夹里的 markdown 文件，提交到 Git 仓库就能让所有人用。比如团队统一一个"代码审查 Skill"，所有人审查代码的时候都按这个 Skill 来，审查质量自然统一。

另外两点也得心里有数。额度是按账号算的，团队多人用 Codex，每个人用自己的 ChatGPT 账号，额度独立；想统一管理额度，可以考虑 Team 或者 Business 套餐，Business 套餐有团队管理功能，但具体套餐和额度会变，以官方页面为准。还有敏感数据要小心——团队场景下让 Codex 处理的数据，可能涉及客户数据、商业机密、内部系统，处理之前要想清楚数据会经过 Codex 的模型，符合公司的数据安全政策；真正机密的内容，谨慎处理或者干脆不用 Codex。

团队协作的核心原则是：**Codex 是个人工具，团队协作靠 Git 和共享配置文件**。每个人用自己的 Codex 干自己的活，通过 Git 合并工作成果，通过共享的 AGENTS.md 和 Skill 保持一致性。不要试图把 Codex 当成"团队共享的工具"，Codex 的设计不是为多人同时操作同一个文件夹准备的。

## 87. 想让 Codex 半自动跑长项目该怎么搭

很多人听完自动化的故事，以为"完全自动化"就是丢一个 prompt 给 Codex 然后去睡觉。社区里真做过的人会告诉你：这种玩法风险极高，真正的"半自动"是另一套东西。

被多人验证过的半自动工作流，由三样东西撑起来：一份任务清单、一个监工机制、频繁的版本留底。

任务清单是你用 ChatGPT 先把需求拆成结构化的清单，写进一个文件（比如叫 `task-list.md`），当作 Codex 的"进度账本"。Codex 知道全局要做什么、做到哪一步了，照着清单往下推进。

监工机制（社区里俗称 watchdog，可以理解成"监工"）负责周期性检查清单进度，把跑偏的 Codex 拽回正轨。Codex 长任务里最容易偏航，光有清单不够，还得有东西盯着它别走神。

版本留底就是每完成一步就给你的项目存一个版本快照（用 Git 就是 commit 一次，不用 Git 就复制一份留底）。万一某一步跑崩了，你能回退到上一个已知良好的状态，不至于全军覆没。

这套循环能跑到什么程度？社区里有实测：数小时无人值守、跨多文件边实现边测试的循环，是可达成的，有人用这套跑过 8 小时的长任务。但现实的边界也要说清楚。消耗大，一次数小时自主跑可能吃掉周额度的一成多。会偏航，所以才需要监工机制。风险高，无人监督的 Codex 是最危险的。

所以"完全自动"这个词得理解对：真正的目标，是搭一套"任务清单 + 监工 + 频繁版本留底"的半自动循环，让你能在不全程盯着的情况下持续推进，同时保留随时介入和回退的能力，而不是"一个 prompt 然后睡觉"。比那种"全交给 AI"的幻想，这种半自动循环诚实得多，也实操得多。

这一套偏进阶，如果你刚上手，先把前面 Q72 到 Q86 的基础场景跑顺，再考虑半自动。普通用户不一定要碰它，知道有这么个方向就行。
