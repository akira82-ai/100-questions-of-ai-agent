# 六、边界、替代与反思

## 88. 什么时候该继续用 Codex 其实已经错了

真正会用一个工具的人，不光知道什么时候用 Codex，更知道什么时候不该用。Codex 也有边界，过了边界硬用，只会烧钱受气。

最常见的是把它当聊天机器人。你只是想问个概念、聊个思路、要一段通用代码片段，这种事丢给普通 ChatGPT 对话就行，几秒就回，不烧额度，让 Codex 等几分钟纯属浪费。一个简单的判断标准：一般任务用普通对话，要处理本地文件或项目才动用 Codex。

再有一种，是拿它干没有验收标准的事。"帮我评估这个设计好不好""帮我想几个产品方向""帮我看看这个市场怎么样"，这类主观判断的活，Codex 的执行能力派不上用场，结果只是用更慢、更贵的方式做了一个普通对话能做的事。

更麻烦的是把它当作"替你想清楚需求"的工具。你脑子里对要做的东西还一团模糊，就丢给它跑，Codex 只会按自己那套模糊理解往下走，结果大概率不对。它没法替你想清楚要什么，它只会替你猜。

最要紧的是高风险且不可逆的事。让 Codex 操作付款、改生产数据库、删重要文件、动银行账号，这些一旦出错没法收。凡是涉及付款、账号设置、删除数据的操作，最好别让 Codex碰。

判断该不该用 Codex，可以顺着这个思路捋一遍：这事到底要不要 Codex"动手执行"？做完了你能不能验收？万一错了代价大不大？几条都想清楚了，再决定是动用 Codex，还是换个工具、自己上手。

## 89. 不想买 Plus 有没有别的平替

有，而且对中国用户来说，平替可能是更优解，不是凑合。

最主流的平替是接国产模型。Codex 不一定要用 OpenAI 官方额度，Codex 支持通过 CC Switch 这种工具接入国产模型——智谱 GLM、Kimi、DeepSeek 是最常用的几家。月费比 Plus 便宜一大截，还不用梯子，国内直连就能用。

很多人日常就是挂着国产模型在用。有人切到 Kimi，说 Kimi 是国内接入 Codex 效果最好的模型之一，原生支持多模态，能识别图片和视频，做 PPT、表格、网页生成审美也很好；有人挂智谱 GLM，说 GLM-4.7 是国产编程模型第一梯队；有人用 DeepSeek，图的就是价格最低，适合预算敏感的人。

但接国产模型这条路，有几个前提得先讲清楚。最根本的一条是，这是非官方玩法，不属于 OpenAI 官方功能，所以模型兼容性、稳定性、隐私、费用、工具调用的支持程度，都得看第三方。配置这一关也比直接登 ChatGPT 账号麻烦得多——得用 CC Switch 这种桥接工具，配 API Key、设路由，不是点一下就完事。功能上同样可能有差异，OpenAI 自家模型在 Codex 上的某些能力（比如某些插件、某些工具调用），国产模型不一定完全兼容。还有一句务实的提醒：重要项目先用测试仓库跑通，再往生产项目里上。

CC Switch 这个工具值得多说两句。CC Switch 把"来回改配置文件"这件事图形化了。以前想切换不同模型，得手动改配置文件里的接入地址和密钥，存盘重启。CC Switch 把这套流程变成了托盘里点一下就完事。CC Switch 还能统一管理 MCP、Skills，统计用量和消耗。对玩多家国产模型的人来说，CC Switch 几乎是必装的工具。

除了接国产云端模型，Codex 现在的格局更大了：它已经支持直接接入任意开源模型——Ollama 本地模型、LM Studio、vLLM 这些都能接进来。换句话说，你甚至可以在自己电脑上跑一个本地小模型，零成本用 Codex 的界面和工作流。更骚的是配置方式：不用自己手改配置文件，直接跟 Codex 说一句话——"把 Ollama 配成 Codex 自定义模型，模型用 qwen3:8b，Base URL 用 localhost:11434"——Codex 会自己去读 config.toml、加配置、检查语法、告诉你要不要重启。全程你不碰一行配置代码。让 Codex 配 Codex 自己，这种递归感很极客。

这种"什么模型都能接"的开放性，带来一种新的省钱思路：分层处理。便宜的本地小模型或者国产轻量模型，放在检索、改小文件、跑批处理这类简单任务上，响应快还省钱；复杂规划和难活再交给强模型。生产环境本来就该这么分层——简单任务用本地小模型，复杂推理才动用贵的云端大模型，整体成本结构会灵活很多。

接国产模型或开源模型适合什么人？预算敏感、不想折腾梯子、愿意花点时间做配置、能接受非官方玩法的兼容性风险。如果你追求最稳、最省心、最完整的功能，还是 ChatGPT Plus 直接登录。两条路各有代价，看你更在乎什么。

## 90. Codex 和 Claude Code 怎么选

Codex 和 Claude Code 是同一个生态位的东西，定位相似但风格不同。重度用户很多是两个装着用，根据场景切换。

先说定位差异。Claude Code 是 Anthropic 出的命令行 agent，跑在终端里。Codex 有 Codex、CLI、IDE 插件、云端多种形态，对普通人更友好的是 Codex。两者都能读你的项目、改文件、跑命令、交付结果，核心能力是重合的。

再说各自的优势。有一句总结很到位：Claude Code 啃复杂代码库更稳，Codex 写 Python 数据脚本和前端样式比较顺。Claude Code 因为是纯命令行，跟终端、git、shell 脚本配合更顺，程序员用起来如鱼得水。Codex 因为是图形界面，能开多个项目并行跑、内置浏览器看效果、能操控电脑，对非程序员和不碰命令行的人更友好。

能力上的差异。横评显示，Claude Code 在 SWE-bench Verified（一个主流的软件工程基准测试）上领先，复杂多文件工作输出质量更高。Codex 在 token 效率上表现好，做同样的任务消耗更少。这意味着 Claude Code 更适合复杂、对质量要求高的任务，Codex 更适合追求性价比的场景。

社区里有一种比喻很到位：**Codex 像个靠谱的中级工程师，听话、快、少翻车；Claude Code 更像个能跟你讨论架构的资深，更聪明、更有想法，但也更贵、更慢、偶尔跑偏**。Codex 在可靠性、失败频率、沙箱体验、每日额度这些"日常干活"的维度上口碑更好；Claude Code 在架构推理、长会话记忆、深度调试这些"啃硬骨头"的维度上更强。一个被反复提到的差别是：Codex 适合"明确任务交给它执行"，Claude Code 适合"模糊需求一起探索"。

有个有意思的细节。Codex 做 code review 的时候，会给每个问题标优先级，这一点社区很喜欢。但 Codex 很少解释"为什么是这个优先级"，透明度不如 Claude。这也是两者风格差异的一个缩影：Codex 偏执行、少解释；Claude 偏推理、多讨论。

那普通人到底该怎么选？头一个看你会不会用命令行：完全不想碰终端，Codex 几乎是唯一选择；习惯命令行，Claude Code 体验更纯粹。再看做的事的类型——写代码、改项目、做技术活，两者都行，看你对哪个模型更信任；而做内容、做 PPT、处理办公文档、操控电脑这些非编程的事，Codex 因为有图形界面和各种插件，能力强一截。预算也得算进去，Claude Code 要 Anthropic 的订阅或 API，Codex 要 OpenAI 的订阅或国产模型接入，两边价格不一样，国产模型接入两边都支持。

重度用户的真实做法是：两个都装着用。啃复杂代码库、做严肃的工程任务，回 Claude Code；做前端、调样式、做内容、处理办公事务，用 Codex。说到底这无关站队，只是根据任务挑合适的工具。

## 91. 已经在用 Cursor 了还有必要开 Codex 吗

看你用 Cursor 干什么。Cursor 和 Codex 是两种不同的东西，不是替代关系。

Cursor 是一个完整的 IDE（代码编辑器），Cursor 的 AI 能力嵌在编辑器里。你写代码的时候，Cursor 能帮你补全、帮你改、帮你查。Cursor 的强项是"在你写代码的过程中实时辅助"，是一种嵌入式的体验。

Codex 是一个独立的 agent，Codex 能进入你的项目、独立完成任务、交付结果。Codex 的强项是"你把一个完整的活交给 Codex，Codex 替你从头干到尾"。这种模式下，你不用全程盯着代码，Codex 自己跑。

有一种用法很好：**用 Cursor 做日常编码和局部修改，用 Codex 做任务推进和工程交付**。这两者是互补关系，不必二选一。日常你在 Cursor 里写代码，需要 AI 帮你补全、改某个函数、解释某段逻辑，用 Cursor 顺手；当你有一个明确的任务要交给 AI 独立完成——做个新功能、修个复杂 bug、重构一个模块——把任务丢给 Codex，让 Codex 替你跑。

还有一个差异是能力范围。Cursor 的能力主要在编码相关的事，Cursor 能做的事都围绕"代码编辑"展开。Codex 的能力更广，Codex 能做 PPT、处理 PDF、操控电脑、做视频、剪素材、分析数据，这些 Cursor 都做不了。如果你除了写代码还有大量非编码的事要处理，Codex 的价值更明显。

判断要不要同时用，可以问自己两件事。一件是你的工作是不是只有写代码——是，Cursor 可能就够；不是，Codex 能帮你处理一堆 Cursor 处理不了的事。另一件是你更偏爱"边写边让 AI 辅助"，还是"把整个任务丢给 AI 跑"——前者 Cursor 体验更好，后者 Codex 更合适。

有一个观点值得记住：**工具站队是让 Codex 只发挥 3% 潜力的习惯之一**。根据身份而不是任务选择工具，是错的。该用什么工具，由当前的任务决定，不由"我是哪个工具的人"决定。

## 92. Codex 的额度会一直涨下去吗

这个问题没人能给确定答案，但可以从趋势里看出一些方向。

从历史看，额度是越来越宽松的。早期的 Codex 额度非常紧，免费档几乎没法用，Plus 也很容易撞顶。随着模型成本下降、用户规模上来、竞争加剧，OpenAI 在持续放松额度。ChatGPT Pro 5x 套餐给到 Plus 的 5 倍额度，还经常有临时加额的活动（比如某段时间 2 倍额度提升）。整体趋势是往更宽松走。

但额度放松不等于"无限"。agent 类任务的成本天然比普通对话高，每跑一轮都要消耗算力，这部分成本是真实的。OpenAI 不可能无限放松，否则商业模式撑不住。社区里一直有用户呼吁更透明的额度系统、更可视化的用量预测，说明现在的额度管理还不算完美，用户焦虑依然存在。

未来额度怎么走，还看几个变量。最直接的是竞争，Claude Code、Cursor 这些对手在后面追，OpenAI 想留住用户，额度就得保持吸引力。模型推理成本如果继续降，额度自然也有放松的空间。规模也在帮忙——Sam Altman 在 2026 年 4 月透露 Codex 已经有约 400 万周活跃用户，规模效应能摊薄成本。还有商业化策略这一层要警惕，OpenAI 可能把更多能力塞进更高档的套餐里，看着基础额度放松了，高级能力却悄悄变贵。

对普通用户来说，与其猜未来额度怎么变，不如建立"不依赖大量额度"的使用习惯。小步用、按需用、盯消耗、不浪费。这种习惯不管额度怎么变都不会吃亏。反之，如果你养成"反正额度多就乱用"的习惯，等哪天额度政策变了，你会非常不适应。

## 93. 为什么 OpenAI 把 Codex 越做越像操作系统

这是个有意思的观察。Codex 现在能管文件、能装应用（插件）、能定时任务（自动化）、能记住你的习惯（记忆）、能控制其他应用（操控电脑）。这些能力加起来，Codex 确实越来越像一个"AI 操作系统"。

为什么会这样？OpenAI 团队的思路透露了逻辑。OpenAI 认为，我们在电脑上做的大部分工作，都和代码息息相关：执行终端命令、浏览网页、调用 API、导出文档、响应事件、触发自动化。当 Codex 延伸到这些领域，Codex 给人的感觉就不再只是编程助手，而是能帮你搞定各种电脑工作的"全能打工人"。

这个方向的逻辑是合理的。如果你有一个能执行任务的 AI，把 Codex 限制在"只写代码"是浪费。让 Codex 能管文件、能操作应用、能定时跑任务、能记住你的偏好，Codex 的价值会指数级放大。一个真正能干活的 AI 助手，自然会演化成一个"操作系统"——因为 Codex 需要调度各种资源来完成任务。

对用户来说，这是好事也是隐忧。好事是，你越来越多的事可以交给 Codex，效率提升明显。隐忧是，Codex 对你的电脑、你的数据、你的工作流介入越来越深，依赖越来越重。那些"幕僚长"对话流、"产品发布"对话流、自动化任务定时跑、知识库持续更新——这些场景里，Codex 已经深度嵌入他们的日常工作，离开 Codex 会很不适应。

这种"AI 操作系统化"还在早期，未来会演化到什么程度不好说。但趋势是明确的：Codex 不满足于只做编程工具，Codex 在往"你电脑上的 AI 总管"方向走。这个方向最终能不能成、用户买不买账，要看 Codex 在便利和风险之间能不能找到平衡。

## 94. Codex 会不会很快被新东西替代

可能会。但要担心的，未必是"另一个 Codex"这类同形态竞品——更值得警惕的是更深层的范式变化。

短期内，Codex 这个形态（本地装一个 AI 应用、能读文件、能操控电脑、能跑任务）大概率会持续存在。这个形态解决了真实需求，用户也有粘性，OpenAI 自己也在持续投入。Sam Altman 透露的 400 万周活数字说明这个产品已经有规模，不会被轻易放弃。

但中长期看，几个变量可能改变格局。最显眼的是模型本身——如果模型变得足够强、足够便宜，"本地装一个独立应用"的必要性可能下降，更多能力会直接嵌进操作系统、嵌进浏览器、嵌进你用的各种工具里。交互方式也值得留意，现在你和 Codex 之间主要靠打字，要是语音、视觉、意图理解成熟到一定程度，交互的形态可能完全变样。更外层还有平台策略这一层，Apple、Microsoft 这些操作系统厂商自己在做 AI 能力，一旦把 AI 深度集成进系统，第三方 AI 应用的空间就会被挤压。

对用户来说，怎么应对这种不确定性？头一条是别把鸡蛋放一个篮子里。Codex 是好工具，但你的核心能力——理解问题、拆解任务、判断结果——应当是工具无关的，换了工具照样能用，这才算真能力。再者，要把功夫下在方法论上：具体的命令、界面、配置项会变，但"怎么给 AI 清晰的任务""怎么验收结果""怎么拆解复杂任务"这些方法是通用的。也别因为已经摸熟了 Codex，就对新东西关上门——新工具出来的时候试一试，往往有意外收获。

根据身份而不是任务选择工具是错的。同理，根据"我是哪个工具的用户"而拒绝尝试新工具也是错的。工具会变，你解决实际问题的能力不能被某个工具绑死。

## 95. 普通人真的需要 AI agent 写代码吗

这个问题没有统一答案，取决于你想解决什么问题。

先说不需要的情况。如果你完全不想理解代码、不想理解技术、只想让 AI 替你把所有事做完，那 agent 对你的价值有限。因为 agent 给你的结果你得能验收，验收不了的话，AI 做得再好对你也没用。Codex 能提高效率，但不能替你做判断。如果你连判断都做不了，agent 帮不到你。

再说需要的情况。你有具体想解决的问题——做个个人网站、做个小工具、自动化处理一些数据、分析一些文档——这些事以前要么得自己学编程、要么得找程序员帮忙。现在有了 agent，你不需要成为程序员，但你需要能描述清楚需求、能验收结果。这种"不写代码但能用 AI 做出东西"的能力，对普通人来说是非常实用的。

那些非程序员的案例很有说服力。做电商的用 Codex 剪视频、做内容创作的用 Codex 写文章、做业务的用 Codex 对账、普通人用 Codex 分析体检报告。这些人都不是程序员，但他们用 Codex 解决了真实的、具体的问题。对这些人来说，AI agent 早就成了手边一件再普通不过的工具，该用就用。

更深层的问题是：AI agent 让"会编程"这件事的门槛发生了变化。以前会编程意味着能写代码，现在"会编程"更多意味着能和 AI 协作把事情做出来——能描述需求、能验收结果、能判断对错、能迭代优化。这种"新会编程"的能力，对普通人来说越来越值得掌握。

所以普通人需不需要 AI agent 写代码，跟"是不是程序员"关系不大，更要紧的是你有没有想用代码解决的问题，愿不愿意学着用 AI 去解决。有，就值得用；没有，先找问题再说。

## 96. 把整个项目交给 AI 改风险有多大

很大，而且比你以为的更大。这是高风险场景。

风险来自好几个方面。最先是 Codex 看不到全局——即便它能读你整个项目，注意力也是局部的。它改 A 的时候，常常意识不到 A 还牵连着 B、C、D，改完 A 把那几处搞坏了，它自己还不知道。更阴险的是它会顺手重构：觉得某段代码"不够优雅"就动手，重命名一个组件，所有引用都得跟着改，改错一处就报错；改全局样式，别的页面布局跟着崩；最要命的是删代码，它觉得"没用"的，往往是你的业务逻辑。还有认知上的错位——Codex 觉得核心功能跑通就算完事，你却还要处理边界情况、写测试、补文档，结果到手的是个半成品。再加上回退成本高，它一改二十个文件，你发现方向错了，回退常常比从头来还麻烦。

防御这些风险的办法其实很全。头一条，别让 Codex一次改一大块——六步法、小步实现，都是为了把改动范围圈住。配套的是 Worktree 模式：给它一个独立副本去改，主项目不动，改坏了直接丢。每一步还要看 diff，发现它顺手改了无关代码立刻回退。指令上要写防御性的话，比如"本次只实现当前功能点，不要顺手重构无关代码，不要修改命名、目录结构、依赖版本和全局配置"。最后别忘了回归测试，改完不只测新功能，还要把原有功能过一遍，看有没有被牵连。

最根本的心态是：**把整个项目全权交给 Codex 去改，是最危险的做法**。哪怕你对 Codex 再信任，哪怕 Codex 历史表现再好，这种"放手让 Codex 干"的用法早晚会翻车。正确的做法是"Codex 改一块、你看一块、确认一块、再让 Codex 改下一块"。慢一点，但稳。有一句话值得反复记：AI 改得很快，但你不知道 Codex 改了什么，也无法放心交付。

## 97. 重度依赖 Codex 会不会让我退化，还算我会写代码吗

这两个问题其实是一回事：工具替你做了越来越多事之后，你的能力是涨了还是废了，你这个人还算是"会"那个事的人吗。

先说退化。退化的风险是真实的。如果你所有事都丢给 Codex，自己不思考、不判断、不学习，长期下来你的某些能力确实会萎缩。就像 GPS 出现之后，很多人的方向感变差了——因为你不再需要自己记路、认路。同理，如果你所有代码都让 Codex 写、所有判断都让 Codex 做，你的代码能力和判断能力可能会变弱。

但这种"退化"不是必然的，取决于你的用法。那些把 Codex 用得很好的人，他们并没有退化，反而在进化。原因在于他们用的是"协作模式"——他们出思路、Codex 执行、他们验收、Codex 再优化。这种过程里，他们学到了新东西，他们的判断力、拆解问题的能力、对代码的理解，都在提升。反之，"外包模式"下把所有事丢给 Codex、自己不参与，那确实会退化。

判断自己到底是协作还是外包，有几个信号可以照一照。你能不能解释 Codex 给你的代码？能解释，说明你理解了，是协作；解释不清，多半只是接受了结果。Codex 出错的时候你能不能发现？能发现，说明你还在判断；发现不了，那就是完全依赖。还有一条更长远：你从 Codex 身上有没有学到东西？学到了，说明你在成长；什么都没学到，说明你在被动接受。

再说"算不算会写代码"这个身份焦虑。本来就会写代码的人，Codex 只是个更聪明的工具，就像 IDE 的自动补全、Stack Overflow 的答案一样，是辅助，你当然还是"会写代码"。不太会写代码、主要靠 Codex 做事的人，你能描述需求、能验收结果、能判断对错、能迭代优化，但你不一定能独立从零写出这些代码——这种情况算不算"会写代码"，要看你怎么定义。如果定义是"能独立产出代码"，那你不算；如果定义是"能让代码被做出来解决你的问题"，那你算。

这个定义之争本身没那么重要。重要的是它揭示了一个现实："会写代码"这件事的含义正在变化。以前"会写代码"意味着掌握语法、能动手写；现在"会写代码"越来越多地意味着能和 AI 协作、能把问题拆解清楚、能验收结果。这是一种新的能力，不比前者低级，只是不同。那些非程序员用 Codex 做出了真实可用的东西——网页、PPT、视频、数据分析流程——他们大概率没法独立写出这些代码，但他们解决了真实的问题。他们算不算"会写代码"不重要，他们用工具把事情做成了，这才是重点。

所以更健康的心态是：不要把"会不会写代码"当成一个身份标签去捍卫。你能不能解决问题、能不能让 AI 帮你做出想要的东西、能不能持续学习和迭代——这些比"算不算会写代码"重要得多。

防御退化的几个习惯也顺便说一下。头一条，重要的判断得自己做——Codex 给方案你审，Codex 给代码你看懂，Codex 给建议你权衡。再一条，保留"手动"的能力——重要的事偶尔自己动手做一遍，别让肌肉记忆完全退化。还可以把 Codex 当老师而不只是工具，让 Codex解释为什么这么写、为什么这么改。最后，把注意力放在方法论上——具体的命令、配置会变，但"怎么拆解任务、怎么验收结果、怎么和 AI 协作"这些方法论会跟着你一辈子。

工具让某些能力变得不那么必要，同时让另一些能力变得更值钱。你的方向感可能变差了，但你能去更多地方；你的代码手感可能退化了，但你能做更复杂的东西。这种演化是好是坏，看你怎么用 Codex。

## 98. 免费用户有没有什么省着用的方法

免费额度少，但有些方法能让你在有限额度里做更多事。

最直接的省法，是把最重的活分摊给免费的外部工具。比如做大量调研、查资料这种事，用免费的 ChatGPT 普通对话、Perplexity、Kimi 就够了，不必动 Codex。把它那点额度留给那些"必须 Codex 执行"的任务——改你本地文件、操控你电脑、跑你项目里的代码。

更划算的是接国产模型扩充额度。CC Switch 加上智谱、Kimi、DeepSeek，月费比 Plus 便宜一大截，有些还带免费额度。把不重要的任务切到国产模型上跑，省着用 OpenAI 的额度；国产模型在 Codex 上的兼容性虽不完美，日常任务基本够用。

任务本身的档位也要会挑。改个变量、问个小问题、做个小调整，这种就用最低推理强度，几秒就完，消耗最少，别什么事都开最高档，那是浪费。动手之前，最好先在计划模式里把路想清楚——规划阶段消耗少，方向对了执行阶段才不浪费；直接把大任务丢出去让 Codex乱跑，跑偏了烧的额度更多。

执行中要舍得喊停。发现 Codex 跑偏、打转、改不对，立刻叫停，别让 Codex无限试下去，每多一轮都是额度。日常还要养出一套"最省"的工作流：能用项目模式就别让 Codex扫整个项目，能指定文件就别让 Codex猜，能用 Skill 固化流程就别每次重新解释。这些小习惯叠起来，能省不少。最后一句容易被忽视：好结果跑出来，立刻复制出来、提交到 git。免费额度换来的成果不存下来就没了，别让 Codex停留在"只在对话里"。

免费额度的核心策略是"精打细算"。每个任务都想想有没有更省的办法，每次用都想想这次是不是真的需要 Codex。养成这种习惯，免费额度能撑很久；不养成这种习惯，给你多少都不够。

## 99. 什么时候该回到手动写代码

会用工具的标志是知道什么时候不用工具。手动写代码在以下场景仍然有不可替代的价值。

比如你需要深度理解某段代码的时候。让 Codex 写完你直接拿去用，和你自己写完、弄懂每一行，是两回事。那些重要的、核心的、经常要改的代码，自己写一遍才能让你真正吃透它，这种理解在后续维护、调试、扩展的时候会救你。

再比如要做特别精细、特别讲究的活。Codex 擅长"做到差不多"，却不太擅长"做到极致"。一个性能极致优化的模块、一条算法的关键路径、一个对细节苛求的交互，自己动手更靠谱——给它 80 分容易，让 Codex给到 100 分很难。

涉及核心业务逻辑和关键安全的地方更要自己上手。权限和安全的核心模块、支付的关键逻辑、生产数据库的操作，这些都不该全权交给 Codex。这些位置一旦出错代价太大，必须攥在自己手里。

还有你正在学习的时候。学习阶段，自己写、自己错、自己调，是建立理解的过程，让 Codex 替你写，你学不到东西。这时候 Codex 更适合当陪练——用来解释、答疑、对比方案，而不是替你交作业。

最后一种情况，是 Codex 反复做不对的事。如果你试了好几次，它在某个特定任务上就是卡住，那多半是这个任务超出了它的能力，或者这件事需要一种它给不了的判断。这种时候自己动手往往更快，别和它较劲。

判断该不该手动，问自己：这事重要吗？重要到自己应该完全理解？Codex 做出来的东西我能验收吗？Codex 做的够不够好？如果答案是"重要、应该理解、我能验收但不够好"，自己动手。

手动写代码谈不上落后，反倒是一种该被保留的能力。工具再强，你的核心能力也不能完全外包给工具。有一句话值得记住：Codex 能提高效率，但不能替你做判断。判断的核心，恰恰是你通过手动做事积累出来的。

## 100. 用了一阵之后最该复盘哪件事，长期最大的收获是什么

最该复盘的是：你能不能解释 Codex 给你做的每一个东西。

这句话听着简单，做起来会暴露很多问题。很多人用了一阵 Codex，做了不少事，但你问 Codex"这段代码为什么这么写"，Codex 答不上来；"这个功能是怎么实现的"，Codex 答不上来；"为什么用这个方案不用那个"，Codex 答不上来。如果答不上来，说明你在"外包"而不是"协作"，你在被工具推着走，而不是驾驭工具。

复盘的具体做法可以这样展开。挑你最近做的几个项目逐个走一遍，每个项目问自己：Codex 做了什么？为什么这么做？有没有更好的做法？如果 Codex 没做对，问题出在哪？你能回答这几个问题，说明你理解了；答不上来，说明你只是接受了。

顺便把有效的任务描述方式沉淀下来。哪些提法效果好？哪些会跑偏？形成你自己的"任务描述模板"，好用的 Prompt 不要每次重新写。用了一阵，你应该也更清楚哪些规矩需要明确、哪些流程需要固化——把这些更新到 AGENTS.md 和 Skill 里，让 Codex 的表现更稳定。还有你踩过的坑，哪些翻车重复出现、根因是什么、怎么避免，记录下来，下次别再踩同样的坑。尝试、失败、记录失败、不再重复同样的错误，这是 AI 辅助工作中最重要的习惯。额度使用也顺便评估一下，哪些花得值、哪些是浪费，据此调整后续用法。

复盘的核心目的是从"被工具推着走"变成"驾驭工具"。那些把 Codex 用得特别好的人，差别往往不在用得更久，倒在复盘得更勤上——他们知道每次为什么成功、为什么失败、下次怎么调整。这种持续迭代的能力，比任何具体技巧都重要。

而长期下来最大的收获，其实也就是这种持续复盘长出来的能力——你和 AI 协作把事情做出来的能力。效率会提升，东西也能做出来，但这些是表象。真正值钱的是几层更基础的能力。比如拆解问题——要让 AI 听懂你的需求，你得先把模糊的想法拆成清晰的步骤，这种拆解能力反过来让你自己思考问题也更清楚。再比如描述需求——把脑子里的东西用语言精确地表达出来，让另一个"智能"能照着做。还有验收结果——判断一个结果对不对、好不好、是不是真的解决了问题，这种能力让你在面对 AI 输出的时候不被带着走。最后是迭代优化，第一次做不对，调整需求、调整思路、再来一次。

这些能力合起来，就是让你从"动手干活的人"变成"提需求、验收结果、做决策的人"。这种转变谈不上让你变懒，反倒让你升了一层——你不再被重复性劳动困住，精力可以放在更有价值的判断和创造上。

还有一个更隐性的收获：你建立了一种对 AI 的"健康信任"。你既不盲目相信 AI（知道 AI 会跑偏、会顺手改错、会给半成品），也不盲目排斥 AI（知道 AI 能做很多事、能提效、能解决真实问题）。你清楚地知道 AI 的边界在哪、你的边界在哪，在合适的场景用 AI、在不合适的场景不用。这种判断力，是用任何教程都学不到的，只能靠真正用一阵、踩几次坑、复盘几轮才能长出来。

有一句话特别值得作为收尾：**当你把 Codex 当成一个可以配置、可以持续改进的队友来对待，它的效果最好**。这话反过来也成立：当你把 Codex 当成队友，你自己也会成为一个更好的"使用者"——更会提需求、更会验收、更会协作、更会判断。这种相互的成长，是长期用下来最值钱的收获。

工具会换，模型会变，但和 AI 协作把事情做出来的能力，会跟着你很久。这是用 Codex 这件事，能给你的最大的东西。
