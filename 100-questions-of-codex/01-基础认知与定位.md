# 一、基础认知与定位

## 1. Codex 到底是个什么东西

很多人第一次听说 Codex，以为就是 ChatGPT 换了个名字，专门用来写代码。这个理解差得有点远。

Codex 是 OpenAI 做的 AI agent，核心能力是"进入一个项目现场，把一件明确的活从头干到尾，交一个能验收的结果"。你给 Codex 一个任务，Codex 会自己读你的文件、自己改、自己跑、自己看报错、再改一轮，跑通了把结果拿给你看。整个过程更像在看一个远端的同事在你项目里操作，会思考、会试错、会回头修正。

和普通对话最大的差别在"动不动手"。普通对话你说一句、它回一段代码，你拿去自己复制、自己跑、自己改，AI 只动嘴。Codex 会真的执行，Codex 给你的是一个跑过、验证过的结果，一段能直接用的产物。

这一点很关键，因为后面所有关于"为什么慢""为什么烧额度""为什么有时会跑偏"的疑问，根子都在这里——Codex 在真的干活，不是在猜。

理解了"动手"这两个字，Codex 是什么就清楚了：Codex 是一个会进入项目现场干活的 AI 助手，干的是落地的事。

## 2. Codex 有哪几种形态，普通人该用哪个

这是新手最容易绕晕的地方，因为 Codex 现在同时存在四种形态，定位完全不同。

桌面版 Codex 是 OpenAI 官方做的图形客户端。最早只有 macOS 版，2026 年 3 月 4 日 Windows 版上线，现在两个平台都能装。打开是一个完整的应用窗口，左边任务列表、中间对话、右边展示结果。Codex 能读你电脑上的本地文件、能操控你电脑上的应用、内置一个浏览器。这是功能最全、对普通人最友好的一种。

CLI 命令行版跑在终端里，没有图形界面。你在哪个文件夹敲 codex，Codex 就在哪个文件夹干活。懂命令行的人喜欢它，因为轻、快、和 git、shell 配合顺。

IDE 插件装在 VS Code、Cursor 这些编辑器里，给程序员在写代码的时候顺手用。

Codex Web / Cloud 跑在云端，专门处理 GitHub 上的项目，改完直接给你开 PR。Codex 跑在 OpenAI 的服务器上，你电脑关了 Codex 照样干活。

四种形态里，普通人、入门用户、不写代码的人，该选桌面版。桌面版有图形界面，不用敲命令，功能也最全——后面这本书的所有内容，默认说的都是桌面版 Codex，除非专门提到别的形态。

## 3. Codex 和我平时在 ChatGPT 里问代码有啥不一样

差别不在聪明程度，在动不动手。

你在普通对话框里问"这段代码为什么报错"，模型读完上下文给你一段分析，再贴一段修改后的代码。对不对、跑不跑得通，全靠你自己再去验证。模型自己并不知道 Codex 给的代码执行起来会怎样，因为 Codex 压根没执行过。

Codex 走的是另一条路。Codex 接到任务后，会先把你说的需求翻译成一串动作：读哪些文件、改哪里、装什么依赖、跑什么测试。然后一步一执行，每一步都看结果回来，看到底成了没。成了往下走，没成就回头再改。这个循环 Codex 可能要走好几轮。

带来的差别很具体：普通对话里你常遇到"它给的代码看着对，一粘贴进去就报错"；Codex 给你的东西，至少在 Codex 那个环境里是跑通过的。Codex 还会顺手把牵连到的别的文件一起改了，普通对话做不到这种连贯改动。

代价也有。普通对话几秒就回，Codex 要搭环境、要试错，一个任务跑几分钟到几十分钟都正常。你换来的是结果的可信度，用的是等待时间。

## 4. 桌面版 Codex 和命令行版 Codex 有什么区别

它俩背后是同一套能力，但用法和门槛差很大。

命令行版没有界面，你打开终端，敲命令和 Codex 交互。好处是轻、快、和 git、shell 脚本、自动化流程配合得很顺，懂命令行的程序员用起来如鱼得水。坏处是门槛高，你得会开终端、会敲命令、看得懂吐出来的日志，完全不碰命令行的人会很痛苦。

桌面版是图形界面，打开就是一个完整的应用窗口。左边能看到所有任务、中间打字对话、右边直接看 Codex 生成的文件、网页、代码改动。Codex 能读本地文件夹、能操控你电脑上的应用、内置浏览器可以直接看效果。所有在命令行里要敲的命令，在桌面版里基本都能点出来或者直接说出来。

对普通人和入门用户来说，桌面版几乎在每一个维度都更友好：不用记命令、看得见正在发生什么、结果直接摆在眼前。桌面版比命令行版好上手得多，符合普通人的使用习惯。

所以结论很简单：你会敲命令行、喜欢终端那种感觉，CLI 适合你；你不是程序员、或者只是不想折腾命令行，直接用桌面版。

## 5. 没有代码仓库能不能用 Codex

能，而且这是 Codex 相比云端版最大的优势之一。

云端版 Codex 最自然的用法是接一个 GitHub 仓库，授权之后 Codex 拉代码、改完开 PR。没有 GitHub 项目，云端版用起来就别扭，因为云端版本来就是围着仓库转的。

Codex 完全不依赖 GitHub。Codex 的"项目"概念，就是你电脑上的一个普通文件夹。你新建一个项目的时候，让 Codex 选一个文件夹，选完之后这个文件夹里的所有文件，Codex 都能读、能写、能改。文件夹里有没有代码、是不是 git 仓库、要不要传到 GitHub，都不影响 Codex 干活。

这意味着你可以拿 Codex 做各种和"代码仓库"八竿子打不着的事。给 Codex 一个放着体检报告 PDF 的文件夹，让 Codex 帮你分析；给 Codex 一个放着素材的文件夹，让 Codex 帮你剪视频；给 Codex 一个空文件夹，让 Codex 从零帮你做一个网页。这些都不需要 GitHub。

唯一要注意的是选哪个文件夹。千万不要选桌面、下载、系统目录这种乱七八糟的地方，也别选中文路径或带空格的路径。专门建一个干净的工作文件夹，把它当作你的专属工作区。

## 6. 用 Codex 要不要先买会员

要看你想用得多深。免费额度能让你体验，但干正事基本得上会员。

免费账号给的额度相当克制，大致就是让你感受一下"哦 agent 是这么回事"，跑一两个稍微像样的任务就见底了。真要把 Codex 当成日常工具，绕不开 ChatGPT Plus 这道坎，每月二十刀。

Plus 给的是一个 5 小时滚动窗口里的额度，窗口里大概能跑相当于几十分钟推理的活，小任务够用，大任务几个就到顶。除了 5 小时窗口还有周限额，连着几天猛用会撞顶。Pro 档更贵但额度大得多，重度用户才需要考虑。

还有一个对中国用户很实际的选项：Codex 不一定要用 OpenAI 官方额度。Codex 支持接入国产模型，通过一个叫 CC Switch 的工具，可以接智谱 GLM、Kimi、DeepSeek 这些国内厂商，价格便宜很多，还不用梯子。很多人日常就是挂着国产模型在用。

所以门槛其实是这样的：只想试水，免费账号够你玩一圈；想正经用，要么买 Plus，要么走国产模型接入。完全不花钱、还想持续干活，这条路基本走不通。

## 7. Codex 能动我电脑上的其他文件吗

能动，但有边界，边界由你来定。

Codex 有一个叫 Sandbox（沙箱）的权限系统，分三档。第一档只读，Codex 只能看你项目里的文件，不改东西。第二档是项目可读写，Codex 能改你当前项目文件夹里的文件，但出不去这个文件夹。第三档是完全放开，整个电脑 Codex 都能碰。日常推荐用第二档，第三档新手别碰。

这意味着什么？只要你把项目限定在一个专门的工作文件夹里，Codex 能改的就只是这个文件夹里的东西，你电脑其他地方的文件 Codex 动不了。你的微信、邮箱、系统配置、别的项目的代码，默认都在 Codex 的边界之外。

但 Codex 还有一个能力会突破这个边界，叫"操控电脑"（Computer Use）。打开这个功能后，Codex 能像人一样看你的屏幕、点按钮、操作你授权的桌面应用。这个能力很强，也更需要谨慎——涉及付款、账号、删除数据的操作，最好别让 Codex 碰，不用的敏感应用先关掉。

所以答案是：默认 Codex 被关在你给的项目文件夹里，想让 Codex 伸到手以外的应用，得你主动开权限、主动授权。开关始终在你手里。

## 8. Codex 算聊天工具还是编程工具

严格说算 agent，介于工具和工具人之间。形态上 Codex 套着聊天的外壳，干的却是落地的活。

聊天工具的特征是对完即止，你说一句 Codex 回一句，结束。编程工具的特征是你给 Codex 任务 Codex 给你产物。Codex 两头都沾一点，又两头都不完全是。

Codex 有聊天的外壳：你在对话框里和 Codex 说话，描述任务、给反馈、让 Codex 改。但 Codex 干的活更像工具：Codex 会真的去改文件、跑代码、操控应用、交付一个可验证的结果。中间 Codex 还自己思考、自己决策、自己试错，这部分又像在和一个同事合作。

OpenAI 自己的定位也越来越清晰：ChatGPT 是入口，Codex 是执行层。你通过对话把任务交进去，但任务真正落地是在 Codex 这个执行层完成的。OpenAI 内部甚至已经基本用 Codex 替代 ChatGPT 来做工程类工作。

对用户的实际影响是：别用聊天的期待去用 Codex。聊天你可以随便问、随时停、没压力。用 Codex 你得有任务意识，每一次对话都在消耗额度，每一次任务都得有验收标准，不然就是烧钱。

## 9. 为什么有人吹 Codex 有人骂它难用

这个问题没有标准答案，但有一个很关键的变量：用的人把 Codex 当什么用。

骂得最凶的，往往是把 Codex 当加强版 ChatGPT 用的人。他们丢一个模糊的需求进去，等着 Codex 像对话那样秒回一个完美结果。结果等了几分钟，拿回来一个改得七七八八的东西，要么漏了需求，要么改岔了，自然觉得难用。这类用户的期待和 Codex 实际的工作方式之间，差着一整个"agent 该怎么用"的认知。

吹得最响的，通常是那些会写需求、会拆任务、会盯过程的人。他们知道 agent 需要清晰的输入，知道大任务得拆开，知道跑偏了要及时叫停重来。同样一个 Codex，在他们手里能稳定产出可用的结果，自然觉得好用。

除了使用方式，Codex 本身也确实有版本波动。社区里经常能看到"最近变难用了""上周还好好的这周就跑不动"这类反馈，这背后是模型版本和系统更新的影响。GPT-5.5 上线之后那阵子，就有不少人反映长任务退化、容易跑偏。这种波动是真实存在的。

一半是用法问题，一半是产品本身还在调；会用的当宝，不会用的当坑，两边都没完全说错。

## 10. Codex 适合从零写还是改现成的

改现成的，比从零写稳得多。

这个结论可能反直觉，因为很多人觉得"从零写更自由，AI 想怎么发挥怎么发挥"。但 Codex 干活的方式决定了 Codex 更需要上下文，而不是更自由。

改现成代码的时候，项目结构、命名约定、依赖关系、已有的测试和配置文件，这些信息 Codex 都能读到。Codex 在改的时候有参照系，知道改哪里、怎么改才和现有代码风格一致，改完也能拿现有测试去验证。这种场景下 Codex 的输出质量很高，经常让你觉得"这改动我自己写也差不多就这样"。

从零写就完全是另一种局面。你只给一段需求描述，没有现成代码做锚点，Codex 要自己决定项目结构、技术选型、文件组织。每一个决定都可能和你心里想的不一样，而 Codex 又没法读你的心。结果经常是 Codex 给了一版完整的代码，你一看技术栈、目录结构、风格全是 Codex 自己定的，和你预期的差很远。

所以如果你要用 Codex 从零起项目，最好的办法是先手动搭一个最小骨架，哪怕只是空的目录结构加一个 README，让 Codex 在这个骨架上接着写。给 Codex 锚点，Codex 就稳；让 Codex 凭空发挥，Codex 就飘。

## 11. 普通人用 Codex 最容易高估哪一点

最容易高估的是 Codex 的"自主性"。

很多人带着一种期待打开 Codex：我把需求一丢，Codex 自己就能从头到尾干完，干完给我一个能用的东西。这种期待一般来自各种 demo 视频和宣传文案，里面展示的往往是任务跑通之后那个漂亮的结果，中间那些反复调整、叫停重来、补充说明的过程都被剪掉了。

真实的体验是这样的：Codex 确实能自主完成很多动作，但 Codex 完成得好不好，极度依赖你给 Codex 的输入有多清晰。一个模糊的需求丢进去，Codex 会按 Codex 自己的理解往下跑，跑出来的东西往往和你想的差一截。你以为 Codex"自主"了，其实 Codex 是在按 Codex 自己的猜测自主，而 Codex 的猜测经常不在你的频道上。

真正用得顺的人，都不会把 Codex 当成一个"丢完就不管"的黑盒。他们会把任务拆细、把验收标准讲清楚、跑偏了及时打断重来、跑完认真审一遍结果。他们信任 Codex 的执行能力，但不放任 Codex 自由发挥。

高估自主性带来的直接代价是额度。你丢一个模糊的大需求，Codex 在自己猜的过程中会跑很多轮、试很多方向，每多一轮都是真金白银。最后拿回来一个还不太对的东西，你既赔了额度又得返工。

## 12. 什么样的活不该交给 Codex

几种活，交过去基本是浪费时间或者烧钱。

需求你自己都说不清的，别丢给 Codex。如果你脑子里对要做的东西还是一团模糊，只有一个大概感觉，Codex 没法替你想清楚需求，Codex 只会按 Codex 自己那团模糊的理解往下跑，结果大概率不对，你还得多花好几轮去纠偏，还不如自己先想明白。

没法验证对错的活，Codex 的优势也用不上。Codex 强在 Codex 能执行、能验证，如果一件事情做完之后没有客观标准判断"成了还是没成"，那 Codex 的执行能力就用不上。比如"帮我想几个产品方向""帮我评估这个设计好不好"，这种主观判断类的事情，普通对话就够了。

强依赖你本地敏感环境或者真实账号的活，尤其是涉及钱的，风险太高。让 Codex 帮你操作付款、登录银行账号、改生产数据库，这些事出了岔子没法收。涉及付款、账号设置、删除数据的操作，最好别让 Codex 碰。

你自己完全看不懂、也无法验收的活，Codex 能给你一个结果，但如果你连这个结果对不对都判断不了，那这个结果对你没用。要么全盘接受（风险高），要么逐行核对（比自己干还累）。

## 13. 不懂编程能用 Codex 吗

能，而且这正是 Codex 相比 CLI 的最大意义之一。

命令行版 Codex 假设你会用终端、看得懂日志、知道怎么处理报错，对完全不碰代码的人门槛很高。Codex 把这些都包进了图形界面，你打字描述需求，Codex 执行，结果直接在右边窗口里展示。整个交互更像在和一个懂技术的同事说话，而不是在操作一个开发工具。

大量非程序员用 Codex 的真实场景已经跑通：做电商的用 Codex 剪视频、拆爆款；做内容的用 Codex 写文章、做封面、做 PPT；普通职员用 Codex 分析体检报告、清理磁盘、对账。这些都不需要懂编程，只要你把"想让 Codex 干什么"说清楚。

但"不懂编程能用"不等于"什么都不用懂"。你还是要懂几件事：怎么给 Codex 一个清晰的任务、怎么验收 Codex 的结果、什么时候该叫停、哪些操作有风险。这些属于用 agent 的常识，跟编程知识没关系。

一句话总结：Codex 把编程门槛降到了零，但把"会用 agent"的门槛留在了那里。前者 Codex 替你解决了，后者还得你自己跨。

## 14. 用 Codex 之前最该先搞清楚哪件事

先搞清楚你这个活，做完了"好不好"能不能自己验。

这句话听着简单，但它是用 Codex 顺不顺的分水岭。很多人用得不顺，根子都在这里：他们把任务丢出去，等结果回来，结果回来了却不知道好不好，只能凭感觉说"好像行吧"。这种用法下，Codex 给你的结果好坏全凭运气。

为什么这件事这么关键。Codex 的整个工作模式，是替你把"改—跑—看结果—再改"这个循环跑一遍，最后给你一个在 Codex 那个环境里通过的版本。Codex 能保证的是"在 Codex 那个环境里跑通了"，但不能保证"这个东西真的解决了你的问题"。这两者之间的差距，只能靠你来填。

你能在拿到结果之后清楚判断"这个对了"还是"这个还不对"，用 Codex 就很顺——对了直接用，不对就告诉 Codex 哪里不对让 Codex 再改。你判断不了，那就只能要么全盘接受，要么逐行核对，哪个都累。

所以用之前，先问自己：这事做完了，我怎么知道成了？有测试可以跑、有明确的预期行为、有可以对照的参考，这些都能帮你验。如果连你自己都不知道"成了"长什么样，先别急着开工，先把验收标准想清楚。

## 15. Codex 会取代我手动干活吗

短回答：会改变你干活的方式，不会让你什么都不用干。

Codex 擅长的是把一件明确的、可验证的活执行完。你给 Codex 清晰的输入，Codex 给你跑通的结果。这部分劳动——那些重复的、有套路的、需要反复试错的——确实会被 Codex 接走。用了之后很多事自己再不想手动干了。

但 Codex 取代不了的，是判断。哪个任务该交给 Codex、任务描述得清不清楚、结果对不对、要不要接受、下一步往哪走，这些判断全都压在你身上。Codex 越能干，你判断的权重反而越重，因为 Codex 的每一次执行都消耗额度、每一次改动都影响项目。

还有一个 Codex 取代不了的，是你对活本身的理解。你完全不懂一件事，把活丢给 Codex，Codex 给你一个结果你验收不了，那这个活对你还是没有真正完成。Codex 最舒服的用法，是你对这件事有足够的理解，让 Codex 替你把执行的部分扛走，你专心做判断和决策。

所以更现实的期待是：Codex 会让你从一个"动手干活的人"，慢慢变成一个"提需求、验收结果、做决策的人"。这个过程里，你干的活变少了，但你想得更多了。这算不算取代，看你从哪个角度想。
