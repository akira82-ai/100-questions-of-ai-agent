# 三、踩坑与反模式

## 36. 为什么 Codex 突然开始原地打转

这是社区里被吐槽最多的一种翻车，几乎每个人都遇到过。

现象是这样的：你给 Codex 一个任务，Codex 开始干，跑了会儿报错，Codex 自己改了一版再跑，又报错，再改，再跑……这么循环好几轮，每一轮的错误信息都差不多，但就是过不去。你看着 Codex 在那转圈，等了十几分钟还在原地，额度哗哗往下掉。

为什么会这样？因为 agent 遇到失败的时候，Codex 的策略是"换个思路再试"。但有时候 Codex 换的"新思路"其实和上一次差不多，或者 Codex 压根没意识到自己在重复。Codex 没有人类那种"我刚才已经试过这个了，这条路明显不通"的元认知。结果就是 Codex 卡在同一个死胡同里反复撞墙。

社区管这种现象叫"无限乱试"或者 endless loop。Reddit 上甚至有人开帖吐槽，说 Codex 突然变得"几乎不可用"，任务都变成"无尽的循环、漏掉的指令、半成品的修复"。

怎么办？**及时叫停**。遇到 Codex 打转的时候直接发这一句：

> 先停一下。请总结目前失败原因，不要继续修改文件。

让 Codex 停下来复盘，比你让 Codex 继续硬试有用得多。复盘完，要么换个完全不同的思路重来，要么缩小任务范围，要么你自己介入判断问题在哪。让 Codex 无限试下去，最后大概率还是不行，但额度肯定烧光。

养成一个习惯：盯着 Codex 的动作日志，发现连续两三轮都在做差不多的事、报差不多的错，立刻叫停，别等。

## 37. 任务跑了半小时最后给我个半成品

这种翻车比原地打转更隐蔽，也更气人。

现象是：任务跑得很顺，没有报错，没有打转，看起来一直在推进。跑了二十分钟、半小时，你满怀期待去验收，结果 Codex 给你的东西只做了一半——要么功能没全实现，要么某些地方明显没改完，要么代码里有半句注释没写完，像 Codex 中途被拔了电源。

为什么会这样？最常见的是上下文超长。长任务里 Codex 要读很多文件、改很多地方，对话历史越堆越长，长到一定程度，Codex 的注意力就分散了，后面的细节开始漏。另一个原因是任务边界没说清楚——Codex 理解的"完成"和你想的不一样，Codex 觉得"核心功能做完就算完了"，但你觉得"还得处理边界情况、写测试、加文档"。

最阴险的是还有一种：Codex 在中途悄悄"自以为完成了"。这是一个已知问题，Codex 有时候会在后台任务还在跑的时候，就丢给你一个"final answer"，把任务提前结束。你以为 Codex 跑了半小时，其实它可能在第 15 分钟就已经"认为"自己做完了，剩下时间都在干别的。

怎么避免？把验收标准写进任务描述里，明确说"完成的标准是 XXX，缺一样都不算完"，让 Codex 没法提前收工。长任务一定要拆成小段，每段跑完验收一次，别让 Codex 一口气跑半小时。还有，跑完之后别只看 Codex 文字总结里那句"已完成"，要打开 diff、看实际产物，到底完成到什么程度，自己确认。

## 38. 明明写清楚了 Codex 为什么还是漏指令

这是最让人崩溃的一种翻车：你把要求一条条列得清清楚楚，Codex 还是漏了几条。

你写"1. 改 A，2. 改 B，3. 改 C，4. 加测试，5. 不要动 D"。Codex 跑完，A 和 B 改了，C 漏了，测试没加，D 还顺手给你改了。你回去看你的描述，明明写得明明白白，Codex 就是不全做到。

为什么？因为 agent 处理长指令的时候，注意力不是均匀分布的。开头几条和结尾几条 Codex 记得牢，中间的容易漏。多条指令堆在一起，Codex 倾向于先抓"Codex 觉得最重要"的几条做，剩下的可能忽略。社区里反馈 Codex 突然变成 "endless loops, missed instructions, half-finished fixes"，漏指令就是这个 missed instructions。

还有一个原因：指令越多，Codex 越倾向于"先把能快速搞定的做完给你看"，剩下的打算回头再补——然后回头就忘了。Codex 没有故意漏，它的工作记忆就那么大，长列表里中间几条自然就被挤掉了。

怎么治这个毛病？编排上有点讲究：重要指令放开头和结尾，中间放次要的，别堆成一片。指令数量也要克制，一个任务四五条要求是上限，再多就拆成两个任务跑。最有用的是计划模式，让 Codex 先把要做的步骤列出来，你一眼就能看出有没有漏掉哪条，确认了再执行。跑完之后，别忘了拿你的原始要求逐条对一遍，发现漏的让 Codex 补，别默认接受。

最有效的还是计划模式。先让 Codex 出计划，计划里缺了你的某条要求，你立刻就能发现并补上，比等 Codex 跑完再发现漏了强十倍。

## 39. Codex 改着改着把好代码也改坏了

这种翻车叫"顺手重构"，是头号反模式。

现象是：你让 Codex 改 A 功能，Codex 改完 A，顺手觉得 B 那段代码"不够优雅"，给你重构了；重构完 B，又觉得 C 的命名不规范，给你改了命名；改完命名，引用 C 的地方全得跟着改，D、E、F 一连串全动了。你以为让 Codex 改一个地方，回头一看 Codex 改了二十几个文件，其中一半和你的任务根本没关系。

为什么 Codex 会这样？因为 agent 有一种"优化倾向"。Codex 看到代码，会本能地觉得"这里可以更好"，然后就去改了。Codex 不区分"你让我改的"和"我顺手想改的"，在 Codex 眼里都是改动。很多老项目本身就有 lint 或类型问题，Codex 觉得某些代码"不够优雅"就会顺手重构，可能带来重命名导致引用出错、改全局样式影响其他页面、升级依赖引发兼容问题，甚至删除 Codex 认为无用但实际是业务逻辑的代码。

后果很严重。重命名一个组件，所有引用这个组件的地方都得跟着改，改错一处就报错。改全局样式，可能让别的页面布局崩掉。最阴险的是删代码——Codex 觉得"这段没用"，其实是你的业务逻辑，删了你都不知道哪坏了。

怎么防御？把这段固定的话术写进任务描述或者 AGENTS.md 里：

> 本次只实现当前功能点。
> 不要顺手重构无关代码。
> 不要修改命名、目录结构、依赖版本和全局配置。
> 如果你发现代码可以优化，请先记录为建议，不要直接修改。

这段话能把绝大多数顺手重构挡住。再加一条更狠的铁律，专门针对删代码：

> 不要删除你没有创建的文件。遇到报错的代码，修复它，不要删掉它。

这条特别重要。Codex 遇到报错时有一种危险倾向：删掉报错的东西来消除报错，哪怕那东西跟当前任务无关。社区里有人反馈，Codex 在做一个小改动时，把一个无关的模块整个删了，理由是"它在报错"。把"禁止删除你没创建的文件"写进 AGENTS.md，能把这种最阴险的翻车挡住。

最后一道防线：每次跑完看 diff，发现 Codex 改了和任务无关的文件，立刻让 Codex 回退那部分改动，别默认接受。

## 40. 为什么简单需求 Codex 非要绕一大圈

你让 Codex"在这个页面加个按钮"，听上去多简单的事，Codex 跑完给你改了十几个文件，建了三个新模块，引入了两个新依赖，折腾了十五分钟。你看着 diff 满头问号：我就让你加个按钮，你这是干嘛？

这是 agent 的另一种常见毛病：过度工程化。Reddit 上经常能看到反馈，说 Codex 经常"过度复杂化简单的任务"，哪怕你给了详细的需求文档，Codex 还是把简单的分类、组织工作搞得绕来绕去。

为什么？根子在于 Codex 见过太多"好实践"了。训练数据里全是各种设计模式、最佳实践、架构思想，所以看到任何一个需求，Codex 本能地想"用更专业的方式来做"。加个按钮？Codex 觉得应该封装成可复用组件。做个简单的数据展示？Codex 觉得应该引入状态管理。Codex 没在故意添乱，它是真心觉得这样"更好"。

更麻烦的是，Codex 不会主动问你"要不要简单点"。它默认你要的是 production-ready 的方案，就照生产标准来做了。但很多时候你只想快速做个原型，或者改个小东西，根本用不着那么讲究。

怎么办？在任务描述里明确说清楚你的预期。常用的话术：

> 用最简单的方式实现，不要引入新依赖，不要新增模块，不要做超出本需求范围的封装。

或者：

> 这是一个原型/演示，不需要生产级的架构，能跑就行。

把"简单"这个要求显式说出来，能治大部分过度工程化。另外，看到 Codex 跑完给了一堆你意料之外的文件改动，先别急着接受，问 Codex"为什么需要这些改动，能不能简化"。多数情况下 Codex 能收敛到更小的改动范围。

## 41. 后台任务是不是在偷偷烧我额度

很可能是。这是 Plus 用户提前烧光额度的头号原因。

现象是这样的：你开了一个任务，跑了一会儿觉得无聊或者要去干别的，关掉浏览器或者切换到别的应用。你以为自己"没在用"，其实任务还在后台跑，每跑一轮都在消耗额度。等你过两小时回来，发现额度掉了一大块，任务可能还挂在半路。

这种"隐形消耗"现象确实存在。有报道说 Codex 的后台进程导致用户"比预期更快地撞到限制"，OpenAI 的工程负责人当时承认这是问题，后续在修。但截至现在，后台任务消耗额度这件事本身没变，变的是界面提示更明显了些。

更阴险的是另一种情况：你开了一个 thread，跑了任务，任务完成了，你以为这个 thread 就"停下"了。但其实如果你开了自动化或者某些持续运行的任务，可能在你看不见的地方继续消耗。

怎么避免？养成几个习惯就行。最关键的一条：不用的时候彻底退出 Codex，注意是真的退出，不是最小化——Windows 要去托盘图标右键选退出，Mac 要在菜单栏里退出。最小化到后台的 Codex 还在跑，额度还在掉。另外，长任务跑之前先看一眼额度，估算够不够撑到任务结束；平时定期查额度，发现掉得比预期快，就排查是不是有后台任务在偷跑。

这里头有个意识得扭过来：Codex 没有"打开才花钱，关掉就免费"这回事。只要 Codex 活着、有任务在跑，就在花钱。不用的时候彻底关掉，是最简单也最有效的省钱办法。

## 42. 为什么关掉窗口额度还在掉

这是 Q34「关掉 Codex 任务还在跑吗」的另一种痛：你以为自己关了，其实没关。

Codex 默认的行为是最小化到系统托盘（Windows）或者菜单栏（Mac），不是真正退出。你点窗口右上角的叉号，窗口消失了，你以为 Codex 关了，其实 Codex 还在后台活着。后台活着的 Codex，如果你之前跑了任务没结束，那个任务还在继续跑、继续消耗额度。

很多人就是这样烧钱的：开了个任务，等得不耐烦，叉掉窗口去干别的，以为"我都关了应该不烧了"。结果第二天发现额度没了，任务跑了个乱七八糟挂在半路。这种亏吃一次就够肉疼了。

区分"关窗口"和"退出 Codex"的办法。Windows 上，点叉号只是关窗口，要去右下角系统托盘找到 Codex 图标，右键选 Quit 或者退出，才是真的退出。Mac 上，点红点是关窗口，要在菜单栏里选"退出 Codex"，或者按 Command+Q，才是真的退出。养成一个判断习惯：每次想关的时候，问自己"我是关了窗口，还是退出了程序"，这两个动作差别巨大。

如果你确实想让长任务在后台继续跑，请回到 Q34 看本地模式和云端模式的区别——只有云端模式做得到"关掉电脑任务继续"，本地模式做不到。

## 43. 为什么我开的任务最后没结果就消失了

这种翻车特别让人火大：你开个任务，跑了一会儿，回头一看，任务没了，没有结果，没有报错，就像从来没存在过。

几种可能。最常见的是任务被意外终止——前面那个已知问题，Codex 在后台任务还在跑的时候丢给你一个"final answer"，把自己结束了。你以为 Codex 还在跑，其实它早就"认为"做完了，悄悄退出了。

也可能是 Codex 崩溃或者被系统杀掉了。本地任务依赖 Codex 进程，Codex 一挂任务就跟着没。如果你电脑内存紧张、系统在清理进程、或者 Codex 本身有 bug 崩了，正在跑的任务会一起消失，连中间结果都来不及保存。

还有一类是环境掉了：登录过期、网络断开、连不上服务器，任务可能直接中断，又没有恢复机制。最后就是额度跑光，跑到一半发现额度不够直接停——这种通常会提示一下，但有时候提示不明显，你以为任务"消失"了。

怎么减少这种事？重要的长任务用云端模式跑，不依赖本地 Codex 进程，Codex 崩了也不影响。任务跑完要立刻验收、立刻把结果存到项目文件夹，别让 Codex 停留在"只在对话里"的状态。登录和网络保持稳定，掉线立刻重连。另外养成看任务列表的习惯，发现任务莫名其妙没了，就去翻动作日志和系统通知，看有没有报错。

## 44. 同样的任务再跑一次结果完全不一样

这是 agent 的"特性"，不是 bug，但你得知道它存在，不然会被坑。

你今天让 Codex"做个登录页"，Codex 给了你一版 A。明天你把任务原封不动再发一遍，Codex 给了你完全不同的 B。A 和 B 都能跑，但风格、结构、用的库、甚至技术选型都不一样。你懵了：同一个任务，怎么差别这么大？

原因是 agent 的输出有随机性。Codex 每次生成的时候，内部有一定的"采样"成分，同样的输入会走到不同的生成路径。再加上模型版本可能更新、上下文可能略有不同、Codex 当次的"判断"可能不一样，结果就是同一个任务每次跑出来都不完全相同。

这个特性带来的麻烦很具体。比如你看到一版很好的结果，想"再跑一次看看有没有更好的"，结果第二版差很多，想回到第一版又回不去（除非你提前存了）。再比如协作场景，你和同事分别跑同一个任务，拿到的结果完全对不上，连"到底用哪版"都没法讨论。

应对办法也很直接。好的结果立刻保存，别想着"再跑一次说不定更好"——把成果复制到项目文件夹、提交到 git，定格下来。想对比多个方案，用多方案预览功能，让 Codex 一次性给你几个方案对比，比分别跑三次靠谱得多。对复现性要求高的任务，把任务描述写到极致详细，因为描述越模糊，每次跑差异越大；描述越精确，差异越小。

记住一条：agent 的输出是"一次性"的，跑出来的东西不保存就没了。别指望 Codex 能"原样再来一遍"。

## 45. Codex 给的方案为啥和我项目对不上

这种翻车经常发生在"贴代码"场景：你把一段代码贴给 Codex 让 Codex 改，Codex 改完给你一版，你高高兴兴贴回项目，一跑全是报错。

为什么？因为 Codex 看不到你项目的全貌。你贴给 Codex 的只是那一段代码，Codex 不知道这段代码周围还有什么、依赖什么、调用了什么。Codex 改的时候，只能按 Codex 对"一般代码"的理解来改，跟你项目里这段代码的实际上下文对不上。结果就是 Codex 改的地方和你项目里的别处打架，引用错、类型错、依赖找不到，全是这种问题。

如果你不用项目模式，只是贴代码让 Codex 改，Codex 验证的那个环境和你的真实项目隔了一层，结果的可信度打折扣。这就是为什么要强调用项目模式——让 Codex 进入你的项目目录，读到你所有的文件，Codex 改的时候才知道周围什么样。

怎么避免这种翻车？能用项目模式就用项目模式，让 Codex 读到你项目的全貌，别只让 Codex看你贴的那一段。如果不得不贴代码（比如项目不方便共享），那就把相关的依赖、import、上下文一起贴上，别只给一段孤零零的代码。Codex 改完之后，记得在你的项目里实际跑一遍——Codex 能给你"看起来对"的代码，但你项目里跑不跑得通，它自己并不知道（因为它没在你项目里跑过）。

最根本的还是项目模式。很多 Codex 任务失败，原因往往不在 Codex 不会写代码，而在 Codex 还没理解项目就被要求直接动手。让 Codex 先读项目，再动手，翻车率会大幅下降。

## 46. Codex 说自己改好了怎么一跑全是报错

这种翻车的根源是：Codex 跑通的环境，和你跑的环境，不是同一个环境。

Codex 在沙箱里跑通了，那个沙箱有 Codex 自己装的依赖、Codex 自己的运行环境、Codex 自己假设的项目结构。Codex"跑通"是在那个环境里跑通的。你拿到 Codex 的代码，放到你真实的项目里，你的环境可能不一样：依赖版本不同、操作系统不同、配置不同、别的代码和 Codex 的改动有冲突。结果就是 Codex 说"跑通了"，你一跑全是错。

有真实的吐槽：有人让 Codex 干个简单的活，Codex 连代码库用什么语言都没识别对，一个简单任务直接失败。这种就是典型的"Codex 没真正进到你环境里就开干"。

更麻烦的是，Codex 有时候根本没真的跑，只是"觉得"自己跑通了。Codex 生成完代码，看着代码觉得"这样写应该没问题"，然后告诉你"已完成"。你没验证，拿去跑，发现根本没跑过。

怎么治？关键就一条：让 Codex 在你项目里跑，别让 Codex在自己的沙箱里自说自话。用项目模式，让 Codex 真的进到你项目目录，这样 Codex 跑通的就是你项目的环境。跑完之后你自己再跑一遍，别全信 Codex 说的"跑通了"。如果报错，就把真实报错信息直接发回给 Codex，让 Codex看到现场，别让 Codex自己脑补。还有，如果 Codex 反复在同一类报错上栽跟头，多半是任务描述不够清楚，或者任务本身超出了它的能力，那就考虑拆小或者换思路。

## 47. 哪些信号说明这个任务该手动叫停

及时叫停是省额度和防翻车的核心技能，比会启动任务更重要。

几个明确的信号，看到任何一个都该停。最典型的就是原地打转——连续几轮报差不多的错、做差不多的尝试，没有实质进展，这种等再久也没用，立刻叫停。

文件数量也能透露问题。你让 Codex 改一个功能，它改了二十几个文件，明显是在顺手重构或者跑偏了，看到 diff 文件数飙升，先暂停看看它到底在干什么。更危险的是 Codex 开始删大量代码——除非你明确让 Codex重构或者清理，否则任何删代码的举动都要警惕，Codex 觉得"没用"的代码，往往是你的业务逻辑。

还有几个信号同样要紧。Codex 引入了新依赖、改了 package.json、动了配置文件，你都得问一句"为什么需要这个"。任务跑了很久还没完成、也没明确进展，本来几分钟该完的事拖到十几分钟还在继续，多半是陷入循环或者卡住了。最后还有一种尤其要命：Codex 的文字总结和实际 diff 对不上。它嘴上说"修改了登录逻辑"，diff 里压根没动登录相关的文件——这种说一套做一套的情况，要立刻叫停问清楚。

遇到任何不对劲都可以发这一句：

> 先停一下。请总结目前做了什么，遇到了什么问题，不要继续修改文件。

让 Codex 停下来汇报，比你让 Codex 继续试更省时间和额度。记住一个原则：宁可多叫停几次重来，也别让 Codex 一个跑偏的任务跑半小时。前者浪费几分钟，后者浪费十几分钟和一大块额度。

## 48. 任务失败了我该重试还是改需求

这两种应对方式差别很大，选错了会越搞越糟。

先说什么时候该重试。如果失败是偶发性的——比如网络断了、Codex 崩了、某个临时报错——直接重试就行，原任务不用改。这种是环境或者运气问题，再来一次大概率能成。

什么时候该改需求。如果失败是"Codex 试了好几次都做不到"，那就别再当运气问题看了，根子在任务本身：可能是任务太大、描述太模糊，或者超出了 Codex 的能力范围。这种情况下直接重试，多半还是失败，白烧额度。正确的做法是停下来，分析为什么失败，然后调整任务。

调整任务常见就几个方向。最常用的是拆小，把大任务拆成几个小步骤，一步一步来，每步成功率高。再把需求说清楚，把模糊的地方具体化、把验收标准明确化。范围也要收紧，别让 Codex"重构整个模块"，让 Codex"只改这一个函数"。如果 Codex 反复在同一种解法上失败，可能是这条思路本身不适合，那就换一个完全不同的方向。

有一个特别重要的习惯：失败的时候让 Codex 先总结原因，而不是继续试。发一句"先停一下，总结目前失败原因，不要继续修改文件"，让 Codex 给你一个失败分析。这个分析能帮你判断是该重试还是该改需求——如果 Codex 说的是"网络问题""临时错误"，重试；如果 Codex 说的是"这个需求涉及的模块太复杂""依赖的 API 不存在"，那必须改需求。

最忌讳的是"无脑重试"。看到失败就直接点重新运行，不改任何东西，结果大概率还是失败，连续失败几次额度就烧光了。每次失败都先停一下想想为什么，再决定下一步。

## 49. 为什么大任务拆开反而更容易废

这个反直觉的现象坑过很多人：你把一个大任务"合理地"拆成几个小任务，本以为是稳妥的做法，结果每个小任务都跑得不顺，连起来更乱，还不如一开始让 Codex 整体干。

为什么拆了反而废？最常见的是拆错了边界。你以为 A、B、C 是三个独立的小任务，其实它们之间有耦合。A 改完影响了 B 依赖的东西，B 跑的时候还基于 A 改之前的状态，结果 B 的改动和 A 冲突——这种"拆了但没拆干净"比不拆还糟。

另一个坑是上下文丢失。大任务一次性跑，Codex 从头到尾待在同一个上下文里，记得前面做了什么；拆成小任务分别跑，每个任务都是"新开始"，Codex 不知道前面干过什么，可能重复改动，也可能和前面矛盾。再加上拆得太碎，Codex 做每个小任务时只盯着这一小块，看不到整体，结果每个小块单独看还行，拼起来风格不统一、结构不一致、甚至逻辑冲突。

怎么拆才对？关键原则是按耦合程度拆，别按"功能模块"拆，紧密耦合的部分放一个任务里，不要硬拆。拆完之后还要给 Codex 整体上下文：每个小任务开始前告诉它"前面已经做了什么、现在要做的是什么、后面还要做什么"，让 Codex有整体感。更稳妥的做法是用计划模式先规划整体，让 Codex 自己提出怎么拆，你看它的拆法合不合理，比你硬拆靠谱。最后记得每拆完一段就验收一段，发现不对立刻调整，别等三个小任务全跑完才发现整体乱了。

拆任务本身没错，错的是"机械拆分"。会拆的人能让复杂任务变简单，不会拆的人能让本来能干的任务变得干不成。

## 50. 怎么避免一个任务把整周额度拖垮

这是 Plus 用户必须学会的自我保护，不然一个跑偏的任务能让你一周都没法用。

危险是怎么发生的。你开了一个看起来不算大的任务，Codex 跑着跑着陷入循环，每多一轮都在烧额度；你忘了盯着 Codex，去做别的事，半小时回来发现额度掉了一大块；你想止损但又不甘心，再让 Codex 试一次，又烧一波；最后任务没成，额度也快没了，剩下几天只能干瞪眼。

避免的办法，核心是给任务设一道止损线。开任务前先预估成本，看任务复杂度，估算大概要跑几轮、每轮消耗多少，心里有个数。再给任务设个时间上限，比如跑超过 10 分钟还没结果就强制叫停，不要无脑等。盯动作日志也很关键，发现 Codex 在打转、在试错、在反复失败，立刻停，别指望它自己想明白。

另一头是"拆和省"。大任务一定要拆小，分多次跑，每次跑完看额度还剩多少，够再跑下一段，不够就先等。小任务用低推理强度，省额度；动手之前用计划模式先规划，规划阶段消耗少，规划对了，执行阶段才不浪费。

第三头是"监控"。经常查额度，发现掉得快立刻排查；不用的时候彻底退出 Codex，不让后台任务偷跑；后台同时只跑一个重任务，多个叠加起来消耗是翻倍的。

最根本的心态调整：不要把 Codex 当成"丢进去等结果"的工具，要当成"需要你看着的工具"。Codex 跑任务的时候，你最好在旁边盯着，发现不对立刻介入。完全放手的任务，要么是云端模式（你不怕 Codex 跑偏），要么是你已经验证过 Codex 能稳定完成的 routine 任务。新任务、复杂任务，一定要盯。

## 51. 用 Codex 最容易翻车的人都是哪几条没做到

把上面这些踩坑读完，你会发现真正救命的招其实就那么几条。社区里翻车最多的人，基本都是这几条底线没守住。整理成一份最低标准清单，照着做能挡住八成以上的灾难。

最要紧的一条是：**每次让 Codex 动手之前，先给你的项目存一个版本**。如果你用 Git，就是先提交一次再让 Codex 干活；如果你不用 Git，至少把整个项目文件夹复制一份留底。Codex 改崩了你还能 reset、还能 checkout 回来，Codex 删了关键代码你还能从备份里捞回来。没有这层"撤销键"，让 Codex 改项目等于不系安全带上高速。

再有一条是：**稍微大点的改动，都用 Worktree 模式跑**。Worktree 相当于给 Codex 一本独立的草稿本，让 Codex 在副本上大胆改，你的主项目纹丝不动。改坏了直接丢掉副本，零风险。重构、加新功能、不确定结果的尝试，都该用 Worktree。

还有一条是：**在 AGENTS.md 里写死"禁止删除你没创建的文件"**。Codex 遇到报错的代码，倾向删掉报错的东西来消除报错——哪怕那东西跟当前任务无关。这条铁律把这种最阴险的行为挡住。

再就是：**别让 Codex 跑无人监督的大动作**。改名、清理、大规模重构、迁移——这类任务风险最高，最容易被 Codex 的"优化倾向"带跑，必须全程盯着。

最后：**小步快跑，每步都看 diff**。把大任务拆成小的、边界清楚的变更，每一步跑完立刻看 diff。这个习惯能把绝大多数翻车挡在合并之前。

这几条没有任何一条是高深技巧，全是底线操作。社区里那些把 Codex 用得风生水起的人，并没有掌握什么秘密武器，他们只是把这几条做到了骨子里。反过来，社区里翻车最惨的案例，几乎都能从这几条里找到违反的地方。
