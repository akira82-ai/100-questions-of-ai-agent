# 100-questions-of-ai-agent Reference

这个文件用于给不同 Agent 提供统一的项目约束、写作口径和执行方式。

## 1. 项目定位

- 这是一个按主题持续生产 `100 questions` 文档/书稿的内容仓库。
- 当前主题示例是 `loop engineering`。
- 每个主题最终会拆成按章独立的 `.md` 文件，而不是所有内容都堆在一个总文件里。

## 2. 当前文件组织方式

- 主题目录示例：`100-questions-of-loop-engineering/`
- 目录内直接按章节拆成独立文件，不再保留总题纲文件，例如：
  - `01-基础认知与定位.md`
  - `02-loop的组成与运行机制.md`
  - `03-踩坑与反模式.md`
  - `04-验证质量与成本控制.md`
  - `05-工程落地与真实工作流.md`
  - `06-边界争议与未来判断.md`

## 3. 写作任务的默认流程

- 不要直接凭印象作答。
- 先按 `airay-100-questions` 技能的思路补资料和搜索，再整理成答案。
- 搜索时优先混合多种来源，不要只盯官方。
- 需要同时参考：
  - 官方文档
  - 官方博客 / 工程文章
  - release notes / changelog（如果是产品或工具主题）
  - GitHub issues / discussions
  - 社区讨论
  - 长文经验帖 / 案例分析

## 4. 单题正文风格

- 单题正文控制在 `300-500` 字，`500` 字以内优先。
- 写法要像书正文，不要像提纲解释，也不要像资料综述。
- 语言要口语、自然、顺着讲。
- 不要写得过度工整、过度对仗、过度“方法论腔”。
- 回答结构默认可参考：
  - 先点出现象变化
  - 再解释原因
  - 再落到今天为什么会这样
  - 最后用一句短判断收住
- 可以保留少量判断句，但不要堆概念，不要装腔，不要写成行业黑话堆砌。

## 5. 硬性风格限制

- 严禁使用：`不是……而是……`
- 这是硬约束，不是“少用”。
- 如果发现已有内容里有这个句式，后续继续写作时要顺手改掉。

## 6. 需要避免的 AI 味道

- 避免那种很整齐、很严谨、很模板化的表达。
- 避免高频使用这些句式：
  - `本质上`
  - `更准确地说`
  - 其他一眼很像 AI 总结腔的句子
- 不要写成“标准答案口吻”。
- 要更像一个真正在讲这件事的人，而不是在输出培训材料。

## 7. 参考资料规范

- 全书在末尾单独设一个「参考资料」附录文件（例如 `07-参考资料.md`），统一汇总来源，成书更干净。
- 正文不再每题单独附来源，只讲内容；所有来源集中到附录里。
- 附录按来源域分组（官方文档、官方博客 / 工程文章、release notes / changelog、GitHub issues / discussions、社区讨论、长文经验帖 / 案例分析等）。
- 组内按被引用次数从多到少排列，方便顺着自己关心的方向往回查原始材料。
- 写作时每题仍要尽量覆盖 `2-4` 条来源，只是最终汇集进附录，不在正文里堆。
- 全书层面尽量让来源分散，避免 100 个问题都反复只依赖同一篇材料。
- 来源结构尽量混合，避免单一官方源统治全书。

## 8. 章节写作方式

- 默认按章推进，不要乱跳。
- 一章写完，再进入下一章。
- 当前已经确认这种节奏是合适的。

## 9. 当前用户已确认喜欢的风格

- 口语化
- 像书正文
- 逻辑清楚，但不要摆出“专家腔”
- 结尾要收住，不要拖泥带水
- 来源要丰富多彩

## 10. 事实审查方法（成稿后必做）

一本 100 问写完以后，必须对照参考资料做一次事实审查。核心原则：**先抽取硬声明清单，再针对性核实**，而不是通读资料后再审。实测下来，这种做法比通读式快 5-10 倍，且漏判更少。

### 10.1 为什么不能通读后再审

- 书稿约 2-3 万字，其中 90% 以上是判断性、建议性、收束性的"软内容"，根本不需要去资料里查证。
- 如果逐句拿去资料里搜，90% 的搜索是白搜的，agent 还容易超时被取消。
- 真正需要核实的"硬声明"通常只有 30-50 条，集中处理这些就够了。

### 10.2 第一步：从书稿里抽取硬声明

读正文时，只标记以下几类内容，软内容一律跳过：

- **具体数字**：专家数、积分数、倍数、版本号、上下文窗口大小、文件大小、日期
- **具体名称**：按钮名、菜单名、文件路径、错误码、文件名、技能名
- **完整清单**：连接器列表、模型列表、支持平台、技能清单
- **因果归因**：类似"官方建议先 X 再 Y""文档把 X 定义为 Y"——这类要查官方是否真这么说过
- **机制描述**：某功能怎么工作、权限怎么分级、数据存到哪里、自动化触发条件

判断性、建议性、收束性的句子（如"任务太宽容易翻车""先 Plan 再 Craft 更稳"）不算硬声明，直接放过。

### 10.3 第二步：逐条去资料里核实

- 用 `grep` 或搜索，按硬声明的关键词去资料里定位反证或支撑。
- **按事实类型批量查，不要按章节查**。一次把所有"路径相关"的声明一起捞，再查所有"连接器相关"的，命中率高，资料只翻一次。

### 10.4 双源交叉，指定各源的权威边界

这是最容易踩坑的一步。资料通常分两类，必须分清谁说了算：

- **官方文档**（docs、HTML 镜像、release notes）：数字、清单、路径、错误码、UI 名称、机制定义的**权威源**。
- **社区文章 / 经验帖**：使用经验、踩坑、边界判断的**权威源**。

两者冲突时，**硬事实以官方为准**，**经验判断以社区为准**。

> 踩坑记录：WorkBuddy 那本书第一次审查时，只读了社区文章合集（资料.txt），没查官方 HTML 文档，把三处真实信息（工作空间路径、错误码编号、Google 连接器）误判为"编造"并删改。后来用官方 HTML 补查，三处全部回退。教训：**审查必须双源交叉，单源容易误判**。

### 10.5 什么时候用 sub-agent，什么时候不用

- **这个规模（2-3 万字书稿 + 20 万字资料），主 agent 直接做 + grep 更快更稳**。总信息量不超出单 agent 上下文，派 sub-agent 的调度成本（启动、上下文重建、结果汇总）反而大于收益。
- 真正适合 sub-agent 的场景：
  - 书稿或资料量再翻几倍，超出单 agent 上下文
  - 需要**不同专业视角**并行（如同时做技术审查 + 文风审查）
  - 子任务**完全独立**且可并行
- 如果一定要用 sub-agent 审查，按以下方式切片：
  - 按**事实类型**切片（如"你只查所有模型相关声明""你只查所有连接器相关声明"），不要按章节切片
  - 给明确的**硬声明定义**（见 10.2），让它只查硬内容，软内容跳过
  - 给**固定输出格式**（题号 / 原文 / 反证 / 建议 四段式）
  - **指明权威源边界**（见 10.4），避免它只信一个源

### 10.6 输出格式

审查报告统一用这个结构：

```
## 审查总结
- 共审查 X 题
- 硬错误 X 处、可疑 X 处、风格建议 X 处
- 整体事实准确度评价（一句话）

## 硬错误清单（必须改）
每条：题号 + 原文 + 问题 + 资料反证（标来源）+ 建议

## 可疑清单（建议核实）
同上格式

## 整体评价
哪些章最稳、哪些风险高。
```

## 11. 给其他 Agent 的建议

- 开始写之前，先读这个 `reference.md`。
- 如果要继续当前主题，也先读对应章节文件，避免风格断层。
- 成稿后做事实审查时，务必按第 10 节的方法走，尤其是双源交叉（10.4）。
- 如果要继续更新当前 Agent 的长期记忆，可以把本文件中的关键约束同步回记忆系统。
- 如果用户新增风格要求，优先更新这个文件，再继续写作。
