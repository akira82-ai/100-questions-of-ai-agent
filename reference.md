# reference

给 Agent 的项目约束。规则是硬约束，`>⚠️` 是触发该规则的历史踩坑。

---

## 一、项目定位

- 按主题持续生产 `100 questions` 书稿的内容仓库
- 每个主题拆成按章独立的 `.md`，不堆在一个总文件里
- 目录命名：`100-questions-of-<主题>/`，章节文件：`01-xxx.md` ~ `07-xxx.md`

---

## 二、写作流程

### 2.1 写作原则

- 不凭印象作答，先补资料再写
- 资料来源混合：官方文档 / changelog / GitHub issues / 社区讨论 / 经验帖

### 2.2 资料搜集工具

| 场景 | 工具 |
|------|------|
| 中文实操经验、踩坑复盘 | 搜索引擎（Bing / Google） |
| 跨子主题、需并行核实 | `deep-research` |
| 英文一手资料、官方文档 | `WebSearch` / `WebFetch` |

原则：**全网搜索实操经验踩坑复盘，硬事实看官方，争议点双源交叉**。

### 2.3 章节推进

- 按章顺序推进，不跳章
- 每章写完 → 跑第四章三个检查 → 再进下一章

---

## 三、写作风格

### 3.1 正文基调

- 单题 `300-500` 字，`500` 以内优先
- 像书正文，不像提纲解释或资料综述
- 口语、自然、顺着讲
- 结构参考：现象 → 原因 → 落到当下 → 短判断收住

### 3.2 段落节奏

- 以"意群"为单位，一个意群讲清一个小命题
- 意群内部用逗号，不要用句号硬切复句
- 不要单句独占一段
- 长短段交错，有呼吸感

### 3.3 用词语气

- 允许「包括」「例如」等引导词
- 允许口语毛刺（「这事儿」），那是人味
- 禁止模板腔（「值得注意的是」「需要强调的是」）

### 3.4 硬性禁令

- **禁止** `不是……而是……`
- **禁止** 破折号 `——`，需要转折时拆成独立句
- **禁止** 正文露出信源（「根据 XX 分析」「深扒 N 份后发现」「Reddit 上有高赞帖」）
- **禁止** AI 总结腔（「本质上」「更准确地说」）
- **禁止** 短句：6～7 个字以内严禁作为一句话，而是改为逗号进行前后连接
- 发现已有内容违反以上禁令时，顺手改掉

## 四、每章质量检查（写完一章后必做）

### 4.1 禁用句式

```bash
grep -n "不是.*而是\|——" <本章文件>
```

命中即改，不等全书完成。

>⚠️ 飞书 CLI 第五、六章各残留 1 处，全书完成才发现。

### 4.2 命令名 / API 名核实

抽取本章所有反引号包裹的 `` `+xxx` ``、`` `lark-cli xxx` ``、环境变量名，逐个去 CHANGELOG / README grep 确认存在。虚构命令占 P0 错误 70%+。

>⚠️ 飞书 CLI 虚构 `task +tasks`、`contact +user-get-by-name`、`im +chats`、`LARK_CLI_TOKEN` 等 10+ 处。

### 4.3 数字核实

版本号、日期、数量类声明，打开原始资料数一遍，不凭印象。

>⚠️ 写「7 月前 8 天发了 4 个版本」，CHANGELOG 实数是 6 个。

---

## 五、事实审查（成稿后必做）

核心原则：**先抽取硬声明清单，再针对性核实**，不通读后再审。比通读式快 5-10 倍。

### 5.1 审查前准备

**资料穷尽**（必须先做）：

```bash
ls 资料/
```

列出所有文件，逐个标记已读/未读，写入审查报告。不能只读汇总版漏掉原始分册。

>⚠️ 飞书 CLI 7 份 Q&A 漏读 2 份，导致 Skills 数量和 token 有效期两个事实误判。

### 5.2 硬声明抽取

只标记以下几类，软内容（判断、建议、收束句）一律跳过：

- 具体数字（版本号、倍数、日期、文件大小）
- 具体名称（命令名、错误码、路径、技能名）
- 完整清单（连接器列表、模型列表）
- 因果归因（"官方建议先 X 再 Y"）
- 机制描述（功能怎么工作、权限怎么分级）

### 5.3 逐条核实

- 用 `grep` 按关键词去资料定位
- **按事实类型批量查，不按章节查**（一次捞所有"路径相关"，再捞所有"命令相关"）

### 5.4 权威层级（双源交叉）

| 层级 | 来源 | 权威领域 |
|------|------|---------|
| 硬权威 | 官方文档（docs、CHANGELOG、release notes） | 数字、命令名、路径、错误码、机制定义 |
| 基线权威 | 用户洞察报告（多人核实） | stars 数、业务域数、Skills 数、竞品对比 |
| 经验权威 | 社区文章 / Q&A / 经验帖 | 使用经验、踩坑、边界判断 |

冲突时：硬事实→官方，战略基线→洞察报告，经验判断→社区。

>⚠️ WorkBuddy 书只读社区不查官方，把三处真实信息误判为"编造"。飞书 CLI 书把洞察报告的"26 个 Skills"降格为普通数据点，误改成"20 多个"。

**数字争议**：不同来源不一致时，取洞察报告采集时点的数字，注明时间。不回退模糊表述。

**标"无法核实"前**：必须用 `grep -rn "关键词" 资料/` 在每个文件搜一遍。

>⚠️ token 有效期标了无法核实，漏掉的 Q&A 里有 AI 客服明确解答。

### 5.5 sub-agent 使用

- 2-3 万字书稿 + 20 万字资料：主 agent + grep 更快，不用 sub-agent
- 适用 sub-agent：资料量超上下文 / 需多专业视角并行 / 子任务完全独立
- 切片方式：按**事实类型**，不按章节

### 5.6 修复后 grep 验证

```bash
grep -rn "旧错误模式" <目录>   # 应 0 命中
grep -rn "新正确模式" <目录>   # 应全部命中
```

两轮通过才算修复完成。

### 5.7 审查报告格式

```
## 资料文件清单
已读：...
未读：...

## 审查总结
- 共审查 X 题，硬错误 X 处、可疑 X 处
- 整体准确度评价

## 硬错误清单（必须改）
题号 + 原文 + 问题 + 资料反证 + 建议

## 可疑清单（建议核实）
同上格式

## 整体评价
哪些章最稳、哪些风险高。
```

## 六、交付规范

### 6.1 字数口径

只算正文 6 章（01-06），不含总纲（00）、附录（07）、审查报告（08）。

```bash
wc -m 01-*.md 02-*.md 03-*.md 04-*.md 05-*.md 06-*.md
```

>⚠️ 飞书 CLI README 写 68000（`*.md`），实际正文 57000，差 11000。

### 6.2 章节标题

- **H1 必须带序号**：`# 第一章 · 认知颠覆`，不能只写 `# 认知颠覆`
- **H2/H3 产品名必须完整**：标题里写"飞书 CLI"不简写"CLI"（正文可简写）

>⚠️ docx 里"第一章 ·"被脚本剥成"· 认知颠覆"；17 处标题简写"CLI"需补全。

### 6.3 docx 生成

- 生成后**人工通读**，重点查形近字（铂书→飞书、沸书→飞书）
- 产物移入主题目录，命名 `关于 XX 的 100 个问题.docx`

### 6.4 README 更新

每本书完成后更新仓库根目录 README：

| 检查项 | 要求 |
|--------|------|
| 章节标题 | 对齐实际 H1，不写旧规划标题 |
| URL 编码 | 用 `urllib.parse.quote` 生成 |
| 字数 | 用正文口径（6.1） |
| 日期 | 写当天 |

>⚠️ 飞书 CLI README 章节名和实际文件名完全对不上，URL 404。

## 七、参考资料规范

- 末尾设 `07-参考资料.md`，集中汇总来源
- 按来源域分组（官方 / changelog / issues / 社区 / 经验帖）
- 组内按引用次数降序
- 每题覆盖 2-4 条来源，全书来源分散，不依赖单一材料

## 八、给 Agent 的 Checklist

写作阶段：
- [ ] 先读本文件
- [ ] 先读对应主题已有章节，避免风格断层
- [ ] 每章写完跑第四章三个检查

审查阶段：
- [ ] 资料穷尽（5.1）
- [ ] 双源交叉（5.4）
- [ ] 修复后 grep 验证（5.6）

交付阶段：
- [ ] 字数口径（6.1）
- [ ] 章节标题规范（6.2）
- [ ] docx 人工通读（6.3）
- [ ] README 更新（6.4）

用户新增风格要求时，优先更新本文件再继续写作。
