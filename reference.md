# 100-questions-of-ai-agent Reference

这个文件用于给不同 Agent 提供统一的项目约束、写作口径和执行方式。

## 一、项目定位与产出结构

### 1.1 项目定位

- 这是一个按主题持续生产 `100 questions` 文档/书稿的内容仓库。
- 当前主题示例是 `loop engineering`。
- 每个主题最终会拆成按章独立的 `.md` 文件，而不是所有内容都堆在一个总文件里。

### 1.2 文件组织方式

- 主题目录示例：`100-questions-of-loop-engineering/`
- 目录内直接按章节拆成独立文件，不再保留总题纲文件，例如：
  - `01-基础认知与定位.md`
  - `02-loop的组成与运行机制.md`
  - `03-踩坑与反模式.md`
  - `04-验证质量与成本控制.md`
  - `05-工程落地与真实工作流.md`
  - `06-边界争议与未来判断.md`

## 二、写作流程

### 2.1 默认流程

- 不要直接凭印象作答。
- 先按 `airay-100-questions` 技能的思路补资料和搜索，再整理成答案。
- 搜索时优先混合多种来源，不要只盯官方。
- 需要同时参考：
  - 官方文档
  - 官方博客 / 工程文章
  - release notes / changelog（如果是产品或工具主题）
  - GitHub issues / discussions
  - 社区讨论
  - 长文经验帖 / 案例分析

### 2.2 资料搜集工具（按场景选）

写一本书前和写每一章前，都要先做资料搜集。下面三个工具按场景搭配用，不要只依赖其中一个。

#### 2.2.1 `wechat-article-search`（公众号文章检索）

中文信息密度最高的来源，优先用。

- 技能路径：`wechat-article-search`（基于搜狗微信搜索）
- 脚本位置：`scripts/sogou_search.py`
- 适用场景：找中文圈的实操经验、行业判断、踩坑复盘、概念解读
- **必须带时间参数**，否则会返回过时旧文章：

```bash
# 最近 7 天
python scripts/sogou_search.py "关键词" --days 7

# 最近 24 小时
python scripts/sogou_search.py "关键词" --hours 24

# 指定时间范围
python scripts/sogou_search.py "关键词" --time-range 2026-01-01 2026-02-03

# 组合：30 篇 + 最近 30 天 + 获取原文
python scripts/sogou_search.py "关键词" --count 30 --days 30 --fetch-content --limit 5

# 导出 JSON
python scripts/sogou_search.py "关键词" -o result.json
```

常用参数：

| 参数 | 说明 |
|------|------|
| `--count` / `-c` | 返回数量（默认 10，最多 100） |
| `--days` / `-d` | 最近 N 天 |
| `--hours` | 最近 N 小时 |
| `--time-range` | 指定范围（YYYY-MM-DD YYYY-MM-DD） |
| `--fetch-content` / `-f` | 获取原文 |
| `--limit` / `-l` | 获取原文数量上限 |
| `--output` / `-o` | 导出 JSON |
| `--no-limit` | 禁用限频（短期高频可承受，长期大量建议加代理） |

时间筛选是「尽力而为」模式：热门词效果好，冷门词可能补充旧文凑数，拿到结果后按 `time` 字段自行二次过滤。

#### 2.2.2 `deep-research`（深度研究，复杂选题用）

当选题跨多个子主题、需要多视角并行核实，或者资料量大到单线程搜不完时，用这个。

- 工作方式：作为 research lead，把选题拆成子任务，派 research_subagent 并行搜，最后汇总成报告
- 触发：用户选择「深度研究」场景，或明确要求做 deep research
- 适用场景：
  - 选题第一次做资料摸底（广度优先）
  - 某个核心概念需要多角度深挖（深度优先）
  - 需要交叉验证争议性事实
- 默认会用 `wechat-article-search` 作为中文信息源之一，不用手动指定
- 报告会存成 `research_report_[topic].md`，可以直接作为本书资料附录的来源

#### 2.2.3 Web 搜索（英文/通用）

公众号搜不到的英文一手资料、官方文档、release notes，用 `WebSearch` / `WebFetch` 补。

#### 2.2.4 三者怎么搭

| 场景 | 工具组合 |
|------|---------|
| 单章动笔前补料 | `wechat-article-search`（快、中文密度高） |
| 新书选题摸底 | `deep-research`（拆子题并行搜） |
| 找官方硬事实 | `WebFetch` 官方文档 / release notes |
| 找英文圈经验 | `WebSearch` + `WebFetch` |
| 争议性结论核实 | `wechat-article-search` + `WebSearch` 双源交叉 |

原则：**中文判断看公众号，硬事实看官方，争议点双源交叉**。

### 2.3 章节推进节奏

- 默认按章推进，不要乱跳。
- 一章写完，再进入下一章。
- 当前已经确认这种节奏是合适的。

## 三、写作风格

### 3.1 正文基调与结构

- 单题正文控制在 `300-500` 字，`500` 字以内优先。
- 写法要像书正文，不要像提纲解释，也不要像资料综述。
- 语言要口语、自然、顺着讲。
- 不要写得过度工整、过度对仗、过度"方法论腔"。
- 回答结构默认可参考：
  - 先点出现象变化
  - 再解释原因
  - 再落到今天为什么会这样
  - 最后用一句短判断收住
- 结尾要收住，不要拖泥带水。
- 可以保留少量判断句，但不要堆概念，不要装腔，不要写成行业黑话堆砌。

### 3.2 段落与句子节奏

- 段落以"意群"为基本单位，一个意群就是讲清楚一个小命题，可以是一句复句，也可以是两三句递进。
- 不要把一个完整复句用句号硬切成多个碎段。意群内部的分句，该用逗号就用逗号。
- 不要写过短的单句独占一段。例如「答案是看产品类型。」「这一档能过简历筛选。」这类要么展开成完整判断，要么并入上一句。
- 段落长短要有交错感。可以是一段较长的完整论述，接一个收束性短句段，形成呼吸节奏，不要全是长段或全是短段。
- 整体读起来要像专业作者写的书正文，有正式度、有层次、有节奏，不是 PPT 的要点罗列，也不是口语碎碎念。

### 3.3 用词与语气

- 允许适度使用「包括」「例如」这类引导词来带具体例子，不要为了去 AI 味就一刀切禁掉，否则列举会变得很生硬。
- 允许语言的粗糙感。真人写作有口语毛刺、有跳跃、有没说圆的半句话，这些不是缺陷，是人味。不要把每句都打磨得无懈可击，那才是 AI 味。
- 允许偶尔的不工整、不对称、不完整。比如用「这事儿」「挺难讲」「说白了」这种带体温的口语，比「值得注意的是」「需要强调的是」这种模板腔强。
- 判断标准：读出来像一个真人在跟你聊天讲事，而不是在念稿。
- 逻辑要清楚，但不摆出"专家腔"。

### 3.4 硬性禁令

#### 3.4.1 句式禁令

- 严禁使用：`不是……而是……`
- 严禁使用破折号 `——`。需要做补充说明或转折时，拆成独立句子或独立段落。
- 这是硬约束，不是"少用"。
- 如果发现已有内容里有这两个句式或破折号，后续继续写作时要顺手改掉。

#### 3.4.2 信源与分析过程的隐藏

- 正文中严禁出现信源指代或分析过程的自述。
- 禁止写：「深扒 54 份 JD 后发现」「a16z 那篇五原则里有个建议」「Lenny's Newsletter 那期播客里」「Reddit 上有一条高赞帖」「根据墨刀的分析」「林舟在面试手册里提到」这类露出资料出处的表达。
- 禁止写：「分析 54 份 JD」「搜索关键词后看到」「抓取数据显示」「统计表明」这类把分析过程暴露出来的句子。
- 所有论点、数据、框架、金句都当作"你自己想说的话"写出来，不要带着资料的影子。
- 资料的真实归属集中在末尾的参考资料附录里，正文不留痕迹。

### 3.5 避免 AI 味

- 避免那种很整齐、很严谨、很模板化的表达。
- 避免高频使用这些句式：
  - `本质上`
  - `更准确地说`
  - 其他一眼很像 AI 总结腔的句子
- 不要写成"标准答案口吻"。
- 要更像一个真正在讲这件事的人，而不是在输出培训材料。

## 四、参考资料规范

- 全书在末尾单独设一个「参考资料」附录文件（例如 `07-参考资料.md`），统一汇总来源，成书更干净。
- 正文不再每题单独附来源，只讲内容；所有来源集中到附录里。
- 附录按来源域分组（官方文档、官方博客 / 工程文章、release notes / changelog、GitHub issues / discussions、社区讨论、长文经验帖 / 案例分析等）。
- 组内按被引用次数从多到少排列，方便顺着自己关心的方向往回查原始材料。
- 写作时每题仍要尽量覆盖 `2-4` 条来源，只是最终汇集进附录，不在正文里堆。
- 全书层面尽量让来源分散，避免 100 个问题都反复只依赖同一篇材料。
- 来源结构尽量混合，避免单一官方源统治全书。

## 五、事实审查方法（成稿后必做）

一本 100 问写完以后，必须对照参考资料做一次事实审查。核心原则：**先抽取硬声明清单，再针对性核实**，而不是通读资料后再审。实测下来，这种做法比通读式快 5-10 倍，且漏判更少。

### 5.1 为什么不能通读后再审

- 书稿约 2-3 万字，其中 90% 以上是判断性、建议性、收束性的"软内容"，根本不需要去资料里查证。
- 如果逐句拿去资料里搜，90% 的搜索是白搜的，agent 还容易超时被取消。
- 真正需要核实的"硬声明"通常只有 30-50 条，集中处理这些就够了。

### 5.2 第一步：从书稿里抽取硬声明

读正文时，只标记以下几类内容，软内容一律跳过：

- **具体数字**：专家数、积分数、倍数、版本号、上下文窗口大小、文件大小、日期
- **具体名称**：按钮名、菜单名、文件路径、错误码、文件名、技能名
- **完整清单**：连接器列表、模型列表、支持平台、技能清单
- **因果归因**：类似"官方建议先 X 再 Y""文档把 X 定义为 Y"——这类要查官方是否真这么说过
- **机制描述**：某功能怎么工作、权限怎么分级、数据存到哪里、自动化触发条件

判断性、建议性、收束性的句子（如"任务太宽容易翻车""先 Plan 再 Craft 更稳"）不算硬声明，直接放过。

### 5.3 第二步：逐条去资料里核实

- 用 `grep` 或搜索，按硬声明的关键词去资料里定位反证或支撑。
- **按事实类型批量查，不要按章节查**。一次把所有"路径相关"的声明一起捞，再查所有"连接器相关"的，命中率高，资料只翻一次。

### 5.4 双源交叉，指定各源的权威边界

这是最容易踩坑的一步。资料通常分两类，必须分清谁说了算：

- **官方文档**（docs、HTML 镜像、release notes）：数字、清单、路径、错误码、UI 名称、机制定义的**权威源**。
- **社区文章 / 经验帖**：使用经验、踩坑、边界判断的**权威源**。

两者冲突时，**硬事实以官方为准**，**经验判断以社区为准**。

> 踩坑记录：WorkBuddy 那本书第一次审查时，只读了社区文章合集（资料.txt），没查官方 HTML 文档，把三处真实信息（工作空间路径、错误码编号、Google 连接器）误判为"编造"并删改。后来用官方 HTML 补查，三处全部回退。教训：**审查必须双源交叉，单源容易误判**。

### 5.5 什么时候用 sub-agent，什么时候不用

- **这个规模（2-3 万字书稿 + 20 万字资料），主 agent 直接做 + grep 更快更稳**。总信息量不超出单 agent 上下文，派 sub-agent 的调度成本（启动、上下文重建、结果汇总）反而大于收益。
- 真正适合 sub-agent 的场景：
  - 书稿或资料量再翻几倍，超出单 agent 上下文
  - 需要**不同专业视角**并行（如同时做技术审查 + 文风审查）
  - 子任务**完全独立**且可并行
- 如果一定要用 sub-agent 审查，按以下方式切片：
  - 按**事实类型**切片（如"你只查所有模型相关声明""你只查所有连接器相关声明"），不要按章节切片
  - 给明确的**硬声明定义**（见 5.2），让它只查硬内容，软内容跳过
  - 给**固定输出格式**（题号 / 原文 / 反证 / 建议 四段式）
  - **指明权威源边界**（见 5.4），避免它只信一个源

### 5.6 输出格式

审查报告统一用这个结构：

```
## 审查总结
- 共审查 X 题
- 硬错误 X 处、可疑 X 处、风格建议 X 处
- 整体事实准确度评价（一句话）

## 硬错误清单（必须改）
每条：题号 + 原文 + 问题 + 资料反证（标来源）+ 建议

## 可疑清单（建议核实）
同上格式

## 整体评价
哪些章最稳、哪些风险高。
```

## 六、给其他 Agent 的建议

- 开始写之前，先读这个 `reference.md`。
- 如果要继续当前主题，也先读对应章节文件，避免风格断层。
- 成稿后做事实审查时，务必按第五节的方法走，尤其是双源交叉（5.4）。
- 如果要继续更新当前 Agent 的长期记忆，可以把本文件中的关键约束同步回记忆系统。
- 如果用户新增风格要求，优先更新这个文件，再继续写作。
